# UC-028: Login

**Actor**: User registrado y verificado
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**User Story**: [US-028-b](../user-stories/US-028-b.md), [US-028-f](../user-stories/US-028-f.md)
**ADR refs**: [ADR-0023](../../decisions/0023-auth-flow-jwt-cookie-layout-guards.md), [ADR-0034](../../decisions/0034-redis-como-cache-y-ephemeral-state.md)

## Pre-condiciones

- Tiene cuenta registrada con `email_verified_at` no nulo.
- No está disabled.

## Flujo principal

1. User envía email + password a `POST /api/identity/sign-in`.
2. Sistema valida credenciales contra el `PasswordHash` del User.
3. Sistema chequea estado: verificado y no disabled.
4. Sistema emite access token (JWT, TTL 15 min) + refresh token (32 bytes, TTL 30 días).
5. Refresh token hash se guarda en Redis con `userId` como valor (revocation list, ADR-0034 patrón #1).
6. Access token va en cookie httpOnly `planb_session`. Refresh token en cookie httpOnly `planb_refresh`.
7. Body retorna `{ userId, email, role }`.

## Flujos alternativos

- **Credenciales inválidas** (email no existe o password mal): 401 con `title: identity.signin.invalid_credentials` (mensaje genérico, anti-enumeration).
- **Email no verificado**: 403 con `title: identity.account.email_not_verified`.
- **Cuenta deshabilitada**: 403 con `title: identity.account.disabled`.

## Post-condiciones

- Sesión activa hasta logout, expiración del refresh, o revocación explícita.
- User puede acceder a rutas authed (member / teacher / staff según rol).
