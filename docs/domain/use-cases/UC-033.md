# UC-033: Recuperación de contraseña

**Actor**: Member que olvidó su password
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**User Story**: [US-033-i](../user-stories/US-033-i.md)
**ADR refs**: [ADR-0023](../../decisions/0023-auth-flow-jwt-cookie-layout-guards.md), [ADR-0033](../../decisions/0033-verification-token-como-child-entity.md)

## Pre-condiciones

- El user tiene cuenta verificada (`email_verified_at NOT NULL`).

## Flujo principal

1. User clickea "¿Olvidaste tu contraseña?" en `/auth` y abre `/forgot-password`.
2. Ingresa su email, submit.
3. Backend (silenciosamente si el email no existe o no está verificado) emite `VerificationToken` con `purpose='password-reset'` y manda mail con link `{frontend}/reset-password?token=...`.
4. Frontend muestra siempre el mismo mensaje genérico.
5. User clickea link en su email, llega a `/reset-password?token=...`.
6. Ingresa contraseña nueva (con confirmación), submit.
7. Backend valida token, hashea password nueva, marca token como consumed, revoca todos los refresh tokens del user.
8. Redirect a `/auth?reset=success` con toast de confirmación.

## Flujos alternativos

- **Token expirado o ya usado**: backend responde 400, frontend muestra mensaje con CTA para volver a `/forgot-password`.
- **Email no existe o no verificado**: backend responde 204 sin mandar mail. Frontend igual muestra el mensaje genérico (anti-enumeración).
- **Password no cumple policy**: backend rechaza con 400, frontend marca el campo con error.

## Post-condiciones

- Si el flujo se completa: el user tiene password nueva, sus refresh tokens viejos están revocados, debe loguearse de nuevo.
- El token de reset queda marcado como consumed y no se puede reusar.
