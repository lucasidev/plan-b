# UC-029: Sign-out

**Actor**: User autenticado
**Epic**: [EPIC-02 — Identidad y autenticación](../epics/EPIC-02.md)
**User Story**: [US-029-i](../user-stories/US-029-i.md)
**ADR refs**: [ADR-0023](../../decisions/0023-auth-flow-jwt-cookie-layout-guards.md), [ADR-0034](../../decisions/0034-redis-como-cache-y-ephemeral-state.md)

## Pre-condiciones

- User autenticado (cookie `planb_session` válida).

## Flujo principal

1. User clickea "Cerrar sesión" en el menu de cuenta.
2. Frontend dispara `POST /api/identity/sign-out`.
3. Backend extrae el refresh token del request, hashea, llama `DEL identity:refresh:{hash}` en Redis.
4. Backend setea `Set-Cookie` con `planb_session` y `planb_refresh` vencidos (limpiar del browser).
5. Backend retorna 204.
6. Frontend redirige a `/sign-in`.

## Flujos alternativos

- **Sin sesión activa** (cookie ausente o expirada): 204 igual. Operación idempotente.
- **Refresh token ya revocado**: 204. El estado deseado ya está logrado.

## Post-condiciones

- Refresh token revocado (no se puede usar para emitir un nuevo access token).
- Cookies del browser limpiadas.
- Si había access token vivo en memoria (otra tab), seguirá funcionando hasta su TTL natural (15 min). Esto es aceptable para MVP — para revocación inmediata habría que mantener una blacklist de access tokens, scope post-MVP.
