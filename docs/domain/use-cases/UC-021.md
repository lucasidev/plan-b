# UC-021: Reenviar email de verificación

**Actor**: User pendiente de verificar
**Epic**: [EPIC-02 — Identidad y autenticación](../epics/EPIC-02.md)
**User Story**: [US-021](../user-stories/US-021.md)
**ADR refs**: [ADR-0033](../../decisions/0033-verification-token-como-child-entity.md)

## Pre-condiciones

- Tiene cuenta no verificada (`email_verified_at=null`, `expired_at=null`).

## Flujo principal

1. User pide reenvío vía formulario con su email.
2. Sistema valida que existe user no verificado.
3. Sistema invalida tokens activos del `purpose=UserEmailVerification`.
4. Sistema emite token nuevo.
5. Sistema envía email.

## Post-condiciones

- User recibe nuevo email.
- Token previo queda invalidated.

## Notas

- Rate-limited (≤ 3 reenvíos/hora).
- Anti-enumeration: si el email no existe o ya está verificado, retorna 200 sin revelar la diferencia.
- Ver ADR-0033 (VerificationToken como child entity).
