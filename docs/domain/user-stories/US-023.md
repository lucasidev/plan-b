## Como alumno, quiero guardar una simulación que me cierra como draft privado para volver a ella después

Como alumno, quiero persistir la combinación que estoy considerando como SimulationDraft visibility=private, listarla, editarla, borrarla y promoverla a plan activo, para no perder el laburo de armar la combinación entre sesiones.

## Acceptance Criteria

### Backend

- [ ] `POST /api/me/simulations/drafts` con `{ subjectIds[], termId, label?, commissionIds? }`.
- [ ] Crea `SimulationDraft` con `visibility = 'private'`, `owner_profile_id = StudentProfile.id`.
- [ ] Valida que `subject_ids` no esté vacío y que las materias pertenezcan al CareerPlan del profile (cross-BC vía `IAcademicQueryService`).
- [ ] `GET /api/me/simulations/drafts` lista los drafts propios (private y shared).
- [ ] `PATCH /api/me/simulations/drafts/{id}` actualiza composición / comisiones / label.
- [ ] `DELETE /api/me/simulations/drafts/{id}` hard delete.
- [ ] `POST /api/me/simulations/drafts/{id}/promote` lo vuelve el plan activo del término; el activo anterior del mismo término pasa a `archived` (flip, no copia; el ADR pendiente del lifecycle se decide acá).
- [ ] Authorization: solo el owner lee/edita/borra/promueve; otros reciben 403. Requiere `StudentProfile` activo.
- [ ] Rate limit: 60 saves/hora por user (`planning:ratelimit:save:{userId}`).

### Frontend

- [ ] El tab "Borradores" de `/plan` consume los drafts reales (reemplaza `MOCK_DRAFTS`); crear desde "+ Nuevo borrador", editar, borrar.
- [ ] "Publicar plan" (promote) funciona desde el modal existente; el tab "En curso" muestra el plan activo real (reemplaza `MOCK_ACTIVE_SIMULATION`).

## Out of scope

Esto NO incluye:

- Compartir al corpus público: [US-024](US-024.md) (mismo sprint).
- Gate premium para guardar (ADR-0028): MVP abierto, future-flag.
- Historial de versiones de un draft: un draft es su último estado.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| Promote con otro plan activo en el mismo término | El anterior pasa a `archived` atómicamente; nunca dos activos por término. |
| Draft con materia que dejó de estar disponible (cursada aprobada después) | El draft se conserva; al evaluarlo, la materia aparece bloqueada/ya aprobada con su motivo. |
| Doble submit del promote | Idempotente: el segundo request encuentra el draft ya activo y responde el estado actual. |
| Borrar el plan activo | Permitido (hard delete, dato efímero); el término queda sin plan activo. |
| Rate limit alcanzado | 429 con el copy honesto en la UI. |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** una combinación armada en la sesión **When** la guardo como borrador con label **Then** aparece en el tab Borradores tras un refresh.
2. **Given** un borrador guardado **When** lo promuevo **Then** pasa a "En curso" y el activo anterior del término queda archivado.
3. **Given** un draft de otro alumno **When** intento leerlo/editarlo por id **Then** 403.

### Cobertura por capa

- **Unit / domain**: invariantes del aggregate `SimulationDraft` (visibility, flip de status).
- **Integration**: CRUD + promote + authorization + rate limit con infra real.
- **E2E**: guardar → refresh → promover → ver "En curso".

## Notas de implementación

- **Hard delete OK porque son efímeros**: el data-model lo declara explícitamente. Una simulación no es histórico que se preserve, es un borrador.
- **BC Planning separado de Enrollments**: ADR-0029. Enrollments captura pasado (qué cursé), Planning captura futuro (qué pienso cursar).
- **`owner_profile_id` como UUID lógico, no FK**: ADR-0017. Planning no tiene FK cross-schema a Identity.
- **Premium feature**: ADR-0028. Simular es libre (US-016). Guardar requiere N reseñas cuando aterrice premium. MVP: abierta, future-flag.
- **El período del borrador admite cualquier cadencia**: el `termId` referencia un `AcademicTerm`, cuyo `kind` puede ser bimestral, cuatrimestral, semestral o anual (año y semestre pueden ser el mismo período según la universidad; vocabulario en [ubiquitous-language.md](../ubiquitous-language.md), sección Planificador). El tipo mock del frontend (`AcademicPeriod.term: '1c' | '2c'`, US-046) es placeholder y no fija el modelo: al aterrizar esta US, el período del draft es un `AcademicTerm` real, no un enum cuatrimestral.
