## Como alumno, quiero guardar una simulación que me cierra como draft privado para volver a ella después

Como alumno, quiero persistir la combinación que estoy considerando como SimulationDraft visibility=private, listarla, editarla y borrarla, para no perder el laburo de armar la combinación entre sesiones.

## Acceptance Criteria

### Backend

- [ ] `POST /api/me/simulations/drafts` con `{ subjectIds[], termId, label? }`.
- [ ] Crea `SimulationDraft` con `visibility = 'private'`, `owner_profile_id = StudentProfile.id`.
- [ ] Valida que `subject_ids` no esté vacío.
- [ ] Valida que las materias pertenezcan al CareerPlan del profile (cross-BC vía `IAcademicQueryService`).
- [ ] `GET /api/me/simulations/drafts` lista los drafts propios (incluye private y shared del user).
- [ ] `PATCH /api/me/simulations/drafts/{id}` actualiza composición / label. Mantiene visibility actual.
- [ ] `DELETE /api/me/simulations/drafts/{id}` hard delete.
- [ ] Requiere `StudentProfile` activo.

### Frontend

- [ ] Lista de drafts en `/cuatrimestre/borradores`.

## Sub-tasks

- [ ] Aggregate `SimulationDraft` en módulo Planning (BC nuevo, ADR-0029)
- [ ] CRUD endpoints Carter
- [ ] UI list + detail draft

## Notas de implementación

- **Hard delete OK porque son efímeros**: el data-model lo declara explícitamente. Una simulación no es histórico que se preserve, es un borrador. Si el alumno la borra, no queremos retenerlo.
- **BC Planning separado de Enrollments**: ADR-0029. Enrollments captura pasado (qué cursé), Planning captura futuro (qué pienso cursar). Mismo data shape (alumno + materia + term) pero distinta semántica y lifecycle.
- **`owner_profile_id` como UUID lógico, no FK**: ADR-0017. Planning no tiene FK cross-schema a Identity. El handler valida vía PublicContract de Identity.
- **Premium feature**: ADR-0028. Simular es libre (US-016). Guardar el draft requiere haber publicado al menos N reseñas (regla a definir cuando aterrice). MVP: feature abierta, future-flag para premium.
