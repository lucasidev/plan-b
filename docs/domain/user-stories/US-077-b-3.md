**ADR refs**: [ADR-0040](../../decisions/0040-notifications-como-bounded-context.md), [ADR-0034](../../decisions/0034-redis-como-cache-y-ephemeral-state.md)

## Como sistema, quiero un canal email para entregar notifs críticas (security alerts, moderator edit requests, plan period opened, plan migration) via SMTP genérico, usando Mailpit en dev y un vendor real en prod, para que el user reciba avisos importantes fuera de la app

Tercer y último slice del Notifications BC. Agrega el canal email encima de la infra de in-app. La impl es **vendor-agnostic**: SMTP genérico que apunta a Mailpit en dev/CI y a Postmark / SES / SendGrid / lo-que-sea en prod, configurado solo por env vars.

## Acceptance Criteria

### Interface y abstracción

- [ ] Interface `INotificationEmailSender` en `Planb.Notifications.Application`:
  ```csharp
  interface INotificationEmailSender {
      Task<EmailDeliveryResult> SendAsync(EmailMessage msg, CancellationToken ct);
  }

  record EmailMessage(string ToAddress, string Subject, string HtmlBody, string TextBody);
  record EmailDeliveryResult(bool Success, string? FailureReason);
  ```
- [ ] **Impl única en MVP**: `SmtpNotificationEmailSender` usa `MailKit` (más moderna que `System.Net.Mail.SmtpClient` para casos modernos; ya está en el ecosistema .NET 10).
- [ ] Configuración via env vars:
  - `SMTP__Host` (string, default `localhost`).
  - `SMTP__Port` (int, default `1025`, alineado a Mailpit).
  - `SMTP__User` (string?, nullable; null = no auth).
  - `SMTP__Pass` (string?, nullable; secret).
  - `SMTP__UseTls` (bool, default `false` para Mailpit; `true` para prod).
  - `SMTP__FromAddress` (string, default `noreply@plan-b.local`; en prod `noreply@plan-b.app`).
  - `SMTP__FromName` (string, default `plan-b`).
- [ ] Registro DI en `Program.cs`: `services.AddScoped<INotificationEmailSender, SmtpNotificationEmailSender>()`.

### Templates

- [ ] Templates HTML por kind en `Planb.Notifications.Application/Templates/`:
  - `UserPasswordChanged.cshtml` (security alert).
  - `ReviewResponded.cshtml`.
  - `ReviewEditRequested.cshtml`.
  - `PlanPeriodOpened.cshtml`.
  - `PlanChanged.cshtml`.
  - `PlanMigrationApplied.cshtml`.
  - `SystemAnnouncement.cshtml` (template con `{title}` + `{body}` parameterizable).
- [ ] **Engine de templates**: Razor estático (no full ASP.NET razor pipeline). Library: `RazorLight` o equivalente. Plain string interpolation también OK para MVP si Razor es overkill.
- [ ] Cada template recibe `NotificationEmailContext` con campos relevantes (`{ userName, actionUrl, actorName, subject, ... }`).
- [ ] Header + footer compartidos (con disclaimer "no afiliado oficialmente a ninguna universidad" + unsubscribe stub).
- [ ] Plain text version (template `.txt.cshtml`) para clientes que no renderean HTML.
- [ ] Subject del email pre-computado en el template (`@{ Layout = null; var subject = "Tu contraseña se cambió"; }`) o como propiedad del context.

### Wolverine job de delivery

- [ ] `DeliverNotificationEmailJob` (Wolverine durable job, NO scheduled; reactivo a flag `email_status = 'pending'`):
  - Levanta notifs con `email_status = 'pending'` ordenadas por `email_requested_at ASC` (FIFO).
  - Para cada una:
    - Resuelve el template según `kind`.
    - Resuelve el `EmailMessage` context (consulta user email via `IUserQueryService` cross-BC).
    - Llama `INotificationEmailSender.SendAsync`.
    - Si success: marca `email_status = 'sent'`, `email_sent_at = now()`.
    - Si failure recuperable (network blip, SMTP timeout): retry exponencial (Wolverine out-of-the-box, 3 intentos con delays 30s / 2m / 10m).
    - Si failure tras N retries: marca `email_status = 'failed'`, `email_failure_reason = exception.Message`. Log warning.
- [ ] Job batching: hasta 50 emails por run para no saturar Mailpit / SMTP vendor.
- [ ] **Idempotencia**: si el job es retried y la notif ya está `sent`, skip.

### Rate limit y backpressure

- [ ] Rate limiter Redis (`IRateLimiter` que ya existe, ADR-0034) por destinatario: max 10 emails/hora por user (anti-spam si algo loopea).
- [ ] Si excede: marca `email_status = 'rate_limited'` (nuevo estado opcional, o stay `pending` con `next_retry_at`).

### Mailpit en dev/CI

- [ ] `docker-compose.yml` del proyecto ya tiene Mailpit. Confirmar que las env vars del backend en dev apuntan ahí (`SMTP__Host=localhost`, `SMTP__Port=1025`, `SMTP__UseTls=false`).
- [ ] Integration tests usan Mailpit como SMTP target. Helper `frontend/e2e/helpers/mailpit.ts` ya existe; replicar uno para backend tests si no hay (`backend/tests/.../MailpitFixture.cs`).

### Vendor de prod (decisión diferida)

- [ ] **Out de esta US**: decisión del vendor (Postmark / SES / SendGrid / etc.).
- [ ] La impl SMTP es vendor-agnostic: cualquier vendor que exponga SMTP standard funciona con cambio de env vars.
- [ ] Cuando se decida vendor en deploy: documentar las env vars de prod en `docs/operations/email-delivery.md` (crear). Sin cambio de código.

### Tests

- [ ] Integration tests por template:
  - Dispatch el event correspondiente → b-2 crea notif con `email_status = 'pending'` → job corre → Mailpit recibe email con shape esperado (subject correcto, body contiene los campos del context).
  - Assertion via helper Mailpit: `await mailpit.WaitForLatestEmail(to: user.Email)`.
- [ ] Test de retry: SMTP mock que tira exception → 3 retries → marca `failed`.
- [ ] Test de rate limit: 11 emails al mismo user en 1 hora → 11th queda `rate_limited`.
- [ ] Test de idempotencia: re-run el job sobre una notif `sent` → no se reenvía.

## Out of scope

- **Decisión de vendor de prod**: se decide en deploy.
- **Bounce tracking, complaint feedback loops, dedicated IP, reputation management**: out de MVP.
- **Unsubscribe links funcionales por kind**: stub en footer; cuando aterricen preferences en US-072 (ajustes), se enchufa.
- **Email i18n**: español rioplatense hardcoded.
- **Email templates con images attached / inline assets**: out, solo HTML + plain text con links.
- **Push notifications móviles**: out (post-MVP).
- **Email digest / aggregation diaria/semanal**: out (1 event = 1 email en MVP; deuda explícita si volumen lo justifica).

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User sin email verificado | Job skip + marca notif `email_status = 'skipped:unverified'` (log warning). Solo enviamos a verified emails. |
| User disabled | Skip + marca `skipped:user_disabled`. |
| Template engine error (campo missing) | Log error, marca `failed`, no retry (es bug, no transient). |
| Mailpit/SMTP offline (network blip) | Retry exponencial via Wolverine. Eventualmente `failed`. |
| Email con caracteres especiales en subject (UTF-8) | MailKit handles native. RFC 2047 encoded. |
| User con email cambiado entre creación de notif y delivery | Email va a la dirección actual (consulta user al momento de delivery, no de creación). |
| Volumen alto (1000 emails pending) | Batch 50/run. Si Wolverine corre cada 1min, drena ~3000/h. Sin saturar vendor. |
| Rate limit hit | Notif queda `rate_limited`. Cron job recupera al cabo de la hora. |

## Notas de implementación

- **MailKit vs `System.Net.Mail.SmtpClient`**: MailKit es la library recomendada por el equipo .NET (Microsoft deprecó `SmtpClient` para nuevos uses). Mejor TLS modern, mejor manejo de retries, soporte STARTTLS.
- **Templates como Razor static**: `RazorLight` permite Razor templates fuera del pipeline web. Buena DX (autocompleta, type-safe) y suficiente para nuestros casos.
- **Job reactivo, no scheduled**: Wolverine puede dispararse on-demand cuando llega un event que marca `email_pending`. Evita un job que corre cada minuto y barre vacío.
- **From address**: en dev `noreply@plan-b.local` (Mailpit acepta). En prod `noreply@plan-b.app` con DKIM + SPF configurados (deuda de deploy).
- **Unsubscribe stub**: footer dice "Para dejar de recibir estos avisos, ajustá tus preferencias en plan-b.app/ajustes" con link. No requiere unsubscribe handler en MVP (el user lo hace desde Ajustes US-072).
- **Tests con Mailpit fixture**: el helper hace `await client.WaitForEmail({ to, subject })` con timeout. Es flaky-resistant.
- **Plain text version**: requerido por buenas prácticas anti-spam. Generado del HTML con un convert simple (strip tags + format basic).
