# US-062: Gestionar Subject + Prerequisite

**Status**: En curso
**Sprint**: S9
**Epic**: [EPIC-08: Backoffice de catálogo](../epics/EPIC-08.md)
**Priority**: High
**Effort**: L (el doc decía M: son dos CRUDs + domain service de grafo + migration + UI)
**UC**: [UC-062](../use-cases/UC-062.md)
**ADR refs**: ADR-0003

## Como admin, quiero CRUD de Subjects y sus correlativas para definir el plan de estudios

Como admin, quiero agregar Subjects al CareerPlan y agregar Prerequisites validando aciclicidad del grafo (domain service), separando para_cursar vs para_rendir (ADR-0003).

## Acceptance Criteria

### Backend

> El backoffice no tiene namespace propio: cada feature admin vive en la API de su módulo dueño
> (Academic). Ver [ADR-0050](../../decisions/0050-backoffice-como-corte-transversal.md). El AC
> original declaraba `/api/admin/...`, que nunca existió.

- [x] CRUD Subject bajo `/api/academic/career-plans/{planId}/subjects`:
  - `POST /api/academic/career-plans/{planId}/subjects` con `{ code, name, yearInPlan, termKind, termInYear?, weeklyHours, totalHours, description? }`.
  - `GET /api/academic/career-plans/{planId}/subjects` (listado admin: incluye archivadas) y `GET .../{subjectId}` (detalle admin, anidado porque `GET /api/academic/subjects/{id}` ya es el público de US-002).
  - `PATCH /api/academic/subjects/{id}` y `POST /api/academic/subjects/{id}/reactivate`.
  - `UNIQUE(career_plan_id, code)`, con 409 `academic.subject.code_already_exists`.
  - CHECK: `term_kind = 'anual'` ⇒ `term_in_year IS NULL`; `term_kind != 'anual'` ⇒ `term_in_year NOT NULL`.
- [x] CRUD Prerequisite bajo `/api/academic/subjects/{subjectId}/prerequisites`:
  - `POST` con `{ requiredSubjectId, type }` (type: `ParaCursar` | `ParaRendir`), `DELETE .../{requiredSubjectId}/{type}`, y `GET /api/academic/career-plans/{planId}/prerequisites` para el grafo del plan.
  - Domain service valida aciclicidad del grafo por type (cada tipo es un DAG separado).
  - Ambas materias deben pertenecer al mismo `career_plan_id`.
- [x] Requiere `role = 'admin'`.
- [x] Soft-delete de Subject: si un Subject tiene Prerequisites apuntando a él (otras materias lo necesitan), borrarlo devuelve 409 `academic.subject.has_dependents` con el listado de materias dependientes. El admin debe re-asignar prerequisites primero.
- [x] El catálogo público deja de mostrar las materias archivadas, pero el historial del alumno (US-013 y el import de PDF) las sigue viendo vía `includeArchived=true`: pudo haber cursado una materia que después se archivó.

### Frontend

- [ ] UI grid de subjects por año + matriz de prerequisites.

## Sub-tasks

- [x] Aggregate Subject (suma `Update` / `Deactivate` / `Reactivate` + `IsActive` + `UpdatedAt`) y entidad `Prerequisite`. **No es child entity de Subject**, como decía este sub-task: [ADR-0003](../../decisions/0003-correlativas-con-dos-tipos.md) y el data-model definen tabla propia con `PRIMARY KEY (subject_id, required_subject_id, type)`, y validar aciclicidad necesita el grafo del plan entero, no las aristas de una materia sola.
- [x] Domain service `IPrerequisiteGraphValidator` (DAG check con DFS iterativo, un grafo por type)
- [x] Endpoints Carter (6 de Subject + 3 de Prerequisite)
- [x] UI admin (feature `manage-subjects`: tabla por año, form, panel de correlativas)
- [x] Integration tests: ciclo simple rechazado, ciclo indirecto rechazado, prerequisite cross-plan rechazado, prerequisite mismo subject rechazado, y los dos grafos independientes entre sí
- [ ] E2E del flujo admin (alta de materia + alta de correlativa + rechazo de ciclo)

## Notas de implementación

- **Dos DAGs separados, uno por type**: ADR-0003. `para_cursar` y `para_rendir` son grafos distintos sobre los mismos subjects. Una materia puede tener correlativa A `para_cursar` y B `para_rendir` sin que se mezclen.
- **Aciclicidad validada en domain service, no en SQL**: ADR-0017. La FK `(subject_id, required_subject_id)` previene self-loops, pero ciclos largos (A → B → C → A) requieren DFS en código.
- **`subject_id != required_subject_id`**: CHECK del data-model. No hay correlativa de uno consigo mismo.
