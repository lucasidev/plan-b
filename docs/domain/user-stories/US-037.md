## Como member recién verificado, quiero un onboarding guiado que me lleve de registrarme a entrar a Inicio con mi historial cargado

Hoy un member que se acaba de registrar y verificar email aterriza directo en `/home` sin contexto sobre qué cargar primero. El rediseño introduce un onboarding de 4 pasos que captura carrera y arma una base mínima de historial antes de mandarlo a la app.

## Acceptance Criteria

- [ ] Ruta `app/(onboarding)/` (route group nuevo, guard distinto al de `(member)`).
- [ ] Guard del layout: `getSession()` verificado + verificar que el user todavía NO tiene `StudentProfile` activo. Si ya tiene, redirige a `/home`.
- [ ] **Paso 01 · Bienvenida**: pantalla con copy de bienvenida + CTA "Empecemos". Sin form. Solo onboarding al producto.
- [ ] **Paso 02 · Carrera**: form con cascadas University → Career → CareerPlan + año de ingreso (`enrollment_year`). Submit dispara `POST /api/me/student-profiles` (US-012-b backend ya implementado).
- [ ] **Paso 03 · Historial**: 3 opciones radio:
  - "Cargar PDF / captura del SIU" — `<input type="file">`. **Diferido** a US-014 (ya en backlog). Por ahora marcador.
  - "Cargarlo manualmente" — link a flow de carga manual. **Diferido** a US-013 (ya en backlog).
  - "Lo cargo después" — saltea al paso 04 sin cargar nada.
- [ ] **Paso 04 · Listo**: pantalla de confirmación con resumen (carrera, año, % historial cargado si aplica) + CTA "Ir a Inicio" → `/home`.
- [ ] Si el user abandona el onboarding (cierra el browser), al re-loguearse arranca de nuevo en el paso donde quedó. Persistencia: el StudentProfile creado en paso 02 ya queda; estado del wizard se infiere de los datos persistidos.
- [ ] Mobile diferido (per ADR-0041 cross-cutting decision: web first).

## Sub-tasks

- [ ] `app/(onboarding)/layout.tsx` con guard "no tiene StudentProfile activo".
- [ ] 4 rutas dentro: `welcome/page.tsx`, `career/page.tsx`, `history/page.tsx`, `done/page.tsx`.
- [ ] Feature `features/onboarding/`:
  - `actions.ts` con server action que crea el StudentProfile (proxy del endpoint US-012-b).
  - `components/{welcome,career-form,history-options,done}.tsx`.
  - `schema.ts` con Zod para validar carrera + año.
- [ ] Para "Cargarlo después" en el paso 3: navega a `/onboarding/done` y luego a `/home`. La home debe mostrar empty state que invite a completar (per US-... estados vacíos, deuda futura).
- [ ] Tests vitest: schema + action.
- [ ] Test E2E Playwright: full flow de Lucía (register → verify → sign-in → onboarding 4 pasos → /home).

## Notas de implementación

- **El StudentProfile se crea en paso 02**, no al final del onboarding. Eso permite que un user que abandona después del paso 2 ya tenga su perfil mínimo.
- **Carga de historial diferida**: las US-013 (manual) y US-014 (PDF) son trabajo separado. Este onboarding las "expone" como opciones pero sus implementaciones son US distintas. En este sprint, las dos opciones de carga muestran "Esta opción va a estar disponible pronto" o redirect a una página stub. Solo "Lo cargo después" es path 100% funcional.
- **Sin StudentProfile no se entra a `(member)`**: el guard de `(member)/layout.tsx` chequea que el user tenga un StudentProfile activo. Si no tiene, redirige a `/onboarding/welcome`. Esa lógica es trabajo de este US (acoplada al onboarding).
- **Auth rebuild (US-036)** debería aterrizar antes o en paralelo: el redirect post-sign-in que hoy va a `/home` pasa a `/onboarding/welcome` si el user no tiene StudentProfile.
