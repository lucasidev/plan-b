# US-037: Onboarding (4 pasos post-registro)

**Status**: Backlog
**Sprint**: candidato a S2
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: High
**Effort**: M
**ADR refs**: [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md), [ADR-0008](../../decisions/0008-roles-exclusivos-profiles-como-capacidades.md)

## Como member recién verificado, quiero un onboarding guiado que me lleve de registrarme a entrar a Inicio con mi historial cargado

Hoy un member que se acaba de registrar y verificar email aterriza directo en `/home` sin contexto sobre qué cargar primero. El rediseño introduce un onboarding de 4 pasos que captura carrera y arma una base mínima de historial antes de mandarlo a la app.

## Acceptance Criteria

- [ ] Ruta `app/(onboarding)/` (route group nuevo, guard distinto al de `(member)`).
- [ ] Guard del layout: `getSession()` verificado + verificar que el user todavía NO tiene `StudentProfile` activo. Si ya tiene, redirige a `/home`.
- [ ] **Paso 01 · Bienvenida**: pantalla con copy de bienvenida + CTA "Empecemos". Sin form. Solo onboarding al producto.
- [ ] **Paso 02 · Carrera**: form con cascadas University → Career → CareerPlan + año de ingreso (`enrollment_year`). Submit dispara `POST /api/me/student-profiles` (US-012-b backend ya implementado).
- [ ] **Paso 03 · Historial**: 3 opciones radio:
  - "Cargar PDF / captura del SIU": `<input type="file">`. **Diferido** a US-014 (ya en backlog). Por ahora marcador.
  - "Cargarlo manualmente": link a flow de carga manual. **Diferido** a US-013 (ya en backlog).
  - "Lo cargo después": saltea al paso 04 sin cargar nada.
- [ ] **Paso 04 · Listo**: pantalla de confirmación con resumen (carrera, año, % historial cargado si aplica) + CTA "Ir a Inicio" → `/home`.
- [ ] Si el user abandona el onboarding (cierra el browser), al re-loguearse arranca de nuevo en el paso donde quedó. Persistencia: el StudentProfile creado en paso 02 ya queda; estado del wizard se infiere de los datos persistidos.
- [ ] Mobile diferido (per ADR-0041 cross-cutting decision: web first).

## Out of scope

Esto NO incluye:

- **Carga de historial PDF** (paso 03 opción 1) → diferido a [US-014](US-014.md). En este sprint la opción muestra "Disponible pronto" y no permite avanzar por ese path.
- **Carga de historial manual entrada por entrada** (paso 03 opción 2) → diferido a [US-013](US-013.md). Misma deuda que arriba.
- **Cambiar de carrera después del onboarding** (post-MVP). Una vez que el StudentProfile se crea con `careerPlanId`, no hay flow para cambiarlo desde el frontend en este sprint. Hard-edit en DB para ahora.
- **Mobile** completo: web-first (ADR-0041). Onboarding renderea en viewport mobile como degraded UX (form usable pero no diseñado).
- **i18n**: copy en español rioplatense hardcoded. Sin pipe de traducción.
- **Backfill de users existentes sin StudentProfile**: el seed de personas (Lucía, Mateo, Paula, Martín) NO se actualiza para crearles StudentProfile. Si un seed user loguea durante S2, va al onboarding y lo completa él mismo (deseable para QA).

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User cierra browser en paso 02 después de submit OK (StudentProfile creado pero `done` no visto) | Al re-loguearse, el guard de `(member)` ve que tiene profile → `/home`. El onboarding se considera "completado" cuando el profile existe; los pasos 03/04 son UX no obligatorios. |
| User selecciona Universidad A → Carrera X. Cambia a Universidad B | El dropdown de Career se resetea (dispara nuevo fetch). El de Plan también. Si tenía año de ingreso, se preserva. |
| `GET /api/academic/universities` falla (5xx, timeout, offline) | Empty state con mensaje "No pudimos cargar las universidades" + botón "Reintentar". El submit queda disabled. |
| Catálogo Academic vacío (sin unis seedeadas) | Empty state explícito "Todavía no hay universidades disponibles. Avisanos a soporte." (no debería pasar en dev/prod, pero lo cubrimos). |
| User abre dos tabs en `/onboarding/career`, ambas hacen submit simultáneo | Backend: el segundo POST devuelve 409 (`identity.student_profile.already_exists`). UI muestra "Ya tenés un perfil. Ir a Inicio" + CTA. |
| User entra directo a `/onboarding/done` sin pasar por `/onboarding/career` | Si NO tiene StudentProfile → redirige a `/onboarding/welcome`. Si SÍ tiene → muestra `done` normal (estado consistente). |
| Back button del browser entre pasos | Navegación stateful: data del form de career está persistida en backend, no en URL state. Welcome / history son stateless. Done también. |
| Sesión expira en paso 02 mid-submit | Backend devuelve 401. El server action lo atrapa, redirige a `/sign-in?expired=1` con un mensaje "Tu sesión expiró. Volvé a entrar y completá tu perfil." |
| Doble click "Crear cuenta" / "Continuar" | Submit button con `disabled={pending}` (React `useFormStatus`). Backend además idempotent: 2do POST con mismo userId devuelve 409 que UI mapea a "ya estás registrado, andá a Inicio". |
| Año de ingreso fuera de rango (< 1990 o > current_year + 1) | Validación client-side bloquea submit + mensaje "Año entre 1990 y <next>". Si pasa al backend, el validator devuelve 400 ValidationProblem. |
| User skipea paso 02 metiéndose a `/onboarding/history` por URL directa | Guard del layout `(onboarding)`: si está en `history` o `done` y NO tiene StudentProfile → redirige a `/onboarding/welcome`. |
| Keyboard-only: completar las 3 cascadas con Tab + arrows | Cada `<select>` (o el componente combobox) navegable con keyboard nativo. Submit accesible con Enter después del último campo. |
| Screen reader: transición entre pasos 01→02→03→04 | Cada page tiene `<h1>` único describiendo el paso. Live region anuncia "Paso 2 de 4: tu carrera". |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** Lucía recién verificó email y nunca completó onboarding, **when** se loguea, **then** el guard de `(member)` la redirige a `/onboarding/welcome` (no a `/home`).
2. **Given** Lucía está en `/onboarding/career` sin haber seleccionado nada, **when** intenta submit con campos vacíos, **then** el form muestra mensaje per-field y el submit queda bloqueado.
3. **Given** Lucía completó paso 02 (StudentProfile creado), **when** elige "Lo cargo después" en paso 03, **then** aterriza en `/onboarding/done` y el CTA "Ir a Inicio" la lleva a `/home` con sesión activa.
4. **Given** Mateo (que ya tiene StudentProfile del seed), **when** intenta entrar a `/onboarding/welcome` directo, **then** el guard del layout `(onboarding)` lo redirige a `/home`.
5. **Given** Lucía en paso 02 y backend Academic está caído, **when** abre la página, **then** ve mensaje "No pudimos cargar..." + retry, NO un crash ni stack trace.
6. **Given** Lucía completa paso 02 OK pero cierra browser, **when** vuelve a loguearse mañana, **then** el guard ve que tiene StudentProfile y la lleva a `/home` (no re-arranca onboarding).
7. **Given** Lucía submit "Continuar" en paso 02, **when** clickea dos veces antes de que termine la primera, **then** el segundo click no dispara nada (button disabled durante pending) y termina sólo en una creación de StudentProfile.

### Cobertura por capa

- **Unit / vitest**: `features/onboarding/schema.ts` (Zod del form de career) + helpers de cascada (resetear career al cambiar uni).
- **Integration backend**: 4 endpoints nuevos (`GET universities`, `GET careers`, `GET career-plans`, `GET /api/me/student-profiles`). Test happy path + 404 + concurrency 409.
- **E2E Playwright**: full flow `feat-onboarding-happy-path.spec.ts` con Lucía: register → verify → sign-in → 4 pasos → /home con AppShell. Y un negativo: Mateo con profile ya creado redirige a /home si entra a /onboarding.

## Notas de implementación

- **El StudentProfile se crea en paso 02**, no al final del onboarding. Eso permite que un user que abandona después del paso 2 ya tenga su perfil mínimo.
- **Carga de historial diferida**: las US-013 (manual) y US-014 (PDF) son trabajo separado. Este onboarding las "expone" como opciones pero sus implementaciones son US distintas. En este sprint, las dos opciones de carga muestran "Esta opción va a estar disponible pronto" o redirect a una página stub. Solo "Lo cargo después" es path 100% funcional.
- **Sin StudentProfile no se entra a `(member)`**: el guard de `(member)/layout.tsx` chequea que el user tenga un StudentProfile activo. Si no tiene, redirige a `/onboarding/welcome`. Esa lógica es trabajo de este US (acoplada al onboarding).
- **Auth rebuild (US-036)** debería aterrizar antes o en paralelo: el redirect post-sign-in que hoy va a `/home` pasa a `/onboarding/welcome` si el user no tiene StudentProfile. (Ya Done en main, este US lo aprovecha.)

## Dependencies

- **Depende de**: [US-012-b](US-012-b.md) (StudentProfile backend POST endpoint, **Done**) y [US-036](US-036.md) (Auth rebuild a 4 rutas, **Done en main**). Pre-conditions cumplidas.
- **Crea backend nuevo no externo**: 4 endpoints Academic + Identity (`GET /api/academic/universities`, `/careers?universityId=`, `/career-plans?careerId=`, `GET /api/me/student-profiles`). No es dependencia bloqueante porque se construye dentro de esta US (sub-divisible en US-037-b si se prefiere).
