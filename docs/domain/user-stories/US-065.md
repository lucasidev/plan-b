## Como admin, quiero CRUD de Commissions y asignación de teachers para que los alumnos puedan referenciar comisiones reales

Como admin, quiero CRUD de Commission (Subject + AcademicTerm + name + modality + capacity) y asignar teachers con role.

## Acceptance Criteria

### Backend

- [ ] CRUD Commission bajo `/api/admin/subjects/{subjectId}/commissions`:
  - Create con `{ termId, name, modality, capacity?, notes? }`.
  - `UNIQUE(subject_id, term_id, name)`.
  - Validación: `subject.term_kind = term.kind` (coherencia de cadencia).
  - Validación: misma university (`subject.career_plan.career.university = term.university`).
- [ ] CRUD CommissionTeacher bajo `/api/admin/commissions/{commissionId}/teachers`:
  - Add teacher con `{ teacherId, role }` (role: `titular` | `adjunto` | `jtp` | `ayudante` | `invitado`).
  - PRIMARY KEY `(commission_id, teacher_id)`.
  - Validación: `teacher.university = subject.career_plan.career.university`.
- [ ] Requiere `role = 'admin'`.
- [ ] Cambio de modality después de tener inscriptos: rechazar con 409 `academic.commission.has_enrollments`. El admin debe crear una commission nueva con la modality correcta y migrar inscriptos manualmente (post-MVP automatizable).

### Frontend

- [ ] UI grid de comisiones por term + asignación de teachers.

## Sub-tasks

- [ ] Aggregate Commission + child entity CommissionTeacher
- [ ] Domain validators (coherencia universitaria total, ADR-0001)
- [ ] Endpoints Carter
- [ ] UI admin
- [ ] Integration tests: UNIQUE name por subject+term, term_kind mismatch rechazado, teacher de otra university rechazado

## Notas de implementación

- **Coherencia universitaria total**: invariante data-model. `Career.university = Subject.career_plan.career.university = AcademicTerm.university = Teacher.university`. El handler valida cross-table antes de persistir.
- **CommissionTeacher como child entity**: vive en el aggregate Commission. PRIMARY KEY compuesto `(commission_id, teacher_id)` permite múltiples teachers por commission con roles distintos (titular + adjunto + JTP).
- **Capacity opcional**: algunas comisiones no tienen cap formal. Se modela como nullable.
