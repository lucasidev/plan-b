# US-033-i: Recuperación de contraseña (integrated)

**Status**: Done
**Sprint**: S1
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: Medium
**Effort**: L
**UC**: [UC-033](../use-cases/UC-033.md)
**ADR refs**: ADR-0023, ADR-0033, ADR-0034

## Como user que olvidó la contraseña, quiero pedir un link de recuperación y elegir una nueva contraseña sin perder mi cuenta

Como user con cuenta verificada que no se acuerda la contraseña, quiero ingresar mi email en `/forgot-password` y recibir un mail con un link de un solo uso. Al clickearlo, llego a `/reset-password?token=...`, elijo contraseña nueva, y vuelvo a `/auth` con un toast de confirmación. Mis sesiones activas (refresh tokens) se invalidan en el reset para que un atacante con un refresh viejo no siga adentro.

Slice integrated: backend + frontend en un solo PR porque el flow no es usable hasta tener las dos puntas (igual que US-029-i).

## Acceptance Criteria

### Backend

- [x] `POST /api/identity/forgot-password` con body `{ email }`. Siempre 204 (anti-enumeración: misma respuesta exista o no, esté verificado o no, esté disabled o no).
- [x] Si el user existe, está verificado y no está disabled: emite un `VerificationToken` con `Purpose=PasswordReset`, TTL 30 min. Por la regla "un token activo por purpose" del aggregate User (ADR-0033), invalida tokens previos de password-reset si los había. Manda mail con link `{frontend}/reset-password?token={raw}`.
- [x] Si el user no existe, no está verificado, o está disabled: silencio, no manda mail, igual responde 204.
- [x] Rate limit por IP: máximo 5 requests por hora a `/forgot-password`. Excedido devuelve 429 con title `identity.rate_limit.exceeded`. (Implementación de rate limit es nueva, sub-task explícita.)
- [x] `POST /api/identity/reset-password` con body `{ token, newPassword }`.
- [x] Happy path: token válido + password cumple policy → 204. Marca `VerificationToken.ConsumedAt`, hashea nueva password con BCrypt cost 12, persiste en `User.PasswordHash`, revoca **todos** los refresh tokens del user (ver Notas), emite `PasswordResetCompleted` domain event.
- [x] Token inexistente o sin match → 404 con title `identity.verification.invalid`.
- [x] Token expirado → 409 con title `identity.verification.expired`.
- [x] Token ya consumido → 409 con title `identity.verification.already_consumed`.
- [x] Token de un purpose distinto (ej. UserEmailVerification) → 409 con title `identity.verification.wrong_purpose`.
- [x] Password no cumple policy (≥ 12 chars) → 400 con title `identity.password.too_weak`.
- [x] Password idéntica a la actual → aceptar (no validamos against history). Es responsabilidad del user; el flow no se traba.
- [x] User en estado disabled o no verificado → 409 con title `identity.account.disabled` o `identity.account.email_not_verified` respectivamente. Aunque el token sea válido, no se permite reset.
- [x] Integration tests: happy path Lucía, token inexistente, token expirado, token ya consumido, token con purpose distinto, password too_weak, user disabled, user unverified, refresh tokens revocados (login + forgot + reset + reintentar refresh con cookie vieja → 401), rate limit excedido en forgot.

### Frontend

- [x] Página `/forgot-password` (`(auth)/forgot-password/page.tsx`) con form de un solo campo email. Validación cliente Zod (formato email).
- [x] Submit dispara server action `forgotPasswordAction` que llama `POST /api/identity/forgot-password` y al 204 redirige siempre a `/forgot-password/check-inbox?email=<email>` con mensaje "Si el email existe, te mandamos un link". Mismo mensaje exista o no.
- [x] Si el rate limit devuelve 429 muestra toast "Esperá unos minutos antes de pedir otro link" en la misma página, sin redirect.
- [x] Página `/reset-password?token=...` (`(auth)/reset-password/page.tsx`) con form de password + confirm-password. Validación cliente Zod (≥ 12 chars, confirm matches).
- [x] Si el user llega a `/reset-password` ya logueado: el guard `(auth)` redirige a `/dashboard` (comportamiento default por ADR-0023). Caso edge documentado: si llega un atacante con el token de Lucía mientras Lucía está logueada en otro tab, ella no ve el reset; el atacante tampoco, porque el guard del frontend lo manda al dashboard antes de mostrar el form. La revocación de refresh tokens en el reset cierra cualquier sesión vieja igual.
- [x] Submit dispara server action `resetPasswordAction` que llama `POST /api/identity/reset-password`.
- [x] 204 → redirect a `/auth?reset=success`. La página `/auth` lee el search param `reset=success` server-side y muestra un toast/banner "Listo, ya podés ingresar con tu nueva contraseña". Toast desaparece a los 8s o al primer click.
- [x] 404 / 409 (token inválido/expirado/usado/wrong_purpose) muestra error in-form con copy específico + CTA "Pedí un link nuevo" que linkea a `/forgot-password`.
- [x] 400 (password too_weak) muestra error en el campo password.
- [x] 409 (account disabled/unverified) muestra error in-form con copy específico, sin CTA al forgot (no tiene sentido).
- [x] El link "¿Olvidaste tu contraseña?" del sign-in form (placeholder en US-028-f, hoy apunta a `/forgot-password` y rinde 404) ahora funciona.
- [x] Estados de UI: loading, error in-form, error per-field. Copy en español rioplatense.

## Sub-tasks

### Backend

- [x] Domain: `User.RequestPasswordReset(IClock)` que invalida tokens activos del purpose `PasswordReset` (si los había) y emite uno nuevo con TTL 30 min. Reusa `IssueVerificationToken` con `Purpose=PasswordReset`.
- [x] Domain: `User.ResetPassword(rawToken, newPlainPassword, IPasswordHasher, IClock)` que valida el token (los mismos 5 estados que VerifyEmail), valida policy de password, hashea, setea `PasswordHash`, marca token `Consumed`, emite `PasswordResetCompleted` domain event con `UserId` y `ResetAt`.
- [x] `UserErrors`: agregar `PasswordTooWeak` y `VerificationWrongPurpose` con sus codes.
- [x] Application: `RequestPasswordResetCommand` + handler. Lookup user por email, si no aplica devuelve `Result.Success` silencioso (siempre 204).
- [x] Application: `ResetPasswordCommand` + handler. Después del éxito del aggregate, llama a `IRefreshTokenStore.RevokeAllForUserAsync(userId)`.
- [x] Endpoint Carter `ForgotPasswordEndpoint` (mapea Result a 204 / 429 si rate limit).
- [x] Endpoint Carter `ResetPasswordEndpoint` (mapea los 8 códigos a 204 / 400 / 404 / 409).
- [x] Email template `password-reset.html` con copy en español rioplatense, link al `frontend/reset-password?token=...`. Reusa `ISmtpEmailSender` existente.
- [x] **Rate limit nuevo**: `IRateLimiter` interface en `Planb.Identity.Application.Abstractions.Security`. Implementación `RedisRateLimiter` siguiendo Patrón #2 de ADR-0034 (sliding window en Redis con key `identity:ratelimit:forgot-password:{ip}` y TTL 1 hora). Sub-task crea infra de rate limiting por primera vez en el proyecto.
- [x] **Extender `IRefreshTokenStore`**: agregar `RevokeAllForUserAsync(UserId, CancellationToken)`. Implementación usa el índice secundario documentado en `redis-key-patterns.md` (Patrón #1, sección "Índice secundario"): `SMEMBERS identity:refresh-by-user:{userId}` para listar hashes activos, `DEL` cada uno, `DEL` el set. `StoreAsync` debe agregar al set; `RevokeAsync` debe quitarlo. Migration de los flows existentes (sign-in, sign-out) para que mantengan el set actualizado.
- [x] Integration tests cubriendo los 11 escenarios listados en AC backend.

### Frontend

- [x] Feature `features/forgot-password/` con `actions.ts` (server action), `api.ts` (fetcher), `schema.ts` (Zod), `types.ts` (FormState), `components/forgot-password-form.tsx`.
- [x] Feature `features/reset-password/` con la misma estructura (action lee `token` del searchParams y lo manda en el body).
- [x] Página `(auth)/forgot-password/page.tsx` thin wrapper.
- [x] Página `(auth)/forgot-password/check-inbox/page.tsx` con mensaje genérico anti-enumeración.
- [x] Página `(auth)/reset-password/page.tsx` que lee searchParam `token` (Server Component) y pasa al form.
- [x] Toast / banner en `(auth)/auth/page.tsx` que reacciona a `?reset=success`. Renderizado server-side, dismissable client-side.
- [x] Wire del link "¿Olvidaste tu contraseña?" en `sign-in-form.tsx` (ya apunta a `/forgot-password`, solo verifica que la ruta existe ahora).
- [x] Test e2e con Playwright opcional (registrar como deuda si no se hace en el sprint).

## Notas de implementación

- **Anti-enumeración**: alineado con US-028-b/f (login). Forgot siempre 204 sin distinguir email existe / verificado / disabled. Frontend muestra un mensaje genérico siempre. La diferencia se ve solo si el user efectivamente recibe el mail.
- **Email no verificado**: si el user nunca verificó, no procesamos el reset. Vector hipotético: alguien registra el email de Lucía sin verificar; después usa "forgot" para apropiárselo. Cortar en `email_verified_at IS NOT NULL` lo previene.
- **Disabled**: si la cuenta fue deshabilitada (US-068), forgot tampoco procesa. El user disabled no debería recibir camino de vuelta a la app.
- **TokenPurpose ya tiene `PasswordReset`**: el enum `Planb.Identity.Domain.Users.TokenPurpose` ya lo incluye y la columna `purpose` es `varchar(64)` sin CHECK constraint, EF guarda enums por nombre. **No hace falta migration**. Solo agregar la lógica en handlers + tests.
- **Revocación de refresh tokens via índice secundario Redis (Patrón #1 ADR-0034)**: el patrón está documentado en `redis-key-patterns.md` pero no implementado todavía (`IRefreshTokenStore` solo tiene `RevokeAsync` para un token, no `RevokeAllForUserAsync`). Esta US extiende la abstracción para soportarlo. Esto también desbloquea US-068 (disable user) en el futuro.
- **Rate limit middleware nuevo**: no existe rate limiting en el proyecto todavía. Esta US trae la abstracción `IRateLimiter` + `RedisRateLimiter` + el binding del endpoint específico. Dejar la abstracción genérica para que US-021-b (resend verification) la pueda reusar después con su propia key prefix.
- **Idempotencia de password idéntica**: aceptar. No validar contra password actual. Razones: el user ya pasó por el flow correcto (recibió el mail), implementarlo requiere `BCrypt.Verify` antes del hash (costo extra sin beneficio claro), y ningún ADR lo manda.
- **User logueado al entrar a `/reset-password`**: el guard del route group `(auth)` redirige a `/dashboard` si hay sesión activa. Eso cubre el edge case del atacante con token de Lucía + Lucía logueada. La revocación de refresh tokens en el reset cierra cualquier sesión activa igual, así que aunque saliera por algún bug del guard, el atacante quedaría afuera al primer 401 del refresh.
- **Sin mockup**: ni `/forgot-password` ni `/reset-password` están en `docs/design/reference/`. La interfaz se construye reusando los primitivos del design system (`AuthSplit`, `Button`, `TextField`, `PasswordField`) con el mismo split layout que sign-up/sign-in. Decisión de scope: si el resultado se ve coherente con el resto del flow de auth, no hace falta mockup nuevo.
- **`PasswordResetCompleted` domain event**: solo lo emitimos para audit log y métricas. No tiene side-effects de cross-BC todavía. Si en el futuro Reviews quiere reaccionar (ej. invalidar tokens cacheados de "review por user X"), el outbox ya está listo para suscribirlo.
