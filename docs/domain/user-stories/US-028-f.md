**UC**: [UC-028](../use-cases/UC-028.md)

## Como user verificado, quiero un formulario de login con feedback claro de cada estado

Como user con cuenta lista para entrar, quiero `/sign-in` con un form de email + password, manejo de los tres caminos de error (credenciales, no verificado, disabled) con copy específico por caso, y al éxito quedo logueado y redirigido a la home protegida.

Comparte el componente `AuthView` con `/sign-up` (mismas tabs, mismo split layout), siguiendo el mockup. La tab "Ingresar" del AuthView postea a este endpoint.

## Acceptance Criteria

- [ ] Página `(auth)/sign-in/page.tsx` renderiza `AuthView` con tab "Ingresar" activa.
- [ ] Form: email + password, validación mínima cliente (no vacío, password >= 12).
- [ ] Submit dispara server action que llama `POST /api/identity/sign-in`.
- [ ] 200 → redirect a `/dashboard` (member home), las cookies se setearon httpOnly.
- [ ] 401 → muestra "Email o contraseña incorrectos" cerca del form (sin distinguir cuál).
- [ ] 403 `email_not_verified` → muestra "Tu cuenta todavía no está verificada" + link "Reenviar mail" (deshabilitado hasta que US-021-f aterrice).
- [ ] 403 `disabled` → muestra "Tu cuenta fue suspendida. Contactá al equipo de moderación."
- [ ] Copy en español rioplatense.
- [ ] Tab "Crear cuenta" navega a `/sign-up` (Next `<Link>` con prefetch).

## Sub-tasks

- [ ] Componente `AuthView` shared con tabs internos (sign-up + sign-in)
- [ ] Página `(auth)/sign-in/page.tsx` que renderiza AuthView con `mode="signin"`
- [ ] Página `(auth)/sign-up/page.tsx` que renderiza AuthView con `mode="signup"` (cierra US-010-f también)
- [ ] Server action `signInAction` que llama el backend y maneja respuestas
- [ ] Update `lib/session.ts` para reflejar la sesión real
- [ ] Test e2e con Playwright (opcional, post-MVP)
