# US-036: Auth rebuild: 4 rutas separadas

**Status**: Done
**Sprint**: S2 (mergeado en commit `ab73b1a`)
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: Medium
**Effort**: M
**ADR refs**: [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md), [ADR-0019](../../decisions/0019-single-nextjs-app-con-route-groups.md), [ADR-0023](../../decisions/0023-auth-flow-jwt-cookie-layout-guards.md)

## Como member visitante, quiero rutas dedicadas para cada flow de auth para no mezclar verbos distintos en una sola pantalla con tabs

El AuthView que aterrizó en S1 (`/auth` con tabs Sign-up / Sign-in) economizó código pero mezcló dos verbos distintos: registrarse y autenticarse. La sesión de claude-design del 2026-05-02 zanjó que cada flow merece su propia ruta con layout diferenciado, sin tabs.

## Acceptance Criteria

- [ ] Rutas top-level dentro del route group `(auth)`. **El route group no aparece en la URL** (Next.js 15 App Router): la carpeta `(auth)/` solo agrupa layout. Repetir `auth/` adentro sería redundante.
  - `/sign-up`: formulario de registro (carpeta `app/(auth)/sign-up/page.tsx`).
  - `/sign-in`: formulario de login (carpeta `app/(auth)/sign-in/page.tsx`).
  - `/forgot-password`: ya existe (US-033-i). Mantener.
  - `/forgot-password/check-inbox`: ya existe (US-033-i). Mantener.
  - `/reset-password`: ya existe (US-033-i). Mantener.
  - `/verify-email`: ya existe (US-011-f). Mantener.
- [ ] Cada ruta nueva (`/sign-in`, `/sign-up`) es Server Component thin que importa el componente del feature correspondiente.
- [ ] El layout `(auth)/layout.tsx` mantiene el guard de redirect a `/home` si ya hay sesión.
- [ ] El componente `AuthView` con tabs se elimina (deuda explícita de S1 cerrada).
- [ ] Las features `sign-up`, `sign-in`, `forgot-password` ya existen (S1) y no requieren rebuild de su lógica: solo se desacoplan del shell con tabs y se conectan a la nueva ruta.
- [ ] Footer links cross-flow:
  - Sign-up: "¿ya tenés cuenta? Iniciá sesión" → `/sign-in`.
  - Sign-in: "¿sos nuevo? Creá tu cuenta" → `/sign-up`.
  - Sign-in: "¿olvidaste tu contraseña?" → `/forgot-password`.
- [ ] Tests E2E existentes (`forgot-password.spec.ts`, `sign-in.spec.ts`, `sign-up.spec.ts`, `resend-verification.spec.ts`, `sign-out.spec.ts`, `route-guards.spec.ts`) actualizados a las nuevas rutas.

## Sub-tasks

- [ ] `app/(auth)/sign-up/page.tsx`: wrapper que importa `<SignUpForm>` del feature.
- [ ] `app/(auth)/sign-in/page.tsx`: wrapper que importa `<SignInForm>`.
- [ ] Borrar `app/(auth)/auth/page.tsx` (la ruta unificada con tabs).
- [ ] Borrar `components/layout/auth-view.tsx` y refs.
- [ ] Update redirects desde `forwardSetCookies` y server actions para que vayan a las rutas nuevas (`/sign-in`, `/sign-up`).
- [ ] Actualizar tests E2E con nuevos selectors / paths.

## Notas de implementación

- **Backend no cambia**: los endpoints `/api/identity/register`, `/api/identity/sign-in`, `/api/identity/forgot-password`, `/api/identity/resend-verification` quedan como están. Solo el shell HTML cambia.
- **Cookie session no cambia**: el flow de JWT + refresh token se preserva.
- **Layout diferenciado por ruta**: el mockup muestra que sign-in tiene un panel lateral con "última actividad / stats" que no aparece en sign-up. Cada ruta arma su propio layout.
- **Sin double-namespacing**: `app/(auth)/auth/sign-in/page.tsx` daría URL `/auth/sign-in` con el "auth" repetido (route group + segment). Eso es redundante. La convención del proyecto es que el route group `(auth)` agrupa páginas auth-related sin que aparezca en la URL, igual que `(public)`, `(member)`, `(staff)`. `forgot-password`, `reset-password`, `verify-email` ya están top-level por la misma razón.
