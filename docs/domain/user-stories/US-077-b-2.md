# US-077-b-2: Notifications subscribers (handlers Wolverine cross-BC)

**Status**: Backlog
**Sprint**:
**Epic**: [EPIC-05: Sistema de reseñas](../epics/EPIC-05.md)

**Effort**: M (un handler por event, cada uno chico)
**Parent US**: [US-077-b](US-077-b.md)

**ADR refs**: [ADR-0040](../../decisions/0040-notifications-como-bounded-context.md), [ADR-0030](../../decisions/0030-cross-bc-consistency-via-wolverine-outbox.md)

## Como sistema, quiero handlers Wolverine que escuchen integration events de otros BCs y creen Notifications correspondientes, para que cada acción relevante en el sistema se traduzca en una notif persistida y visible en el bell del user

Segundo slice del Notifications BC. Enchufa los disparos: cada integration event que dispara una notif tiene su handler que crea la row.

## Acceptance Criteria

### Handlers por event

Cada handler vive en `Planb.Notifications.Application.EventHandlers.<EventName>NotificationHandler.cs`.

- [ ] **`ReviewResponded`** → notif `kind='review_response'`:
  - Title: "Brandt respondió tu reseña" (formato `{teacherName} respondió tu reseña`).
  - Body: snippet de la respuesta (primeros 100 chars).
  - actorAvatarInit: inicial del teacher (`B` para Brandt).
  - targetUrl: `/reseñas/{reviewId}?highlight=response`.
  - email: NO required (puede activarse en preferences de US-072 futuro).

- [ ] **`ReviewMarkedUseful`** → notif `kind='review_useful'`:
  - Title: "Alguien marcó útil tu reseña" (anónimo).
  - Body: nombre de la materia (ej. "ISW301").
  - targetUrl: `/reseñas?tab=mias&highlight={reviewId}`.
  - email: NO.

- [ ] **`NewReviewInFollowedSubject`** (cuando se implementen follows; deuda visible si no aterrizan en MVP) → notif `kind='review_new_in_subject'`:
  - Title: "Nueva reseña en ISW302" (formato `Nueva reseña en {subjectName}`).
  - Body: rating + snippet.
  - targetUrl: `/mi-carrera/materia/{subjectCode}#latest`.
  - email: NO.

- [ ] **`PlanPeriodOpened`** → notif `kind='plan_period_open'`:
  - Title: "Se abrió la inscripción para 2027·1c".
  - Body: "Activá tu borrador para confirmar materias antes del {deadline}.".
  - targetUrl: `/planificar?tab=borrador`.
  - email: SÍ (`emailStatus = 'pending'`).

- [ ] **`PlanChanged`** → notif `kind='plan_change'`:
  - Title: "Cambio en tu plan: docente nuevo en ISW302" (formato variable según el change).
  - Body: contexto del cambio.
  - targetUrl: `/mi-carrera/materia/{subjectCode}`.
  - email: SÍ.

- [ ] **`UserPasswordChanged`** (de [US-079-i](US-079-i.md)) → notif `kind='system'`:
  - Title: "Tu contraseña se cambió hace un momento".
  - Body: "Si no fuiste vos, escribinos a soporte@plan-b.app.".
  - targetUrl: `/ajustes#seguridad`.
  - email: SÍ. **Crítico**: este es el único caso donde el email es indispensable (security alert).

- [ ] **`ReviewEditRequested`** (de [US-085](US-085.md)) → notif `kind='review_edit_requested'`:
  - Title: "Un moderador te pidió revisar tu reseña de ISW301".
  - Body: el `moderatorMessage` (truncado a 200 chars).
  - targetUrl: `/reseñas/{reviewId}/editar?fromModeratorRequest={reportId}`.
  - email: SÍ.

- [ ] **`PlanMigrationApplied`** (de [US-084](US-084.md)) → notif `kind='system'`:
  - Title: "Tu carrera se actualizó al plan 2023".
  - Body: "Migramos {N} materias automáticamente. Revisá tu historial.".
  - targetUrl: `/mi-carrera?tab=historial`.
  - email: SÍ.

- [ ] **`SystemAnnouncement`** (admins pueden disparar manualmente desde dashboard) → notif `kind='system'`:
  - Title + body + targetUrl vienen del payload del event.
  - email: opcional, depende del payload.

### Resolución de actor + target

- [ ] Cada handler resuelve **el actor** (quien disparó la acción) consultando un query service del BC fuente (`ITeacherQueryService` para teacher info, `ISubjectQueryService` para subject name, etc.). Cross-BC via interfaces, no joins.
- [ ] Si el actor query service no devuelve data (race condition o BC offline): handler degrada gracefully usando IDs (`Title = "Tu reseña de {reviewId} recibió una respuesta"`). Mejor data parcial que crash.

### Wolverine wiring

- [ ] Los handlers se descubren automáticamente por convención Wolverine (sin registro manual).
- [ ] Cada handler es idempotente: si recibe el mismo event dos veces, NO crea dos notifs. Implementación: UNIQUE constraint `(user_id, kind, source_event_id)` o check antes de insert.
- [ ] Retry: Wolverine maneja retries con backoff exponencial out-of-the-box.

### Tests

- [ ] Por cada handler:
  - Integration test: dispatch el event → handler crea row con shape esperado.
  - Test de idempotencia: dispatch del mismo event dos veces → 1 row.
  - Test de graceful degradation: query service mockeado que devuelve null → handler crea notif con título degraded.

## Out of scope

- **Email delivery**: cubierto por [US-077-b-3](US-077-b-3.md). Esta US solo marca `emailStatus = 'pending'` cuando aplica; el job de b-3 lo procesa.
- **Follows de subjects**: `NewReviewInFollowedSubject` queda como deuda si la feature de follows no aterrizó. Documentar.
- **Frontend del panel**: [US-077-f](US-077-f.md).
- **System announcement como feature de admin**: el handler escucha el event; el endpoint admin para disparar `SystemAnnouncement` (form en dashboard ops) es US separada futura.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| Event con `userId` apuntando a un user borrado | Handler crea notif igual (FK lógica, sin constraint). Si se intenta `GET` luego, se filtra en el read API. |
| Event duplicado (race) | Idempotente por UNIQUE `(user_id, kind, source_event_id)`. Segunda insertion es no-op. |
| Query service del BC fuente offline | Degraded notif con IDs. |
| Event con payload malformed | Handler valida shape; si falta campo crítico (userId), logea warning + descarta (no crashea Wolverine). |
| User disabled antes de que el event llegue | Handler crea la notif igual; el read API filtra por `User.disabled = false`, así que no aparece en el bell. |
| Volumen alto (1000 events/min) | Wolverine paraleliza handlers. Hasta 10/min por handler es feasible. |

## Sub-tasks

- [ ] Por cada evento listado: crear handler en `Planb.Notifications.Application.EventHandlers.`.
- [ ] Definir query services cross-BC necesarios:
  - `ITeacherQueryService.GetByIdAsync(teacherId)` → `{ id, firstName, lastName, initials }` (Academic).
  - `ISubjectQueryService.GetByIdAsync(subjectId)` → `{ id, code, name }` (Academic).
  - Otros según lo que necesite cada handler.
- [ ] Migration: agregar `source_event_id` y constraint UNIQUE `(user_id, kind, source_event_id)`.
- [ ] Tests integration por handler.
- [ ] Update `Program.cs` para registrar el módulo si requiere config adicional.

## Notas de implementación

- **Idempotencia con `source_event_id`**: cada integration event de Wolverine tiene un `EventId` único. Persistirlo en la row de Notification permite UNIQUE constraint. Si el event llega de nuevo (retry), el insert falla con `409` y el handler completa OK (idempotente por excepción manejada).
- **Cross-BC query services**: las interfaces declaradas en `Planb.Notifications.Application.Contracts` (o donde corresponda); las impls viven en los BCs respectivos. Inyección por DI. No persistence-coupling.
- **Degradación gracefully**: mejor notif con "{reviewId}" que crashearse y dejar al user sin aviso.
- **Handlers chicos**: cada uno hace una cosa. Sin lógica de negocio compleja. Si crece, splittear.
- **Tests con shared Postgres**: ADR-0027.
