## Como user que olvidó la contraseña, quiero pedir un link de recuperación y elegir una nueva contraseña sin perder mi cuenta

Como user que no se acuerda la contraseña, quiero ingresar mi email en un formulario y recibir un link de recuperación con un token de un solo uso. Al clickearlo, llego a una página donde elijo contraseña nueva y vuelvo al login.

Anti-enumeración: el endpoint de "forgot" siempre responde 204, exista o no el email. El usuario no debe poder distinguir si el email está registrado.

## Acceptance Criteria

### Backend

- [ ] `POST /api/identity/forgot-password` con `{ email }`. Siempre 204 (anti-enumeración).
- [ ] Si el email existe y está verificado: emite un `VerificationToken` con `purpose='password-reset'`, TTL 30 min, hash en DB. Manda mail con link `{frontend}/reset-password?token={raw}`.
- [ ] Si el email no existe o no está verificado: silencia, no manda mail, igual responde 204.
- [ ] `POST /api/identity/reset-password` con `{ token, newPassword }` → 204 si OK, 400 si token expirado / inválido / ya usado.
- [ ] Reset valida la nueva contraseña con la misma policy que register (≥ 12 chars, etc.) y revoca todos los refresh tokens del user (Redis `DEL identity:refresh:user:{id}:*` o equivalente).
- [ ] Token de reset se marca como consumed después del éxito.
- [ ] Rate limit por IP en `forgot-password`: máx 5 req/min para prevenir abuso de envío de mails.

### Frontend

- [ ] Página `/forgot-password` con form de un solo campo (email). Submit → server action → 204 → pantalla "Si el email existe, te mandamos un link" (mismo mensaje exista o no).
- [ ] Página `/reset-password?token=...` con form de password + confirm-password. Submit → server action → si OK, redirect a `/auth?reset=success` con un toast.
- [ ] Si el token llega expirado/inválido en el reset, mensaje de error con CTA para volver a `/forgot-password`.
- [ ] El link "¿Olvidaste tu contraseña?" en sign-in form ya existe (placeholder en US-028-f), apuntar a la ruta nueva.

## Sub-tasks

### Backend

- [ ] Reusar entidad `VerificationToken` (ADR-0033) con nuevo `purpose='password-reset'`. Migration si hace falta agregar al enum / check constraint.
- [ ] `RequestPasswordResetCommand` + handler (silencioso si email no existe)
- [ ] `ResetPasswordCommand` + handler (valida token, hashea password nueva, revoca refresh tokens del user)
- [ ] Endpoints Carter `ForgotPasswordEndpoint`, `ResetPasswordEndpoint`
- [ ] Email template `password-reset.html` consumido por `IEmailSender`
- [ ] Rate limit: extender el middleware existente para cubrir `/forgot-password`
- [ ] Integration tests: happy path, token expirado, token usado dos veces, email inexistente devuelve 204, password policy rechazada, refresh tokens revocados

### Frontend

- [ ] Feature `features/forgot-password/` (action + form + page)
- [ ] Feature `features/reset-password/` (action + form + page que lee `token` de searchParams)
- [ ] Schema Zod compartido entre form y action
- [ ] Toast en `/auth?reset=success` (consumir search param desde el server component)

## Notas de implementación

- **Anti-enumeración**: el backend responde 204 incluso si el email no existe. El frontend muestra el mismo mensaje genérico. Esto es coherente con la política del login (US-028-b/f).
- **Revocar refresh tokens**: cambiar contraseña debe expulsar sesiones activas, sino el atacante con un refresh viejo sigue logueado. Patrón #1 ADR-0034 a través del set tracking por user (si existe; si no, agregar).
- **Reuso del VerificationToken**: ya existe la entidad para email-verify, sumamos un purpose más en el mismo modelo. Evita una tabla nueva.
- **Email no verificado**: si el user nunca verificó, no procesamos el reset (sería un vector para hijack: alguien registra email ajeno sin verificar, después usa "forgot" para apropiárselo). Mejor: silencio.
