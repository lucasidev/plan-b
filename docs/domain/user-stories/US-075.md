# US-075: Member deshabilita su propia cuenta (self-disable)

**Status**: Backlog
**Sprint**: candidato a S3
**Epic**: [EPIC-02 — Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: Medium
**Effort**: S
**ADR refs**: [ADR-0008](../../decisions/0008-roles-exclusivos-profiles-como-capacidades.md), [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md)

## Como member, quiero poder dar de baja mi propia cuenta desde Mi perfil para no depender de un admin si quiero dejar la plataforma

US-068 cubre el flow de moderator/admin deshabilitando a otro user (backoffice). Esta US es el flow paralelo: el alumno deshabilita **su propia** cuenta desde Mi perfil (sección "Zona peligrosa").

## Acceptance Criteria

### Backend

- [ ] `POST /api/users/me/disable` con `{ reason? }` (opcional).
- [ ] Requiere user autenticado con `role='member'` (admins / mods / docentes no usan este flow; tienen procedimientos administrativos).
- [ ] Setea `User.disabled_at = now()`, `disabled_reason = reason ?? 'self-disable'`, `disabled_by = self.id`.
- [ ] Invalida sesión actual: revoca todos los refresh tokens via `IRefreshTokenStore.RevokeAllForUserAsync` (infra de US-033-i).
- [ ] Limpia cookies de sesión en la response (`Set-Cookie` con `Max-Age=0`).
- [ ] Emite `UserDisabled` integration event vía Wolverine outbox (mismo evento que US-068, mismo handler en Reviews / Notifications).
- [ ] Idempotente: self-disable sobre user ya disabled retorna 200 sin tocar.
- [ ] **Diferencia con US-068**: `disabled_by = self.id` (no admin). El audit log distingue self vs admin disable por comparación `disabled_by == userId`.
- [ ] Tests integration: self-disable + login posterior rechazado, sesión invalidada inmediatamente, evento publicado.

### Frontend

- [ ] Sección "Zona peligrosa" al pie de `/mi-perfil` (US-047) con CTA "Dar de baja mi cuenta".
- [ ] Click en CTA abre **modal de confirmación de 2 pasos**:
  1. Texto explicativo: "Vas a deshabilitar tu cuenta. Tus reseñas se ocultan del público. Podés volver a activarla escribiendo a soporte. ¿Seguís?"
  2. Input texto pidiendo escribir "DAR DE BAJA" para habilitar el botón rojo "Confirmar".
- [ ] Confirmar → POST self-disable → redirect a `/auth/sign-in?disabled=true` con banner "Tu cuenta fue deshabilitada. Podés reactivarla escribiendo a soporte".
- [ ] Reason opcional via textarea en el modal (helps soporte si pide reactivación con contexto).

## Sub-tasks

### Backend

- [ ] Comando `SelfDisableUserCommand` + handler.
- [ ] Endpoint Carter `POST /api/users/me/disable`.
- [ ] Reusa el mismo `User.Disable` domain method que US-068 (con `disabled_by = self`).
- [ ] Tests: happy path, idempotente, rol no-member rechazado.

### Frontend

- [ ] `features/disable-my-account/{api.ts,actions.ts,schema.ts,components/{danger-zone,confirm-modal}.tsx}`.
- [ ] Modal con dos pasos.
- [ ] Integración con sign-in para mostrar banner cuando viene `?disabled=true`.

## Notas de implementación

- **Reusa el mismo evento `UserDisabled`** que US-068. El consumidor (Reviews, Notifications) no necesita distinguir self vs admin disable: oculta reviews y deja de enviar notifs igual.
- **Audit log distingue origen** (self vs admin) por `disabled_by`. Eso permite a un futuro dashboard de soporte ver "5 self-disables en la última semana" diferenciado de "1 admin-disable".
- **Re-enable es flow distinto**: el alumno escribe a soporte y un admin lo re-habilita. US-069 (post-MVP) cubre el endpoint `enable`. Para esta US, el flow asume "escribí a soporte" y punto.
- **Modal de 2 pasos**: el "DAR DE BAJA" obligatorio es la fricción mínima necesaria contra clicks accidentales o trolling.
- **Banner en sign-in cuando `?disabled=true`**: el alumno que se da de baja vuelve a sign-in y entiende qué pasó. Si intenta loguearse, el endpoint le devuelve `403 identity.account.disabled` (la lógica existente de US-028-b).
