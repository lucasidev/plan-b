# US-088: Importar plan de estudios desde PDF en onboarding

**Status**: Done
**Sprint**: S3
**Epic**: [EPIC-08: Backoffice de catálogo](../epics/EPIC-08.md)
**Priority**: High
**Effort**: L
**UC**: pendiente (crear UC-088 cuando arranque)
**ADR refs**: ADR-0006, ADR-0017

## Como alumno cuya carrera no está en el catálogo, quiero subir el PDF de mi plan de estudios para poder completar el onboarding y empezar a usar plan-b

Cuando un alumno entra a `/onboarding/career` y elige Universidad → Carrera → Plan, hoy queda colgado si su carrera o plan no figura en el catálogo Academic (la base seedeada cubre solo TUDCS UNSTA + lo que algún admin haya precargado). Esta US destranca ese caso: el alumno hace click en "Mi plan no está, lo subo", sube el PDF que descargó del SIU de su universidad, plan-b lo parsea heurísticamente y crea un `CareerPlan` + sus `Subject`s asociados, marcados como **no oficial** (`isOfficial = false`). El alumno completa el onboarding apuntando a ese plan. Si el plan ya existe pero solo en versión "no oficial" (otra alumna lo subió antes), el sistema sugiere reusarlo en lugar de duplicar.

## Decisiones tomadas (pre-AC)

- **Visibilidad**: el plan subido por alumno queda **público con badge "No oficial"** en el catálogo. Cuando el admin/staff carga el mismo plan desde el backoffice (US separada), entra como `isOfficial = true` y ese gana en duplicación. Resuelve el conflicto crowdsourced vs. fuente de verdad institucional sin perder el path de auto-servicio.
- **UX**: cascadas Universidad → Carrera → Plan siguen siendo el path default en paso 2. Debajo del último dropdown hay link "Mi plan no aparece, lo subo". El click abre el flow de upload (modal o página separada, TBD).
- **Scope del parser**: detecta solo materias (código + nombre + año en plan + cuatrimestre). **Correlativas out of scope** en MVP, las cargará el admin desde el backoffice cuando el plan se valide.

## Acceptance Criteria

### Backend

- [ ] Aggregate `CareerPlanImport` en módulo Academic. Lifecycle Pending → Parsing → Parsed → Failed → Approved (similar a `HistorialImport` de US-014). El estado terminal `Approved` indica que el alumno confirmó el preview y se materializaron `CareerPlan` + `Subject`s.
- [ ] `POST /api/me/career-plan-imports` (autenticado, alumno). Acepta multipart (PDF, 5MB max) o JSON con `rawText` + metadata mínima: `universityId` (del catálogo, ya elegido en paso 2), `careerName` (string libre, el alumno lo escribe), `planYear` (año del plan), `studentEnrollmentYear` (año de ingreso del alumno). Responde 202 con `id`.
- [ ] `GET /api/me/career-plan-imports/{id}` para polling. Devuelve `{ status, error?, parsed_at?, payload? }` con shape similar a US-014.
- [ ] `POST /api/me/career-plan-imports/{id}/approve` para confirmar el preview editado por el alumno. Atómicamente crea el `CareerPlan` (`isOfficial = false`), las `Subject`s (asociadas al CareerPlan), y devuelve el `careerPlanId` para que el alumno lo use en su `StudentProfile` (paso 2 del onboarding).
- [ ] Worker async (Wolverine) que toma el aggregate Pending, extrae texto del PDF (PdfPig), parsea heurísticamente, y deja el aggregate en Parsed con el payload.
- [ ] Parser: detecta líneas con `código + nombre + año/cuatrimestre`. Patrones SIU típicos. Por cada item parseado: `confidence` (Low/Medium/High) + `issues[]`. **No** intenta inferir correlativas (out of scope).
- [ ] Cross-BC contract: nuevo método `IAcademicQueryService.GetCareerPlansByUniversityAsync(universityId)` ya existe o se extiende para incluir `IsOfficial` en el DTO `CareerPlanListItem`. El frontend lo usa para mostrar el badge "No oficial" en las cascadas.
- [ ] `Subject` y `CareerPlan` agregan campo `IsOfficial: bool` (default `false`). Migration. El backoffice futuro (US separada) tendrá un toggle para marcar oficial.
- [ ] Conflict resolution: si ya existe un `CareerPlan` con `(universityId, careerName, planYear)` y `isOfficial = false`, devolver 409 con el `careerPlanId` existente y un mensaje que ofrezca reusarlo. Si existe con `isOfficial = true`, mismo path (el oficial gana). El alumno elige.
- [ ] PDFs encriptados → 422 `academic.plan_import.encrypted`. Payload > 5MB → 413 `academic.plan_import.payload_too_large`. Worker timeout 60s.
- [ ] Audit log: emitir integration event `CareerPlanImported` con `{ careerPlanId, importedByUserId, isOfficial: false, source: 'pdf' | 'text' }`. Lo consume el moderation/admin audit log para visibilidad ("este plan lo subió un alumno, no un admin").

### Frontend

- [ ] En `(onboarding)/career/page.tsx`: debajo de la cascada Plan, agregar link "Mi plan no aparece, lo subo →" que abre `/onboarding/career/plan-import` (o un modal in-place, TBD durante implementación).
- [ ] Feature `features/import-career-plan/` siguiendo el pattern de `import-historial` (US-014): `upload-form.tsx` (PDF o texto), `preview-table.tsx` (materias detectadas editables), `import-flow.tsx` (container con polling).
- [ ] El preview muestra el nombre+año+cuatrimestre del plan en encabezado + tabla editable de materias detectadas. Cada fila: checkbox seleccionar + código + nombre + año en plan + cuatrimestre + confidence badge + issues.
- [ ] Al confirmar: el frontend dispara `/approve`, recibe el `careerPlanId` y vuelve a `/onboarding/career` con ese plan pre-seleccionado. El alumno termina paso 2 normal.
- [ ] Badge "No oficial" visible en la cascada Plan cuando el `CareerPlan` tiene `isOfficial = false`. Color sutil (border accent), texto pequeño.
- [ ] Si el backend devuelve 409 (plan duplicado existente), el frontend ofrece dos botones: "Usar el plan existente" (autoselecciona en la cascada) o "Subir igualmente" (?force=true) — ESTA segunda opción no en MVP, post-MVP.

## Out of scope

Esto NO incluye:

- **Correlativas**: el parser no las detecta. El admin las carga después desde el backoffice. La US-016 (simulación) tolera planes sin correlativas (las muestra como "sin requisitos").
- **Edit del plan después de creado**: el alumno que lo sube no puede editarlo después. Si encuentra un error, contacta soporte. Admin lo edita desde backoffice (US separada).
- **Validación contra fuente externa**: no validamos contra el SIU real. El admin valida ojo a ojo cuando promueve a `isOfficial = true`.
- **Workflow de moderación de planes**: post-MVP. Hoy el plan crowdsourced queda visible para todos los alumnos con badge "No oficial". Si aparece spam/abuso, el admin lo borra desde el backoffice.
- **Múltiples planes de la misma carrera**: si dos alumnos suben el plan 2018 y el plan 2024 de la misma carrera, ambos existen como CareerPlan separados (diferente `planYear`). El alumno elige el suyo.
- **Detección de carrera duplicada**: el alumno tipea `careerName` libre. Si dos alumnos escriben "Ingeniería en Sistemas" y "Ing. Sistemas" para la misma carrera, quedan duplicadas. La consolidación la hace el admin desde backoffice (`Career` merge, US separada).

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| Alumno sube PDF y el parser no detecta ninguna materia | `Parsed` con `payload.items = []` + issue global. El alumno ve "No detectamos materias" + opción de pegar texto o cancelar. |
| Alumno cierra browser mid-upload | El aggregate queda en estado intermedio (Pending/Parsing). El próximo login vuelve a paso 2 onboarding y puede reintentar; el import viejo se queda como huérfano (no bloquea). |
| Alumno sube plan, lo aprueba, completa onboarding y un admin después sube el oficial | El admin puede merger el plan no-oficial con el oficial usando un workflow de backoffice (US separada). Mientras, los alumnos que usan el plan no-oficial siguen usándolo. Al merge, los `EnrollmentRecord` apuntando al plan viejo se reapuntan al nuevo. |
| El PDF es de un plan que ya existe en el catálogo (oficial) pero el alumno no lo encontró en la cascada | El conflict resolution del POST devuelve 409 con el `careerPlanId` del oficial. El frontend lo muestra y el alumno lo elige. No se crea duplicado. |
| Cascada Plan vacía: la universidad existe en el catálogo pero no tiene ningún plan cargado | El link "Mi plan no está, lo subo" es la única opción habilitada. Sin él el alumno queda totalmente bloqueado. |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** Lucía elige UNSTA y su carrera tiene un plan no listado, **when** click "Mi plan no aparece, lo subo" y sube un PDF válido de plan, **then** el preview muestra ≥ 5 materias detectadas con confidence Medium+ y el botón "Confirmar" está habilitado.
2. **Given** Lucía revisa el preview y confirma, **then** el endpoint /approve crea `CareerPlan` (isOfficial=false) + sus `Subject`s, el frontend vuelve a paso 2 con el plan pre-seleccionado, y completa onboarding normalmente.
3. **Given** Mateo (otro alumno) entra a `/onboarding/career` después que Lucía confirmó su plan, **when** elige la misma universidad + carrera, **then** el plan que subió Lucía aparece en la cascada con badge "No oficial" y puede elegirlo sin re-subir.
4. **Given** Lucía sube un PDF encriptado, **when** el worker procesa, **then** el aggregate queda Failed con error claro y el frontend ofrece reintentar o pegar texto.

### Cobertura por capa

- **Domain unit**: `CareerPlanImport` aggregate (transiciones de estado, invariantes).
- **Parser unit**: similar a US-014, fixtures de plans típicos SIU + edge cases.
- **Integration**: 3 endpoints (POST upload, GET status, POST approve), 5MB limit, encrypted PDF, ownership.
- **E2E Playwright**: flow completo onboarding paso 2 con plan no listado → subir PDF → confirmar → completar onboarding.

## Notas de implementación

- **Aggregate separado del HistorialImport**: aunque comparten pattern (upload → worker → preview → confirm), viven en bounded contexts distintos (Academic vs Enrollments). Cero acoplamiento entre ellos.
- **Parser de plan vs parser de historial**: distintos. El de historial parsea filas con código + nota + status + período. El de plan parsea filas con código + nombre + año en plan + cuatrimestre + (opcional) carga horaria. Compartir helpers genéricos (extracción de código, año) en un módulo común si conviene.
- **`isOfficial` per Subject + per CareerPlan**: la flag baja al Subject porque si después el admin acepta el CareerPlan como oficial, las Subjects asociadas también pasan a oficiales en bloque. Cascading manual desde el aggregate, no FK constraint.
- **Conflict resolution alumno vs admin**: si un admin sube el plan oficial post-MVP cuando ya existe la versión no-oficial, el admin tiene una opción "merge" en backoffice. Los EnrollmentRecord migrados (post-merge) requieren cuidado: ese ítem lo aborda la US de backoffice cuando aterrice.
- **No bloqueamos onboarding**: si el plan no parsea, el alumno puede cancelar y elegir "Lo cargo después" (paso 3 actual). Pero el StudentProfile necesita careerPlanId, así que sin plan creado, no avanza paso 2. Trade-off: en pre-MVP siempre hay un plan oficial seedeado de TUDCS UNSTA, así que el caso real es "plan que no es TUDCS". Cuando aterricen más planes oficiales, este path importa menos.
- **Audit log via integration event**: el `CareerPlanImported` event lo consume el módulo de moderation (cuando exista) o un audit log central. MVP: solo emitir, no consumir aún.
