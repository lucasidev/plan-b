## Como admin, quiero CRUD de AcademicTerms para que los EnrollmentRecords y Commissions tengan cuatrimestres a referenciar

Como admin, quiero CRUD de AcademicTerm con year, number, kind, start/end_date, enrollment_opens/closes; el label se computa automáticamente.

## Acceptance Criteria

### Backend

- [ ] CRUD AcademicTerm bajo `/api/admin/universities/{universityId}/terms`:
  - Create con `{ year, number, kind, startDate, endDate, enrollmentOpens, enrollmentCloses }`.
  - `UNIQUE(university_id, year, number, kind)`.
  - CHECK: `end_date > start_date`, `enrollment_closes > enrollment_opens`.
- [ ] `label` se computa al insertar: ej. `2026-C1` (cuatrimestral 1), `2026-B3` (bimestral 3), `2026` (anual).
- [ ] Requiere `role = 'admin'`.

### Frontend

- [ ] UI con calendar picker.

## Sub-tasks

- [ ] Aggregate AcademicTerm con factory que computa `label`
- [ ] Endpoints Carter
- [ ] UI admin
- [ ] Integration tests: label correcto por kind, dates inválidas rechazadas

## Notas de implementación

- **`term_kind` genérico**: `bimestral`, `cuatrimestral`, `semestral`, `anual`. Distintas universidades operan con cadencias distintas. Multi-universidad desde día 1 (ADR-0001) lo requiere.
- **`label` computado en factory**: la lógica de "2026-C1" vs "2026-B3" vive en el aggregate al construirse. Persistir como columna evita recomputar al leer y permite sort por label string-wise.
- **`enrollment_opens` y `enrollment_closes` para gating de inscripciones futuras**: hoy no usado en el dominio (los EnrollmentRecords son input del usuario, no inscripción real), pero la información está para cuando aterrice integración real con sistemas universitarios.
