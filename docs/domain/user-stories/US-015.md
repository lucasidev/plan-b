## Como alumno, quiero modificar una entrada de mi historial para corregir datos o actualizar status

Como alumno, quiero un endpoint PATCH que modifique una EnrollmentRecord propia (corregir datos, actualizar status de "cursando" a "aprobada" al terminar el cuatrimestre), re-validando invariantes y, si la edición invalida una Review existente, emitir un evento que la mande a `under_review` (ADR-0032).

## Acceptance Criteria

### Backend

- [ ] `PATCH /api/me/enrollment-records/{id}` con cambios parciales sobre `status`, `grade`, `approval_method`, `commission_id`, `term_id`.
- [ ] Valida ownership: el `EnrollmentRecord.student_id` coincide con el `StudentProfile` del user.
- [ ] Re-valida todas las invariantes del data-model (mismas que US-013).
- [ ] Si la edición es destructive (cambio a `status = 'cursando'` cuando había una Review publicada anclada al enrollment): emite integration event `EnrollmentRecordEdited` vía Wolverine outbox (ADR-0030).
- [ ] Reviews consume el event y pone la Review afectada en `status = 'under_review'` (ADR-0032).
- [ ] Frontend pide confirmación explícita en destructive edits con copy claro: "Esto va a poner tu reseña en revisión".
- [ ] Retorna 200 con el record actualizado.
- [ ] Idempotencia del PATCH: si el client manda el mismo payload dos veces seguidas, segundo request es no-op (no emite evento, no actualiza updated_at). Implementado con check de equivalencia antes del save.
- [ ] Si el evento de outbox falla post-commit (Wolverine retry exhausted), el state ya cambió en DB. La review afectada queda flagged con `needs_revalidation=true` para una pasada manual o un job de catch-up.

### Frontend

- [ ] Form de edit con confirmación condicional para destructive edits.

## Sub-tasks

- [ ] Comando `UpdateEnrollmentCommand` + handler
- [ ] Endpoint Carter PATCH
- [ ] Detección de "destructive edit": comparar status anterior con nuevo. `aprobada` → `cursando` o `regular` → `cursando` con Review existente es destructive.
- [ ] Integration event `EnrollmentRecordEdited` con outbox durable (Wolverine)
- [ ] Handler en módulo Reviews que invalide (mueve a `under_review`) la review afectada
- [ ] Integration tests: edit no-destructive, edit destructive con review (verificar que la review pasó a under_review post-event)

## Notas de implementación

- **Cross-BC via Wolverine outbox**: ADR-0030. El evento `EnrollmentRecordEdited` no se procesa síncrono; se persiste en el outbox y Wolverine lo entrega a Reviews after commit. Garantiza atomicidad con el update del enrollment, sin acoplar transaccionalmente los dos módulos.
- **Por qué `under_review` y no `removed`**: ADR-0032. La edición destructive no implica que la reseña sea inválida (el alumno puede haberla escrito antes y la cursada cambió por error administrativo). Mover a `under_review` da espacio a moderar caso por caso.
- **Ownership como primer check**: si el `student_id` del enrollment no es el del user, retornar 404 (no 403) para evitar enumerar IDs ajenos.
- **No hay endpoint de DELETE de enrollment todavía**: por simplicidad. Si un alumno quiere "borrar" una cursada, edita a `abandonada`. Si en el futuro hay un caso real para hard-delete, se separa en US.
