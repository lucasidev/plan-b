# US-014: Importar historial desde PDF/texto

**Status**: Done
**Sprint**: 
**Epic**: [EPIC-03: Historial académico](../epics/EPIC-03.md)
**Priority**: Low
**Effort**: L
**UC**: [UC-014](../use-cases/UC-014.md)
**ADR refs**: ADR-0006

## Como alumno avanzado, quiero subir mi historial en PDF o texto para no cargar 30 materias a mano

Como alumno con muchas materias cursadas, quiero subir un PDF (o pegar texto) con mi historial, que un worker async lo procese, resuelva materias contra el catálogo via códigos, me muestre un preview editable de lo que detectó, y al confirmar cree los EnrollmentRecord respetando lo que ya tenía cargado manualmente.

## Acceptance Criteria

### Backend

- [x] `POST /api/me/historial-imports` con multipart (PDF) o body texto + `{ sourceType }` (`pdf` | `text`).
- [x] Crea `HistorialImport` con `raw_payload` JSONB (output del parser) y `status = 'pending'`.
- [x] Worker async (Wolverine handler) procesa: extrae materias, status, grades del raw_payload.
- [x] Resuelve contra `Subject.code` del CareerPlan del profile.
- [x] Por cada match: crea `EnrollmentRecord`. Si ya existía uno con mismo `(student, subject, term)`, respeta el existente y registra la colisión en results (no sobrescribe).
- [x] Reporte de no-resueltos persiste en `HistorialImport.results` (parte del raw_payload, JSONB).
- [x] Cliente puede consultar status: `GET /api/me/historial-imports/{id}` con `{ status, error?, parsed_at?, results }`.
- [x] Si el parser falla, `status = 'failed'`, `error = <msg>`. El alumno reintenta o cae al flujo manual (US-013).
- [x] Tamaño máximo del upload: 5 MB (request body). Excedido devuelve 413 `enrollments.import.payload_too_large`.
- [x] PDFs protegidos con password: rechazar con 422 `enrollments.import.encrypted` y mensaje 'Subí el PDF sin contraseña' al user.
- [x] Multi-page: soportado. El parser concatena texto de todas las páginas antes de parsear.
- [x] Worker timeout: 60s por import. Si excede, marca `HistorialImport.status='failed'` con `errorReason='timeout'`. El user puede reintentar.

### Frontend

- [x] Página de upload + status polling. **Extendido al implementar**: shipped con preview editable + confirm flow (ver "Desviación de diseño" abajo).

## Desviaciones del diseño original

### Preview + Confirm en lugar de auto-create en el worker

**Original**: el worker async crea los `EnrollmentRecord` directamente al parsear.

**Implementado**: el worker termina en estado `Parsed` con el payload de items detectados; el usuario revisa el preview editable (con notas, status, método, term y selección por checkbox) y dispara un `POST /confirm` separado que crea los `EnrollmentRecord` finales.

**Motivo**:
- El parser heurístico no es 100% confiable; la nota o el cuatrimestre pueden quedar mal detectados. Forzar al user a borrar 30 EnrollmentRecord y rehacerlos a mano sería peor UX que un preview.
- Permite overrides per-item antes de persistir.
- Mantiene la garantía "el import respeta lo cargado manualmente" sin ser destructivo.
- El conflict resolution (skip si ya existe la triple) sigue aplicándose en `/confirm`.

**Trade-off**: 1 endpoint adicional (`POST /confirm`), 1 transición extra del aggregate (`Parsed → Confirmed`), y la operación deja de ser "fire-and-forget" para el usuario. Aceptable porque la US la lanza el alumno avanzado consciente.

### Códigos de error

- 400 `enrollments.import.empty_payload` para body vacío (no estaba explícito en la US, se agregó).
- 422 también cubre `enrollments.import.invalid_pdf` (PDF corrupto o no parseable), no solo encryption.

## Sub-tasks

- [x] Aggregate `HistorialImport` con `raw_payload` JSONB
- [x] Endpoint upload (multipart o body texto)
- [x] Endpoint status `GET /api/me/historial-imports/{id}`
- [x] Worker Wolverine handler que procesa pending → parsed/failed
- [x] Parser PDF: PdfPig **1.7.0-custom-5** (la versión 0.1.10+ mencionada en el ticket original no existe en NuGet; el alpha 0.1.9 trae deps transitivas rotas. La 1.7.0-custom-5 es la única estable y open-source que carga en .NET 10).
- [x] Resolver de materias contra `Subject.code`
- [x] Integration tests: PDF válido con 30 materias (cubierto por unit del parser con fixture multi-línea), PDF inválido, texto pegado con typos en códigos, colisión con EnrollmentRecord existente

## Notas de implementación

- **JSONB para raw_payload**: ADR-0006. El shape del payload varía según `source_type` (PDF parseado vs texto pegado), JSONB lo absorbe sin migrations cada vez que mejoramos el parser. El reprocesado es trivial: rehacemos las columnas tipadas desde el JSONB.
- **Async via Wolverine**: el upload responde 202 con el ID; el alumno hace polling al status. Procesar inline bloquearía el request por segundos (PDF parsing).
- **Conflict resolution**: si ya existe un `EnrollmentRecord` para `(student, subject, term)`, el import lo respeta (no sobrescribe). El reporte muestra "MAT201 ya estaba cargada manualmente, salteada". Evita destruir trabajo previo del alumno.
- **Materias no resueltas**: el reporte deja claro qué códigos no se pudieron mapear (Issues per item + Summary con counts por confianza). El alumno puede agregarlas manualmente con UC-013. No fallamos el import por no-resoluciones parciales.
- **Worker timeout**: implementado con `CancellationTokenSource` lincado al token del host. Si excede 60s, captura `OperationCanceledException` del timeout específico (no del host shutdown) y marca el aggregate Failed con mensaje claro. Constante pública `ProcessHistorialImportCommandHandler.ProcessingTimeout`.
- **Parser puro**: el `HistorialParser` no toca DB ni network. Recibe los inputs (subjects + terms) por parámetro. El handler hace los reads cross-BC y pasa los lookups cacheados. Esto deja el parser testeable sin fixtures de DB (15 unit tests cubren los casos típicos del SIU).
