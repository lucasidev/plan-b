# US-053: Ver audit log

**Status**: Backlog
**Sprint**: 
**Epic**: [EPIC-07: Moderación](../epics/EPIC-07.md)
**Priority**: Medium
**Effort**: S
**UC**: [UC-053](../use-cases/UC-053.md)
**ADR refs**: ADR-0031

## Como moderator, quiero ver el historial completo de events de una reseña para tener contexto al moderar

Como moderator, quiero un endpoint que devuelva todas las entries del ReviewAuditLog (projection, ADR-0031) ordenadas por at DESC, con identidad del actor visible (no anonimizado).

## Acceptance Criteria

### Backend

- [ ] `GET /api/admin/reviews/{id}/audit-log` retorna entries del `ReviewAuditLog` para esa review, ordenadas por `at DESC`.
- [ ] Cada entry: `at`, `action` (`published`, `edited`, `reported`, `removed`, `restored`), actor (con identidad visible para moderator, no anonimizado), `changes` (JSONB con before/after diff cuando aplica).
- [ ] Sin paginación inicial (espera bajo volumen por review). Si crece, agregar.
- [ ] Requiere `role IN ('moderator', 'admin')`.

### Frontend

- [ ] Timeline view en la página de detalle de review en backoffice.

## Sub-tasks

- [ ] Projection `ReviewAuditLog` (ADR-0031)
- [ ] Endpoint Carter
- [ ] UI timeline view

## Notas de implementación

- **Audit log como projection, no aggregate**: ADR-0031. Las entries se generan desde events (`ReviewPublished`, `ReviewEdited`, `ReportFiled`, etc.) por un Wolverine handler proyector. No hay write directo, solo lectura. Append-only.
- **Identidad visible al moderator**: ADR-0009. Anonimato es regla de presentación pública, no de moderación. El moderator necesita el actor para entender quién hizo qué.
- **`changes` con before/after solo cuando aplica**: las acciones `published`, `removed`, `restored` no tienen diff (estado terminal change). `edited` y `reported` sí.
