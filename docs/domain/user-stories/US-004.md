## Como visitante anónimo, quiero buscar materia o docente por texto libre para llegar rápido a lo que me interesa

Como visitante anónimo, quiero un único campo de búsqueda que me devuelva resultados mixtos (materias + docentes) ordenados por relevancia, para no tener que navegar el árbol completo.

## Acceptance Criteria

### Backend

- [ ] `GET /api/search?q=<text>` busca en `Subject.name`, `Subject.code`, `Teacher.first_name`, `Teacher.last_name`.
- [ ] Ordenado por relevancia: prioridad a code exacto, luego prefix match, luego full-text rank.
- [ ] Retorna mix de subjects y teachers con campo `type` discriminado (`subject` | `teacher`).
- [ ] Cada item incluye link directo (`/materias/{id}` o `/docentes/{id}`).
- [ ] Sin autenticación requerida.
- [ ] Limit configurable, default 20.
- [ ] Rate limit por IP: 30 req/min sobre `/api/search` para evitar scrape masivo. Reusa IRateLimiter de US-033-i con key prefix `academic:ratelimit:search:{ip}`.
- [ ] Longitud mínima de query: 2 chars. Menos devuelve 400 con title `academic.search.query_too_short` sin contar contra el rate limit.

### Frontend

- [ ] Search bar global (`AppShell` topbar lo expone con `⌘K`, ver US-042-f) que dispara este endpoint.
- [ ] Resultados live con debounce (≥ 200ms).
- [ ] Vista mixta: subjects y teachers visualmente distinguibles, con el tipo claro al lado.

## Sub-tasks

- [ ] Indexes Postgres: `to_tsvector('spanish', ...)` o trigram (`pg_trgm`) según pruebas
- [ ] Endpoint Carter `/api/search`
- [ ] UI search bar con resultados live (Combobox de shadcn/ui)
- [ ] Integration tests con queries variadas: code exacto, typo cercano (trigram), nombre parcial

## Notas de implementación

- **Sin pgvector acá**: la búsqueda semántica de reseñas usa pgvector (ADR-0007) pero esta US es búsqueda léxica simple sobre catálogo. Postgres full-text + trigram alcanzan sobrado y son baratos. Mezclar pgvector acá complicaría sin beneficio.
- **Multi-universidad scoping**: pendiente de decisión. Si el visitante navega bajo una university (`/universidades/unsta/...`), el search debería estar scoped a esa university; si está en home global, debería ser cross-university. Por ahora, default cross-university y filtrar client-side; cuando aparezca el contexto de university en sesión, agregar param `?universityId=`.
- **Multi-universidad scoping (cierre)**: por default search global; con query param `?universityId=` filtra. La UI del header global pasa el universityId del user logueado si lo tiene en session.

## Estado de implementación (S6, 2026-06-23)

Entregado **materia-only**. La rama docente del AC está **diferida a US-063** (no existe entidad Teacher todavía: toda reseña apunta a `PLACEHOLDER_TEACHER_ID`). El contrato del endpoint ya viene discriminado por `type` (`{ type, id, label, sublabel }`, hoy solo `type=subject`), así que injertar docentes cuando aterrice US-063 es aditivo, no rework.

- **Backend**: `GET /api/search?q=&limit=` (Carter, anónimo) con reader Dapper. Ranking code-exacto > prefix > similitud trigram, todo vía `unaccent` (búsqueda insensible a acentos, clave en un catálogo en español; "analisis" matchea "Análisis"). Migración `pg_trgm` + `unaccent` + índice GIN trigram. Mín 2 chars → 400 `academic.search.query_too_short`. 6 integration tests.
- **Frontend**: feature `global-search` (combobox liviano sobre el design system, sin cmdk) que reemplaza el `SearchBar` stub del topbar. Debounce 250ms, navegación por teclado + ⌘K, navega a `/subjects/{id}`.
- **Diferido**: rate limit por IP (AC: 30/min vía `IRateLimiter`), bajo valor pre-deploy; rama docente → US-063.
- **Decisión (anula AC con razón)**: el `href` no viaja en la respuesta (el AC lo pedía); el front lo deriva de `type`+`id`, para no acoplar el backend a rutas de frontend. El `type` discriminado habilita esa derivación.
