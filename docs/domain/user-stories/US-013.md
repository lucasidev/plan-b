# US-013: Cargar historial manual

**Status**: Backlog
**Sprint**: —
**Epic**: [EPIC-03 — Historial académico](../epics/EPIC-03.md)
**Priority**: High
**Effort**: M
**UC**: [UC-013](../use-cases/UC-013.md)
**ADR refs**: ADR-0004

## Como alumno, quiero agregar entradas al historial materia por materia para registrar lo que ya cursé

Como alumno con StudentProfile activo, quiero un endpoint que reciba materia + cuatrimestre + status (+ grade + approvalMethod si corresponde) y persista una EnrollmentRecord, validando invariantes de status/grade y que la materia pertenezca a mi plan.

## Acceptance Criteria

### Backend

- [ ] `POST /api/me/enrollment-records` con `{ subjectId, academicTermId, status, grade?, approvalMethod?, commissionId? }`.
- [ ] Requiere `StudentProfile` activo.
- [ ] Valida `subjectId` pertenece a los subjects del `CareerPlan` del profile (cross-BC vía `IAcademicQueryService.IsSubjectInPlan`).
- [ ] Invariantes del data-model:
  - `status = 'aprobada'` requiere `grade NOT NULL` y `approval_method NOT NULL`.
  - `status = 'regular'` requiere `grade NOT NULL` y `approval_method IS NULL`.
  - `status IN ('cursando','reprobada','abandonada')` requiere `grade IS NULL` y `approval_method IS NULL`.
  - `grade ∈ [0, 10]`.
  - `approval_method = 'equivalencia'` requiere `commission_id IS NULL` y `term_id IS NULL`.
  - `approval_method = 'final_libre'` requiere `commission_id IS NULL` y `term_id IS NOT NULL`.
- [ ] Sin duplicados: `UNIQUE(student_id, subject_id, term_id)`.
- [ ] Retorna 201.

### Frontend

- [ ] Form de carga con campos condicionales: si `status = 'aprobada'` aparecen `grade` + `approval_method`; si es equivalencia, `term` + `commission` se ocultan.

## Sub-tasks

- [ ] Comando `RegisterEnrollmentCommand` + handler
- [ ] Endpoint Carter
- [ ] Value objects para `Grade` (0..10), `EnrollmentStatus`, `ApprovalMethod`
- [ ] Read cross-BC: validar subject ∈ plan vía PublicContracts de Academic
- [ ] UI form con campos condicionales según status
- [ ] Integration tests cubriendo cada combinación válida + casos inválidos (grade fuera de rango, status inconsistente con approval_method, equivalencia con commission)

## Notas de implementación

- **Enrollment guarda hechos, no estados derivados**: ADR-0004. El status del enrollment es lo que el alumno reporta como hecho ocurrido (aprobé, reprobé, regularicé). El estado del grafo (qué materias me habilitan) se calcula al vuelo en el simulador, no se persiste.
- **Recursada como nueva row**: si el alumno reprobó MAT201 en 2024-C1 y la cursa de nuevo en 2025-C1, son dos `EnrollmentRecord` distintas con distinto `term_id`. La unicidad es `(student, subject, term)`, no `(student, subject)`.
- **Equivalencia sin term ni commission**: una equivalencia es un reconocimiento académico, no una cursada. Los CHECK del data-model lo enforcan a nivel app-validation y SQL constraint.
- **Validación de subject ∈ plan**: cross-BC via PublicContract de Academic, no via JOIN ni FK (ADR-0017). Si Academic está caído, el create falla limpio.
