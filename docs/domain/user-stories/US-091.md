> Parent US. Al entrar a sprint se subdivide por capa (`-b` backend del mecanismo, `-f` flujo de subir evidencia + badge, `-i` storage de evidencia si hace falta). El motor es **genérico para alumno y docente**: dos consumidores reales justifican compartirlo (mindset #5).

## Como alumno o docente, quiero poder probar mi condición aportando evidencia para ganar un badge "oficial" que me distinga, de forma opcional

[ADR-0048](../../decisions/0048-oficializacion-de-condicion-opt-in.md): la oficialización de condición es opcional, self-initiated y por evidencia, no por email. Esta US construye el motor: el usuario aporta evidencia, un revisor la evalúa, y se otorga (o no) el badge oficial. Aplica a `StudentProfile` y `TeacherProfile`.

## Acceptance Criteria

### Backend

- [ ] `StudentProfile` y `TeacherProfile` ganan un estado de oficialización independiente del email: `none | pending | official | rejected` (o el shape que se decida), con `officializedAt` y `rejectionReason`.
- [ ] Endpoint para **aportar evidencia** (la condición: archivo/dato), que deja el perfil en `pending`.
- [ ] Flujo de **revisión** (MVP: manual por staff) que aprueba (`official`) o rechaza (`rejected` + motivo).
- [ ] El badge "oficial" se deriva solo del estado `official`.
- [ ] Autorización: aportar evidencia = el propio user; revisar = staff.

### Frontend

- [ ] Flujo para que el user suba su evidencia y vea el estado (`pending` / `official` / `rejected`).
- [ ] El badge "oficial" (alumno/docente) se muestra donde corresponda solo con oficialización aprobada.
- [ ] Panel de revisión para staff (o dentro del backoffice existente).

## Out of scope

- **Verificación automática por afiliación con la universidad**: futura ([ADR-0048](../../decisions/0048-oficializacion-de-condicion-opt-in.md) alt B); el modelo la admite pero no se implementa acá.
- **El copy honesto de alumno**: es [US-090-f](US-090-f.md).
- **Deprecar el email institucional de docente**: se hace en [US-092](US-092.md) consumiendo este motor.

## Sub-tasks

Se desglosan al entrar a sprint (subdivisión -b / -f / -i). Alto nivel:

- [ ] Backend: estado de oficialización en los profiles + endpoints submit/review + storage de evidencia.
- [ ] Frontend: flujo de submit + estado + badge oficial.
- [ ] Revisión staff (backoffice).

## Notas de implementación

- **Motor genérico**: mismo mecanismo para alumno y docente. Evaluar si vive en identity (donde están los profiles) o como capacidad compartida.
- **Evidencia**: qué se acepta (constancia de alumno regular, credencial, etc.) y cómo se almacena (privado, dato sensible) se define al entrar a sprint.
- **MVP manual**: la revisión arranca manual (staff). Automatización (OCR, integración) es deuda futura.
