# US-022-b: Expirar registros no verificados (backend)

**Status**: Done
**Sprint**: S1
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: Medium
**Effort**: S
**UC**: [UC-022](../use-cases/UC-022.md)
**ADR refs**: ADR-0015, ADR-0033

## Como sistema, quiero el dominio + handler que expira registros no verificados después de 7 días

Como aggregate `User`, quiero un método `ExpireRegistration(IClock)` que invalide los `VerificationToken` activos en cascada y emita `UnverifiedRegistrationExpired` domain event. Como handler de aplicación, quiero `ExpireUnverifiedRegistrationsCommand` que use un read service Dapper para encontrar candidates (`email_verified_at IS NULL`, no expirado, `created_at < now() - 7d`) y llame al método del aggregate por cada uno. El sign-in flow se actualiza para tratar a un user `expired_at NOT NULL` como inexistente (anti-enumeración: 401 invalid credentials).

Esta US es **la lógica backend del expirado**. La parte de scheduling (cuándo se dispara el handler) vive en [US-022-i](US-022-i.md).

## Acceptance Criteria

- [x] Domain: `User.ExpiredAt` property (nullable `DateTimeOffset?`).
- [x] Domain: método `User.ExpireRegistration(IClock)`. Precondición: `email_verified_at IS NULL && expired_at IS NULL`. Si la precondición falla retorna `Result<Error>` con `UserErrors.NotEligibleForExpiration` (defensa: el query del handler ya filtra inelegibles, pero el aggregate también valida).
- [x] Domain: `ExpireRegistration` setea `ExpiredAt = clock.UtcNow`, recorre los `VerificationTokens` activos del user y los invalida (`token.Invalidate(clock)`), emite `UnverifiedRegistrationExpired` domain event con `UserId` y `ExpiredAt`.
- [x] `UserErrors.NotEligibleForExpiration` con code `identity.expiration.not_eligible`.
- [x] Application: `ExpireUnverifiedRegistrationsCommand` (sin payload, dispara la corrida) + handler.
- [x] Application: `IIdentityReadService.GetExpiredCandidatesAsync(IClock, CancellationToken)` con Dapper que ejecuta `SELECT id FROM identity.users WHERE email_verified_at IS NULL AND expired_at IS NULL AND created_at < now() - INTERVAL '7 days'`. Devuelve `IReadOnlyList<UserId>`.
- [x] Handler: por cada `UserId`, carga el aggregate, llama `ExpireRegistration`, persiste con `IUserRepository.Update`. Wolverine `[Transactional]` middleware commitea por command (1 transaction per command run, OK para el volumen esperado).
- [x] Handler tolerante a fallos: si un user falla por lock timeout o cualquier excepción, log estructurado con el ID, sigue con el resto. El user fallido se reprocesa en el siguiente run (lo dispara US-022-i).
- [x] Update sign-in flow (`SignInCommandHandler`): cuando el user existe pero `expired_at IS NOT NULL`, retorna `UserErrors.InvalidCredentials` (mismo error que email no existe / password mal). Anti-enumeración: indistinguible de "user no existe".
- [x] Logging estructurado: cantidad procesada por run + IDs (en debug). Si la query devuelve 0 matches, log info "No expired registrations to process at {timestamp}".
- [x] Idempotencia: si el command corre dos veces seguidas, el segundo run no encuentra candidates (la query excluye a los ya marcados con `expired_at NOT NULL`).
- [x] Integration tests: insertar 4 users (uno old non-verified con tokens activos, uno old verified, uno reciente non-verified, uno already-expired), invocar el command vía bus, verificar que solo el primero quedó expired + sus tokens invalidated. Re-invocar, verificar 0 matches. Test del sign-in path: registrar + esperar (modificar `created_at` en DB) + invocar el command + sign-in con ese email retorna 401 invalid_credentials.

## Notas de implementación

- **Cascada sobre `VerificationToken` dentro del aggregate**: cuando un user expira, sus tokens activos se invalidan en el mismo método (no por outbox). Esto asegura que si el job corre antes que un mail con link viejo le llegue al user, el link no funciona (verify-email rechaza con `identity.verification.invalidated`).
- **Por qué `expired_at` y no hard delete**: trazabilidad. Si el mismo email vuelve a registrarse después y termina haciendo abuso, podemos correlacionar con el registro fallido viejo. Hard delete pierde esa señal y los stats de "abandonment rate" del flow de registro.
- **`created_at` se reusa, no agregamos `registered_at`**: la columna existente `created_at` representa el momento de creación del row (que es el registro). No agregamos columna nueva por consistencia con el resto del schema.
- **Sign-in con expired = 401 invalid_credentials**: un user expirado que intenta loguearse debería recibir 401 (no 403 disabled): el row existe en DB pero está "como si nunca hubiera sido". Anti-enumeración indistinguible de "user no existe", consistente con login normal.
- **Decisión de no commitear todo en una transacción global**: el handler hace 1 transaction per command, no por user. Si N users expiran y uno falla a mitad, los anteriores ya quedaron persistidos. Reprocesar el run no rompe (idempotencia). Hacer un transaction global por todos los users juntos sería costoso y la falla de uno bloquearía a todos.
