# US-034: Stats públicas agregadas en hero

**Status**: Backlog
**Sprint**: —
**Epic**: [EPIC-01 — Catálogo público y exploración](../epics/EPIC-01.md)
**Priority**: Low
**Effort**: S
**UC**: [UC-034](../use-cases/UC-034.md)
**ADR refs**: ADR-0009, ADR-0034

## Como visitor, quiero ver en la landing métricas reales del proyecto (universidades, alumnos, reseñas) para entender la escala antes de registrarme

El hero del `/auth` tiene tres números (universidades, alumnos, reseñas) que actualmente están hardcodeados como placeholder. Cuando el producto crezca, esos valores deben reflejar el estado real para dar legitimidad social.

## Acceptance Criteria

### Backend

- [ ] `GET /api/stats/public`. Sin auth.
- [ ] Devuelve `{ universities: number, members: number, reviews: number }`.
- [ ] Counts cross-module: `universities` desde Academic, `members` desde Identity (role='member' && !disabled_at), `reviews` desde Reviews (no soft-deleted).
- [ ] Cache en Redis con TTL 1h (Patrón #3 hot reads, ADR-0034). Key: `stats:public`.
- [ ] Stale-while-revalidate aceptable: si Redis tiene un valor viejo (TTL expirado pero cache hit), se devuelve y se gatilla refresh async.
- [ ] No expone PII ni datos por universidad individual: solo agregados globales.

### Frontend

- [ ] `AuthView` (componente del hero) consume `/api/stats/public` vía RSC + TanStack prefetch + HydrationBoundary.
- [ ] Si el endpoint falla, el hero muestra los valores hardcodeados actuales como fallback (no romper la home por una stat).
- [ ] Reemplaza el const `HERO_STATS` actual en `frontend/src/components/layout/auth-view.tsx`.

## Sub-tasks

### Backend

- [ ] Read model: query Dapper que junta `COUNT(*)` de las tres tablas. Cross-schema OK porque es query de presentación, no FKs.
- [ ] `GetPublicStatsQuery` + handler
- [ ] Endpoint Carter `PublicStatsEndpoint` con cache Redis
- [ ] Strategy de invalidación: TTL 1h, no event-driven (los counts no necesitan ser exactos en tiempo real)
- [ ] Integration test: hit endpoint dos veces seguidas, segundo viene de cache; fallback a DB cuando cache miss

### Frontend

- [ ] `features/landing-stats/api.ts` con `queryOptions` y fetcher
- [ ] Refactor `AuthView` para recibir stats por prop o consumir el query
- [ ] Skeleton / hardcoded fallback si el query falla

## Notas de implementación

- **Cross-schema query es legal acá**: ADR-0017 prohíbe FKs cross-schema y navigation cross-module en EF, pero para reads de presentación con Dapper se permite. La aggregation no crea acoplamiento de dominio, solo de schema visible para reporting.
- **Por qué no event-driven**: los counts cambian pero no es crítico que sean exactos. Re-calcular cada hora con `COUNT(*)` sobre tres tablas es barato y simple.
- **Por qué no por universidad**: agregado global mantiene el anonimato de scale (ADR-0009). Stats por universidad podrían ser un feature post-MVP en el dashboard institucional (EPIC-10).
