## Como user recién registrado, quiero verificar mi email para activar la cuenta y poder loguearme

Como user que se registró pero todavía no se logueó, quiero clickear el link del email que me llegó, llegar a `/auth/verify-email?token=...`, ver confirmación de que mi cuenta quedó activa, y poder ingresar normalmente. Backend consume el `VerificationToken` con `purpose=UserEmailVerification` y marca `email_verified_at`. Frontend renderiza el resultado discriminado por código de error si el token no aplica (inválido, expirado, ya consumido, invalidado).

Esta US es el **parent value-increment** del slice de verificación. Para implementar se subdividió en dos work items que ya están done:

- [US-011-b](US-011-b.md): Verify email backend
- [US-011-f](US-011-f.md): Verify email frontend

## Sub-tasks (subdivision del parent)

### Backend (cubierto por US-011-b, done)

- [x] Endpoint `POST /api/identity/verify-email` con `{ token }` consume el VerificationToken.
- [x] Happy path: 200 con `VerifyEmailResponse(UserId, VerifiedAt)`. Marca `User.EmailVerifiedAt = now()` y `Token.ConsumedAt`.
- [x] 5 códigos de error distinguibles: `identity.verification.invalid` (404), `expired` (409), `already_consumed` (409), `invalidated` (409), `token_required` (400).
- [x] Refactor `EmailVerificationToken` a `VerificationToken` child entity con `Purpose` enum (ADR-0033).
- [x] Integration tests cubren los 5 paths.

### Frontend (cubierto por US-011-f, done)

- [x] Página `(auth)/verify-email/page.tsx` que lee `token` del searchParam y dispara la verificación.
- [x] Component `VerifyEmailResult` con discriminated union por outcome del backend.
- [x] Copy específico por código de error (incluye `already_consumed` como cuasi-éxito).
- [x] CTA al sign-in en happy path.

## Notas de implementación

- **POST con body, no GET con query**: el AC original mencionaba GET. Final: POST para evitar tokens en logs/referer headers. Aplica al frontend igual: el `token` viene en el query string del link del mail pero la página lo manda al backend en el body.
- **5 códigos en lugar de 3**: la implementación distingue `already_consumed` y `token_required` separados (originalmente todo era 400). El frontend usa el title como discriminator de copy.
- **VerificationToken como child entity**: ADR-0033. Vive dentro del aggregate User; las invariantes (un token activo por purpose) las garantiza el aggregate, no el schema.
