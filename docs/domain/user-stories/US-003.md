## Como visitante anónimo, quiero ver un docente con las reseñas donde aparece como docente reseñado para tener contexto al elegir comisión

Como visitante anónimo, quiero la página de un docente con metadata, agregados y reseñas donde figura como docente_reseñado, para usar el dato al elegir comisión.

## Acceptance Criteria

### Backend

- [ ] `GET /api/teachers/{id}` retorna metadata (`first_name` + `last_name` con title case en display, `title`, `bio`, `photo_url`, `university`) + agregados (rating promedio + histograma + conteo de reviews donde el Teacher fue `docente_reseñado_id`).
- [ ] `GET /api/teachers/{id}/reviews` lista paginada de reviews donde el Teacher fue `docente_reseñado_id`, status `published`, anonimizadas.
- [ ] Cada review muestra contexto: subject del enrollment, term, dificultad y textos.
- [ ] Sin autenticación requerida.

### Frontend

- [ ] Página `/docentes/[id]` (RSC) con header (foto, nombre, title, bio) + agregados + lista de reseñas.

## Sub-tasks

- [ ] Read model agregados por teacher (Dapper)
- [ ] Endpoint Carter
- [ ] Página frontend con paginación
- [ ] Integration tests: agregados solo cuentan reviews `published`; reviews donde el teacher participó pero no era el `docente_reseñado_id` no aparecen acá

## Notas de implementación

- **`docente_reseñado_id` no es "todos los docentes de la comisión"**: una review apunta a un solo Teacher como objeto del comentario textual, aunque la commission tenga varios docentes. Las reviews acá son las que apuntan explícitamente al teacher, no las de cualquier commission donde figure.
- **Title case en display, lowercase en storage**: convención del data-model. `Teacher.first_name` y `last_name` se guardan lowercase y se rendarizan en title case desde la API. Ahorra normalización al ingestar y centraliza la presentación.
- **Anonimato igual que en US-002**: el endpoint nunca expone identidad del reseñador. Misma invariante.
