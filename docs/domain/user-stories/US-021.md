## Como user que no recibió el mail de verificación, quiero pedir un reenvío sin volver a registrarme

Como user que se registró pero el mail no llegó (o el token venció antes de clickearlo), quiero un botón "Reenviar email de verificación" en `/auth/check-inbox` (post-registro) y otro en el banner 403 del sign-in cuando intento loguearme sin verificar. Click dispara endpoint anónimo que invalida el token activo si lo había, emite uno nuevo con TTL 24h, manda mail SMTP, y aplica rate limit por IP + anti-enumeración. Yo recibo siempre la misma respuesta gentle "Te lo mandamos de nuevo" exista o no el email.

Esta US es el **parent value-increment** del slice de resend verification. Para implementar se subdivide en dos work items:

- [US-021-b](US-021-b.md): Resend verification backend
- [US-021-f](US-021-f.md): Resend verification frontend

## Sub-tasks (subdivision del parent)

### Backend (cubierto por US-021-b)

- [ ] Endpoint anónimo `POST /api/identity/resend-verification` con `{ email }`.
- [ ] Anti-enumeración: 204 siempre (existe / no existe / verificado / disabled).
- [ ] Domain method `User.RequestVerificationResend(IClock)` que invalida token activo y emite uno nuevo (ADR-0033).
- [ ] Rate limit por IP: 5 req/h. Reusa `IRateLimiter` introducido por US-033-i (Patrón #2 ADR-0034).
- [ ] Domain events `VerificationTokenInvalidated` + `VerificationTokenIssued`.
- [ ] Integration tests con Mailpit.

### Frontend (cubierto por US-021-f)

- [ ] Componente `<ResendVerificationButton email />` reusable.
- [ ] Integración en `/auth/check-inbox` (post-registro) y en el banner 403 del sign-in form.
- [ ] Estados: idle / loading / sent / rate-limited / error.
- [ ] Cooldown visual de 60s post-click.
- [ ] Anti-enumeración: mismo toast siempre.
- [ ] Sin ruta nueva, consolidar en `/auth/check-inbox` existente.

## Notas de implementación

- **Endpoint anónimo**: el user no está logueado al pedir resend. Acepta `{ email }` en el body. El rate limit por IP compensa la falta de identificación de user.
- **Rate limit alineado con forgot-password**: 5/h por IP, mismo cap. Reusar la misma abstracción `IRateLimiter` evita duplicar código de Redis sliding window.
- **Resend en dos lugares**: el path real-user es registrar, cerrar el browser, volver días después, intentar loguearse, recibir 403. Sin botón en el banner 403, queda atorado. Por eso el resend vive tanto en check-inbox como en sign-in.
