# US-092: Migrar la verificación de docente del email institucional a oficialización por evidencia

**Status**: Backlog
**Sprint**:
**Epic**: [EPIC-06: Claim e identidad docente](../epics/EPIC-06.md)
**Priority**: Medium
**Effort**: M (parent; se subdivide -b / -f / -doc al entrar a sprint)
**ADR refs**: [ADR-0048](../../decisions/0048-oficializacion-de-condicion-opt-in.md), [ADR-0033](../../decisions/0033-verification-token-como-child-entity.md), [ADR-0023](../../decisions/0023-auth-flow-jwt-cookie-layout-guards.md)

> Parent US. Depende de [US-091](US-091.md) (el motor de oficialización). Toca backend (deprecar el flujo de email institucional), frontend (el claim/verificación docente) y docs (reconciliar el modelo viejo).

## Como docente, quiero oficializarme aportando evidencia de mi condición (no por email institucional) para responder reseñas con un badge coherente con el modelo nuevo

[ADR-0048](../../decisions/0048-oficializacion-de-condicion-opt-in.md): el email institucional se deprecia como verificación automática de docente. El flujo de claim docente (US-030/031) pasa a usar el motor de oficialización por evidencia de [US-091](US-091.md).

## Acceptance Criteria

### Backend

- [ ] El claim de docente deja de verificar por dominio de email. `TeacherProfile.SubmitInstitutionalEmail` + `verification_method='institutional_email'` se deprecan (decidir si migrar datos de docentes ya verificados o solo cortar el camino nuevo).
- [ ] La oficialización de `TeacherProfile` pasa a derivarse del motor de [US-091](US-091.md).
- [ ] Evaluar `University.institutionalEmailDomains` (US-060): queda sin uso; deprecar el campo + su CRUD, o repurpose.

### Frontend

- [ ] El flujo `teacher-claim` / `teacher-verifier` pasa del email institucional a subir evidencia (motor de [US-091](US-091.md)).
- [ ] El copy de docente deja de prometer verificación por email.

### Docs

- [ ] Reconciliar US-031, `docs/domain/verification-flows.md`, `docs/architecture/data-model.md`, `docs/domain/ubiquitous-language.md` con el modelo nuevo.
- [ ] Marcar US-031 como superada / reescrita según corresponda.

## Out of scope

- **El motor de oficialización**: es [US-091](US-091.md) (esta US lo consume).
- **Afiliación con universidades** como fuente de verificación docente: futura ([ADR-0048](../../decisions/0048-oficializacion-de-condicion-opt-in.md) alt B).

## Sub-tasks

Se desglosan al entrar a sprint. Alto nivel:

- [ ] Backend: deprecar el flujo de email institucional docente + enganchar el motor de US-091 + decisión sobre `institutionalEmailDomains`.
- [ ] Frontend: migrar el flujo de claim / verificación docente.
- [ ] Docs: reconciliar el modelo (US-031, verification-flows, data-model, ubiquitous-language).

## Notas de implementación

- **Migración de datos**: definir qué pasa con los `TeacherProfile` ya verificados por email institucional (si hay): grandfathering a `official`, o pedir re-oficializarse.
- **`institutionalEmailDomains`**: si no queda otro consumidor, deprecar el campo + su CRUD (US-060) o dejarlo documentado como no usado.
