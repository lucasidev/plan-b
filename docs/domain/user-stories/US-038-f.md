## Como member, quiero una pantalla en mi cuenta donde pueda borrar mi cuenta con confirmación clara para ejercer el derecho de supresión sin riesgo de hacerlo por accidente

Frontend slice de [US-038](US-038.md). Cubre la pantalla, el modal anti-accidental, el server action que llama el backend, el sign-out post-delete, el redirect final, y la actualización del spec E2E del onboarding para que use el endpoint nuevo en `test.afterEach` (saca la deuda de cleanup explícita de US-037-f).

Pre-condition: [US-038-b](US-038-b.md) Done en main (endpoint disponible).

## Acceptance Criteria

- [ ] **Pantalla `app/(member)/settings/account/page.tsx`** con sección "Eliminar mi cuenta" + texto explicativo (irreversible, supresión total, sin período de gracia) + CTA "Eliminar mi cuenta".
- [ ] **Modal de confirmación** que se abre al click del CTA. Pide retipear el email del user logueado (anti-accidental) + segundo botón "Eliminar definitivamente" disabled hasta que el email matchee. ESC / click fuera cierra el modal sin acción.
- [ ] **Server action `deleteAccountAction`** en `features/delete-account/actions.ts` ('use server' al tope, único async export). Llama `DELETE /api/me/account` con `apiFetch` (forward de session cookie). Al éxito: hace sign-out (limpia cookies via `forward-set-cookies`) y `redirect('/sign-in?deleted=1')`.
- [ ] **Flash post-delete** en `/sign-in` cuando viene `?deleted=1`: "Tu cuenta fue eliminada. Volvé cuando quieras." Mostrar como banner verde en la parte superior del form, dismiss-able.
- [ ] **Manejo de errores**: si backend devuelve 5xx, modal muestra "No pudimos eliminar tu cuenta. Probá de nuevo o contactanos." sin cerrar el modal. Si devuelve 401 (sesión expirada mid-action), redirect a `/sign-in?expired=1`.
- [ ] **Spec E2E `frontend/e2e/auth/onboarding.spec.ts`**: agregar `test.afterEach` que llama el endpoint para borrar el user descartable creado por el test. Saca la deuda explícita declarada en el comment del spec actual ("La US no limpia los users que crea — eso requiere el endpoint `DELETE /api/me/account` (US-038-f)...").
- [ ] **Tests vitest**: schema del action (mock fetch / next/navigation), component test del modal (validación del email, CTA disabled correcto, flow happy path con action mockeado).

## Out of scope

- **Endpoint backend**: cubre [US-038-b](US-038-b.md).
- **"Mi perfil" como pantalla más grande** ([US-047](US-047.md)): solo agregamos `/settings/account` mínima. La integración con el shell completo de "Mi perfil" llega cuando aterrice US-047. Si US-047 ya está en main al momento de hacer este slice, ubicamos el botón dentro de su layout; si no, página dedicada.
- **Export "download my data" antes de borrar**: post-MVP. Cuando aterrice será US separada que se inserta antes del CTA de delete.
- **Animaciones / micro-interactions del modal**: usable + accesible alcanza, sin ornamentación extra. La animación viene del componente shadcn nativo si lo usamos.
- **Mobile específico**: web-first per [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md). Mobile renderea como degraded UX.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User abre el modal y cierra con ESC sin confirmar | Modal cierra. Cero llamada al backend. Foco vuelve al CTA original. |
| User retipea el email con typo (case sensitivity) | Comparación case-insensitive (.toLowerCase() ambos lados). El email mostrado en el modal aparece tal cual el user lo registró. |
| User retipea el email mal | Submit deshabilitado. Mensaje inline "El email no coincide" visible y vinculado al input por `aria-describedby`. |
| User confirma OK pero el backend devuelve 500 | Modal queda abierto. Banner de error rojo arriba del email input. Botón "Eliminar definitivamente" vuelve a estar disponible para retry. |
| User confirma OK pero el sign-out falla (cookies no se borran) | UI igual hace `redirect('/sign-in?deleted=1')`. Cualquier intento futuro de usar la sesión devuelve 401 (el user ya no existe en backend) y limpia. |
| User abre dos tabs, en una borra cuenta | Tab que borró: redirect a `/sign-in`. Tab residual: la próxima request da 401 → redirect a `/sign-in?expired=1`. |
| JS deshabilitado | El form server action funciona sin JS (es un `<form action={action}>` real). El modal anti-accidental sí requiere JS (es un client component); sin JS el user puede entrar a la página y ver el botón pero no abre modal — esto es aceptable, el delete es destructivo, queremos JS sí o sí. Documentado. |
| Screen reader | El modal usa `role="dialog"` + `aria-labelledby` + `aria-describedby`. Foco se mueve al input email al abrir; vuelve al CTA al cerrar. La validación del email se anuncia via `aria-live="polite"`. |
| Keyboard-only | Tab navega entre inputs y botones; Enter en el input email submit el form solo si el botón está enabled; ESC cierra el modal. |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** Lucía está logueada y entra a `/settings/account`, **when** click "Eliminar mi cuenta", **then** se abre el modal pidiéndole retipear `lucia.mansilla@gmail.com`.
2. **Given** Lucía abre el modal, **when** tipea su email correcto, **then** el botón "Eliminar definitivamente" se habilita.
3. **Given** Lucía confirma con el email correcto, **when** el backend devuelve 204, **then** la app hace sign-out, redirige a `/sign-in?deleted=1`, y el banner verde "Tu cuenta fue eliminada" aparece arriba del form.
4. **Given** un E2E creó un user descartable y completó el flow del onboarding, **when** el `test.afterEach` se ejecuta, **then** la DB queda limpia (no hay user con ese email en `identity.users`).
5. **Given** Lucía abre el modal, **when** tipea un email mal, **then** el botón queda disabled y el error inline es visible.
6. **Given** Lucía confirma OK pero backend devuelve 500, **when** ve el banner de error en el modal, **then** puede reintentar sin cerrar el modal.

### Cobertura por capa

- **Schema test** (vitest): si el action tiene Zod schema (probablemente trivial — solo userId desde sesión), no hace falta test específico.
- **Action test** (vitest + fetch mockeado): redirección a `/sign-in?deleted=1` al éxito; manejo de 401, 5xx; sign-out invocado al éxito.
- **Component test** (vitest + RTL + user-event): modal render, validación del email, CTA disabled correcto, ESC cierra, flow happy path con action mockeado.
- **E2E** (Playwright): el `test.afterEach` del spec onboarding (no test E2E nuevo dedicado al delete; sería overkill — los unit/component cubren el comportamiento).

## Notas de implementación

- **El modal anti-accidental tipea email, no contraseña**: la contraseña ya autenticó la sesión; pedir retipear contraseña es redundante. Tipear el email es la convención de GitHub / Vercel / Linear / mainstream — distinto suficiente del flow normal para forzar atención.
- **Comparación case-insensitive**: emails son case-insensitive por RFC 5321; algunos users escriben mayúsculas / minúsculas distinto. No queremos bloquear por eso.
- **Server action vs API route**: usamos Server Action por consistencia con el resto del frontend (delete-account es side-effect del user, encaja con el patrón). El action llama el endpoint backend con `apiFetch` (que ya hace forward de cookies).
- **Cleanup E2E**: cuando agregamos `test.afterEach`, el spec del onboarding deja de acumular users `*@planb.local` localmente. La regla "infra OK / dominio = endpoints reales" de `docs/testing/conventions.md` se cumple: usamos endpoint real, no helper postgres directo.
