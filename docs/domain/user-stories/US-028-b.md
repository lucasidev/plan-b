## Como user verificado, quiero autenticarme con email + password y recibir una sesión persistente

Como user con cuenta verificada y activa, quiero `POST /api/identity/sign-in` con `{ email, password }` que valide credenciales, emita un par access token + refresh token, los devuelva en cookies httpOnly, y mantenga el refresh token en una revocation list en Redis para poder cerrar sesión limpiamente.

## Acceptance Criteria

- [ ] `POST /api/identity/sign-in` con `{ email, password }`.
- [ ] Credenciales válidas + verified + not disabled → 200 con `{ userId, email, role }` + cookies `planb_session` (access JWT, 15 min) y `planb_refresh` (refresh, 30 días), ambas httpOnly + SameSite=Lax.
- [ ] Credenciales inválidas (email no existe O password mal) → 401 con `title: identity.signin.invalid_credentials` (mensaje genérico, anti-enumeration).
- [ ] Email no verificado → 403 con `title: identity.account.email_not_verified`.
- [ ] Cuenta disabled → 403 con `title: identity.account.disabled`.
- [ ] Refresh token hasheado se guarda en Redis con key `identity:refresh:{hash}` y valor `{userId}`, TTL = 30 días (patrón #1 de [redis-key-patterns](../../architecture/redis-key-patterns.md)).
- [ ] Endpoint `POST /api/identity/refresh` rota el access token usando el refresh cookie (mismo PR para que el flow sea utilizable).
- [ ] Integration tests: happy path con Lucía, password mal con Lucía → 401, login con Paula (disabled) → 403, login con Martín (unverified) → 403, refresh con cookie válida → 200 con access nuevo, refresh con cookie revocada → 401.
- [ ] `DevSeedHostedService` deja las 4 personas seedeadas en Development (ver [`docs/domain/personas.md`](../personas.md)).

## Sub-tasks

- [ ] Domain: `User.Authenticate(plainPassword, hasher, clock)` retorna `Result<User>` o error específico
- [ ] Application: `SignInCommand` + handler + `RefreshCommand` + handler
- [ ] Infrastructure: `IJwtIssuer` + `JoseJwtIssuer` (HS256, secret en `JWT__Secret`), `IRefreshTokenStore` sobre `IConnectionMultiplexer`
- [ ] Endpoint Carter `SignInEndpoint` y `RefreshEndpoint` con cookie set/clear
- [ ] `DevSeedHostedService` con las 4 personas
- [x] Integration tests cubriendo los 4 paths principales

## Notas de implementación

- **Anti-enumeración en 401**: cuando el email no existe o la password está mal, devolvemos el mismo error code y title genérico (`identity.signin.invalid_credentials`). Un atacante no debería poder mapear qué emails están registrados midiendo qué errores le devuelve la API.
- **403 vs 401**: 401 reservado para credenciales malas (auth desconocida); 403 para credenciales OK pero estado de cuenta que bloquea el acceso (unverified, disabled). El frontend usa el `title` del ProblemDetails para discriminar y mostrar la UI apropiada (banner "verificá tu email" vs banner "cuenta deshabilitada").
- **Refresh token con hash en Redis (Patrón #1 ADR-0034)**: guardamos `SHA256(refreshToken)` como key, no el token raw. Si se compromete Redis, el atacante solo ve hashes que no le sirven para autenticarse. La cookie httpOnly mantiene el raw en el navegador, fuera del alcance de XSS.
- **DevSeedHostedService**: las 4 personas (Lucía, Mateo, Paula, Martín) se crean en cada arranque de Development con `IHostedService` que corre antes del web pipeline. En Production no se monta. Esto garantiza que `just dev` siempre tiene un dataset reproducible para probar los 4 paths del login.
- **Refresh endpoint en el mismo PR**: aterrizó junto con sign-in porque sin él el access token de 15 minutos haría que la sesión muriera muy pronto. Splittearlo en otro sprint dejaría sign-in usable solo en demos cortas.
