## Como member con email verificado, quiero crear un StudentProfile asociando carrera + plan + año de ingreso para desbloquear capabilities de alumno

Como member con email verificado, quiero crear un StudentProfile (CareerPlan + enrollmentYear) para desbloquear capabilities de alumno: cargar historial, simular, reseñar.

## Acceptance Criteria

### Backend

- [ ] `POST /api/me/student-profiles` con `{ careerPlanId, enrollmentYear }`.
- [ ] Requiere User authenticated + `email_verified_at NOT NULL`.
- [ ] Valida CareerPlan existe (vía `IAcademicQueryService.CareerPlanExistsAsync`).
- [ ] Valida que no haya `StudentProfile` activo del mismo user para el mismo plan (constraint `UNIQUE(user_id, career_id)` + check `status = 'active'`).
- [ ] `enrollment_year` ∈ [1950, año actual].
- [ ] `enrollmentYear` floor: 2010. Antes la universidad puede no haber existido en la plataforma. AC del backend: rechazar < 2010 con 400 `identity.student-profile.enrollment_year_too_old`.
- [ ] Crea `StudentProfile` con `status = 'active'`.
- [ ] Solo users con `role = 'member'` pueden crear profiles (invariante data-model: staff no tiene profiles).
- [ ] Retorna 201 con `{ id, careerPlanId, enrollmentYear, status }`.
- [ ] Integration tests: happy path con Lucía, duplicate plan retorna 409, year fuera de rango retorna 400, CareerPlan inexistente retorna 404, email no verificado retorna 403.

### Frontend

- [ ] Form de selección University → Career → CareerPlan + enrollment year. Por ahora, US backend-only. Cuando aterrice el flow de UI (post-MVP de auth), AC de frontend se separan.

## Sub-tasks

- [ ] Comando `CreateStudentProfileCommand` + handler en `Planb.Identity.Application`
- [ ] Endpoint Carter `CreateStudentProfileEndpoint`
- [ ] Validar CareerPlan via PublicContracts de Academic (no FK cross-schema, ADR-0017)
- [ ] Domain method `User.AddStudentProfile(careerPlanId, enrollmentYear, clock)` que enforca invariantes
- [ ] Integration tests cubriendo los caminos arriba

## Notas de implementación

- **Profile como capability, no rol**: ADR-0008. El user sigue siendo `role = 'member'`. Crear `StudentProfile` desbloquea capabilities (cargar historial, simular, reseñar) pero no cambia el rol. Un mismo member puede después claimear `TeacherProfile` y tener ambas capacidades.
- **Validación cross-BC sin FK**: ADR-0017. El handler consulta Academic vía `IAcademicQueryService.CareerPlanExistsAsync(careerPlanId)`, no via FK Postgres. Si Academic está caído, el create falla con error claro, no con FK violation.
- **Multiple profiles por user**: el constraint es `UNIQUE(user_id, career_id)`, no `UNIQUE(user_id)`. Lucía puede tener StudentProfile en TUDCS y otro en una segunda carrera. La UI futura tendrá que manejar profile selection cuando haya más de uno.
- **Por qué `enrollment_year` y no fecha completa**: el dominio académico opera por años académicos, no por fechas exactas. El formato del data model es `INT`. UI muestra como year picker.
- **Floor 2010**: elegido como compromiso entre permitir alumnos rezagados (que entraron hace ~16 años) y rechazar valores claramente erróneos.
