# US-038-b: Delete account backend: endpoint DELETE /api/me/account + cascade Identity + audit log + integration event

**Status**: Sprint actual (S2)
**Sprint**: S2
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: Medium
**Effort**: S
**Parent US**: [US-038](US-038.md) (delete account integral)
**ADR refs**: [ADR-0017](../../decisions/0017-persistence-ignorance.md), [ADR-0030](../../decisions/0030-cross-bc-consistency-via-wolverine-outbox.md)

## Como backend del módulo Identity, quiero exponer `DELETE /api/me/account` para que el frontend pueda implementar el self-service de borrado y los E2E tengan cleanup natural

Backend slice de [US-038](US-038.md). Cubre el endpoint, el comando, el handler, la cascade interna del aggregate Identity, la revocación de refresh tokens, el audit log y el integration event que cierran el contrato.

Pre-condition: ninguna. La US arranca desde main como está hoy.

## Acceptance Criteria

- [ ] **Endpoint `DELETE /api/me/account`** con `[Authorize]`, role `member`. Lee el `userId` de la sesión (no del path). Devuelve `204 No Content` al éxito.
- [ ] **Comando `DeleteAccountCommand(UserId)`** + `DeleteAccountCommandHandler` siguiendo el patrón vertical slice (`Planb.Identity.Application/Features/DeleteAccount/`). Handler con métodos estáticos (Wolverine), sin throw para business failures (`Result<T>`).
- [ ] **Cascade interno Identity**: borrar `User` borra `StudentProfile` asociado vía EF cascade (`OnDelete(DeleteBehavior.Cascade)` en la EF config). FK dentro del schema `identity`, no cross-schema.
- [ ] **Revocación de refresh tokens** del user: el handler invoca `IRefreshTokenStore.RevokeAllForUserAsync(userId)` para que cualquier sesión activa caiga al próximo refresh (Redis key del user, ver [ADR-0034](../../decisions/0034-redis-como-cache-y-ephemeral-state.md)).
- [ ] **Tabla `identity.user_deletion_log`** + migration EF Core con: `user_id` (Guid), `email_hash` (string, BCrypt o SHA-256, decisión por costo del lookup futuro), `deleted_at` (DateTime UTC), opcional `deleted_by_user_id` (auto-borrado o admin futuro). El log se inserta en la misma transacción que el delete (Wolverine `[Transactional]`).
- [ ] **Integration event `UserAccountDeleted`** publicado vía Wolverine outbox después del delete OK. Schema mínimo: `UserId`, `DeletedAt`. Otros módulos lo consumen cuando aterricen (Reviews, Enrollments, Moderation). Hoy ningún módulo lo consume: solo se publica para futuro.
- [ ] **Idempotencia**: segundo `DELETE /api/me/account` con la misma sesión devuelve `404 Not Found` (el user ya no existe en `identity.users`), no `500`. La sesión queda inválida desde el primer delete (refresh token revocado).
- [ ] **Tests unit del handler**: cascade chain, integration event publicado, refresh tokens revocados (con NSubstitute para puertos).
- [ ] **Tests integration del endpoint**: `WebApplicationFactory` + Postgres real verifica `204` con auth, row gone en `identity.users`, row inserted en `identity.user_deletion_log`, refresh token entry revocada en Redis (helper de fixture lee la key).

## Out of scope

- **Comportamiento del frontend** (modal, action, redirect post-delete): cubre [US-038-f](US-038-f.md).
- **Handlers en otros módulos** (Reviews, Enrollments, Moderation) que consuman el `UserAccountDeleted` event: cuando aterricen las US de esos módulos. Hoy el event se publica al outbox y queda sin consumers: eso es OK (Wolverine no falla por eso).
- **Soft delete / período de gracia**: hard delete único, sin "30 días para recuperar". Razón en la US integral.
- **Endpoint admin para borrar otra cuenta**: scope cero acá. Solo self-service del dueño de la sesión.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User borra cuenta y otra request del mismo user llega antes del cascade commit | Wolverine `[Transactional]` middleware serializa: la segunda transacción no ve el delete hasta que la primera commitee. Si la segunda es otro `DELETE` → 404. |
| Postgres rechaza la migration (FK existente bloquea) | Migration falla en CI. La FK `identity.student_profiles.user_id → identity.users.id` debe agregarse con `OnDelete(Cascade)` desde su config inicial; si ya existía sin cascade, requiere `ALTER TABLE` explícito en la migration. |
| Wolverine outbox no puede publicar el event (Postgres outbox table down) | El delete del User igual completa (transacción local OK). Wolverine reintenta el publish hasta que Postgres outbox vuelva. La consistencia cross-BC es eventual por diseño ([ADR-0030](../../decisions/0030-cross-bc-consistency-via-wolverine-outbox.md)). |
| Redis no responde cuando se intenta revocar refresh tokens | El handler degrada con warning (per [ADR-0034](../../decisions/0034-redis-como-cache-y-ephemeral-state.md)). El delete completa. La sesión activa cae al próximo intento de refresh porque el user ya no existe en Postgres (sign-in del refresh devuelve 404). |
| Email hash colisión (improbable con SHA-256) | Aceptado. El audit log permite múltiples rows con el mismo hash; el `user_id` los distingue. |
| Sesión con JWT válido pero refresh token revocado | Access token sigue válido hasta su TTL natural (~15 min). Es trade-off documentado: revocación instantánea de refresh tokens, no de access tokens. Si necesitamos kill-switch instantáneo, llega con otra US (lista negra de access tokens en Redis). |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** Lucía está logueada y tiene un StudentProfile, **when** llama `DELETE /api/me/account`, **then** el endpoint devuelve `204`, no existe row en `identity.users` con su id, no existe row en `identity.student_profiles`, y se publica `UserAccountDeleted` integration event al outbox.
2. **Given** Lucía borró su cuenta, **when** un consumer del outbox intenta procesar `UserAccountDeleted`, **then** Wolverine deliver el event al handler con el `UserId` correcto (validable inspeccionando la outbox table).
3. **Given** Lucía borró su cuenta, **when** otro request llega con el JWT viejo intentando hacer sign-in con su email, **then** backend devuelve `401 invalid_credentials` (el user no existe).
4. **Given** Lucía borró su cuenta hace 1 minuto, **when** una request usa el access token todavía válido (TTL no expirado), **then** el endpoint llamado falla con `404` o `401` según el handler busque el user (la mayoría busca → falla con error de "user no existe"). Trade-off documentado.
5. **Given** Lucía borró su cuenta, **when** se inserta una row en `identity.user_deletion_log`, **then** la row tiene `user_id`, `email_hash`, `deleted_at` y NO contiene el email plano de Lucía.
6. **Given** un user no autenticado, **when** llama `DELETE /api/me/account`, **then** devuelve `401`.

### Cobertura por capa

- **Domain unit** (xUnit): test de cascade interno del aggregate `User` (al borrar, también borra `StudentProfile`).
- **Handler unit** (xUnit + NSubstitute): `DeleteAccountCommandHandler` invoca repo, publica integration event vía `IMessageBus`, revoca refresh tokens, inserta audit log. Mockear todos los puertos.
- **Integration** (WebApplicationFactory + Postgres + Redis reales): endpoint con auth, verifica row gone, audit log inserted, refresh tokens revocados, integration event en outbox table.

## Notas de implementación

- **Email hash con SHA-256** (no BCrypt): el audit log no es para autenticar, solo para correlacionar. SHA-256 es determinístico (hash del mismo email → misma key, permite "este user existió y fue borrado" lookup), BCrypt no es. Costo cero de lookup.
- **Wolverine `[Transactional]`** ya está en el patrón estándar (ver handlers existentes). Asegurarse de que el handler lo use, así el publish del integration event y el delete son atómicos vía outbox.
- **Cascade Reviews / Enrollments / Moderation cuando aterricen**: cada US de esos módulos suma el handler del `UserAccountDeleted` event. Convención documentada en `docs/testing/conventions.md` sección "Dominio vs infra" (la US obligada por entidad owned por user).
