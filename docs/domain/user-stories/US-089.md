# US-089: Enabler: persistir modelo completo de reseña (rating, horas, tags, recomendaciones)

**Status**: Sprint Backlog
**Sprint**: S6
**Epic**: [EPIC-05: Sistema de reseñas](../epics/EPIC-05.md)
**Priority**: High (bloquea US-002)
**Effort**: M
**UC**:
**ADR refs**: [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md), [ADR-0005](../../decisions/0005-reseña-anclada-al-enrollment.md), [ADR-0036](../../decisions/0036-testing-pyramid-cross-stack.md)

## Como dev, quiero que el aggregate Review persista el modelo completo del editor (rating general, horas semanales, tags, recomendaciones) para que los crowd insights de US-002 tengan datos reales y no solo difficulty + texto

El editor (US-049/US-048) captura 6 campos: rating general (1-5), dificultad (1-5), horas/semana, texto libre, tags y 2 toggles de recomendación. El `PublishReviewCommand` y el `EditReviewCommand` solo llevan `difficulty + text`; los 4 campos extras se silencian en las server actions. `Review` tampoco tiene propiedades para ellos.

Consecuencia: US-002 (crowd insights: promedio de rating, histograma, "% lo recomienda") no puede computar nada con los datos existentes. Este US es el enabler que elimina el mapping lossy antes de que US-002 consuma el corpus.

El cambio es backend (aggregate, VOs, migration, commands, handlers) + un ajuste mínimo en los actions del frontend (dejar de descartar los campos). No hay cambio de UI.

## Acceptance Criteria

### Backend

- [ ] Nuevo value object `OverallRating` en `Planb.Reviews.Domain.Reviews` (struct readonly, rango 1-5, error `ReviewErrors.OverallRatingOutOfRange`). Mismo patrón que `DifficultyRating`.
- [ ] `Review` aggregate agrega 5 propiedades:
  - `OverallRating` (VO, requerido)
  - `HoursPerWeek` (`int?`, 0-30, opcional)
  - `Tags` (`IReadOnlyList<string>`, requerido, puede ser vacío)
  - `WouldRecommendCourse` (`bool`, requerido)
  - `WouldRetakeTeacher` (`bool`, requerido)
- [ ] `Review.Publish()` acepta los 5 nuevos campos y los asigna al aggregate.
- [ ] `Review.Edit()` acepta los 5 nuevos campos en modo partial update (`null` = sin cambio para los opcionales; para los booleans y el OverallRating, una flag `*Provided` distingue "no enviado" de "enviado con valor actual").
- [ ] `PublishReviewCommand` lleva los 5 campos.
- [ ] `PublishReviewRequest` (HTTP DTO) lleva los 5 campos. `OverallRating`, `WouldRecommendCourse`, `WouldRetakeTeacher` son requeridos; `HoursPerWeek` y `Tags` son opcionales (default `[]`).
- [ ] `PublishReviewValidator` valida: `OverallRating` en [1,5], `HoursPerWeek` en [0,30] si presente, `Tags` contiene solo valores del allowed set (lista centralizada en constante en Application).
- [ ] `PublishReviewCommandHandler` mapea request -> command -> `Review.Publish()`.
- [ ] `EditReviewCommand` lleva los 5 campos con sus flags `*Provided`.
- [ ] `EditReviewRequest` y `EditReviewValidator` análogos.
- [ ] `EditReviewCommandHandler` mapea los 5 campos al `Review.Edit()`.
- [ ] Migration EF Core: 5 columnas nuevas en la tabla `reviews`:
  - `overall_rating SMALLINT NOT NULL` con CHECK `BETWEEN 1 AND 5`
  - `hours_per_week SMALLINT NULL` con CHECK `IS NULL OR BETWEEN 0 AND 30`
  - `tags TEXT[] NOT NULL DEFAULT '{}'`
  - `would_recommend_course BOOLEAN NOT NULL`
  - `would_retake_teacher BOOLEAN NOT NULL`
- [ ] `ReviewConfiguration` EF Core: mapear los 5 campos nuevos (smallint para los VOs de rango, array de text para tags, bool para booleans).
- [ ] Tests de integración: happy path de publicar incluye los 5 campos y verifica que se persisten y devuelven. Edit parcial de los nuevos campos. Valores fuera de rango devuelven 400.

### Frontend

- [ ] Server action `publishReview` deja de silenciar `rating`, `hoursPerWeek`, `tags`, `wouldRecommendCourse`, `wouldRetakeTeacher`: los incluye en el body del `POST /api/reviews`.
- [ ] Server action `editReview`: manda `overallRating` solo cuando el alumno lo seteó (`rating >= 1`). NO manda `hoursPerWeek`, `tags`, ni las recomendaciones (ver nota de implementación: el editor en modo edit todavía no carga los valores persistidos, así que mandar los defaults pisaría datos reales). El backend soporta los 5 en partial-update; el frontend los completará cuando el read side cargue la reseña (US-002).
- [ ] No hay cambio de componentes ni de UI: los valores ya llegan al action desde el form, solo dejan de descartarse.

## Out of scope (intencional)

- [ ] Crowd insights computados: eso es US-002. Este US solo persiste los campos.
- [ ] Exposición de los campos en el browse feed o en la review card: los Dapper query services del browse no necesitan los campos nuevos aun. Se agregan cuando US-002 los consuma.
- [ ] Validación del allowed set de tags en el frontend: ya existe en el componente de US-048. El backend valida igual; no duplicar lógica mas alla de lo ya presente.

## Notas de implementación

- **Migration default para rows existentes**: pre-prod; datos de dev. La migration puede usar `DEFAULT 3` para `overall_rating` (mitad de rango, marcador de "legacy sin dato real"), `DEFAULT TRUE` para los booleans y `DEFAULT '{}'` para tags. Las rows existentes no tienen valor real para esos campos y no afectan US-002 si se filtra por fecha o si US-002 ignora rows con `overall_rating` usando un valor sentinel (a definir en US-002).
- **Allowed set de tags**: la constante ya vive en el frontend (US-048). Centralizarla en `Planb.Reviews.Application/Constants/AllowedTags.cs` como `string[]` para que el validator la use sin duplicar la lista.
- **`Tags` como `TEXT[]` en Postgres**: Npgsql nativamente mapea `string[]` / `List<string>`. En `ReviewConfiguration` usar `.HasColumnType("text[]")`. No requiere paquete adicional.
- **OverallRating vs DifficultyRating como el mismo VO**: son rangos identicos (1-5) pero semanticamente distintos. Reusar `DifficultyRating` para `OverallRating` ahorraría un archivo pero confundiria el lector ("la dificultad de la materia" vs "el rating general que le doy"). Crear el VO separado.
- **Patrón de `Edit()` para booleans**: los booleans no son nullable, entonces `null` no puede codificar "no enviado". El command los lleva como `bool?` (null = no provisto); `OverallRating` como `int?` y `Tags` como `IReadOnlyList<string>?` (null = no provisto); `HoursPerWeek` usa una flag `HoursPerWeekProvided` separada porque null es un valor válido (el alumno limpió el campo). El endpoint distingue "no provisto" leyendo `JsonElement` (key ausente vs presente).
- **Edit action del frontend, scope acotado (decisión que anula con razón el "manda los 5" literal del AC)**: en modo edit el editor todavía NO carga los valores persistidos de `hoursPerWeek` / `tags` / recomendaciones (eso es read side, US-002), así que el draft tiene los defaults del editor, no el estado real. Mandar esos defaults en el PATCH pisaría datos reales del alumno (data loss silencioso). Por eso el edit action manda solo lo que el editor representa fielmente: `difficulty`, `text`, y `overallRating` cuando el alumno realmente eligió uno (`rating >= 1`). El backend soporta los 5 en partial-update; cuando US-002 hidrate el editor con la reseña persistida, el action puede mandarlos todos. El publish action sí manda los 5 (ahí el input es real).
