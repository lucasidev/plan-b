# US-022-i: Expirar registros no verificados (infra: migrations + scheduling)

**Status**: Sprint backlog
**Sprint**: S1
**Epic**: [EPIC-02 — Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: Medium
**Effort**: XS
**UC**: [UC-022](../use-cases/UC-022.md)
**ADR refs**: ADR-0015

## Como sistema, quiero las migrations + cron que materializan el expirado en DB y lo disparan diariamente

Como infraestructura del expirado, quiero las dos migrations EF (agregar columna `expired_at`, reemplazar el unique constraint sobre email por uno parcial) y la registración Wolverine `[ScheduledJob("0 3 * * *")]` que dispara el `ExpireUnverifiedRegistrationsCommand` (definido en [US-022-b](US-022-b.md)) diariamente a las 3 AM UTC.

Convención de sufijo: `-i` acá significa "infra/scheduling", siguiendo la excepción documentada en STATUS.md (mismo uso que US-F03-i, US-F04-i). NO significa "integrated".

## Acceptance Criteria

- [ ] Migration `AddExpiredAtToUser`: agregar columna `expired_at TIMESTAMPTZ NULL` a `identity.users`. Sin default. Sin backfill (todas las rows existentes quedan NULL = activas).
- [ ] Migration `ReplaceUserEmailUniqueWithPartialIndex`: drop del unique constraint actual sobre `email` + `CREATE UNIQUE INDEX user_email_active ON identity.users (email) WHERE expired_at IS NULL`. Permite que el email vuelva a estar disponible cuando el user previo expira sin perder la row vieja.
- [ ] El `ExpireUnverifiedRegistrationsCommand` (creado en US-022-b) queda registrado con Wolverine `[ScheduledJob("0 3 * * *")]` (cron 3 AM UTC, daily).
- [ ] Si por algún motivo Wolverine no expone `[ScheduledJob]` directamente, configurar el schedule en `Program.cs` via `opts.Policies.ScheduledJobs(...)` o equivalente. El cronograma final es `0 3 * * *`.
- [ ] Integration test que confirma que el job se ejecuta cuando se avanza el tiempo (test fixture con `IDateTimeProvider` mockeable que simula 3 AM UTC + verificación de que el handler corrió).

## Sub-tasks

- [ ] Migration EF Core `AddExpiredAtToUser` (módulo Identity).
- [ ] Migration EF Core `ReplaceUserEmailUniqueWithPartialIndex` (drop existing unique, create partial unique).
- [ ] Wolverine: agregar `[ScheduledJob("0 3 * * *")]` al `ExpireUnverifiedRegistrationsCommand` (o configurar via `opts.Schedule(...).Daily()` en `Program.cs`).
- [ ] Integration test del scheduling: usar test host con clock fake, avanzar a 3 AM UTC, verificar que el handler corrió (ej. checkear logs estructurados o un test side-effect).
- [ ] Documentar en `STATUS.md` (sección de schemas Postgres) que `users.expired_at` existe y que el unique de email es parcial.

## Notas de implementación

- **Wolverine `ScheduledJob` ganó sobre `IHostedService`**: ya está configurado con PostgreSQL persistence (líneas 77-83 de `Program.cs`), maneja retry y outbox, y mantiene los scheduled jobs en la misma infra que el resto del messaging. `IHostedService` con `Timer` requiere boilerplate adicional (cancellation, error handling, scope) que Wolverine ya resuelve.
- **3 AM UTC para minimizar contención**: el job hace UPDATE de N rows (vía el handler de US-022-b). Correrlo en horario bajo evita pelearse con el tráfico de auth real. Si el sistema crece a multi-región, reconsiderar.
- **Migration con drop + recreate del unique**: el unique constraint existente sobre `email` no permite duplicates entre rows expirados. Hace falta dropearlo y reemplazarlo por el parcial. La migration tiene que ser cuidadosa con el orden: drop primero, create después, en una sola transacción (default de EF Core).
- **Sufijo `-i` como infra**: convención excepción documentada en STATUS.md. La US-022 (parent) se decompone en US-022-b (lógica) + US-022-i (infra/scheduling). Aterrizan en un mismo PR (no son independientemente útiles), pero el split existe para tracking.
