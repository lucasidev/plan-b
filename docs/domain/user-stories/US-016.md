# US-016: Simular inscripción

**Status**: Backlog
**Sprint**: —
**Epic**: [EPIC-04 — Planificación de cuatrimestre](../epics/EPIC-04.md)
**Priority**: High
**Effort**: L
**UC**: [UC-016](../use-cases/UC-016.md)
**ADR refs**: ADR-0003, ADR-0004

## Como alumno, quiero probar combinaciones de materias para el próximo cuatrimestre para decidir cuáles cursar antes de inscribirme

Como alumno, quiero el simulador: un endpoint que liste materias disponibles (cumplen correlativas para_cursar) y bloqueadas (con razón) según mi historial, y otro que evalúe una combinación devolviendo carga horaria, dificultad ponderada e histograma de resultados de combinaciones similares cursadas por otros.

## Acceptance Criteria

### Backend

- [ ] `GET /api/me/simulator/available?termId=` retorna lista de materias del CareerPlan con flag `disponible` o `bloqueada`. Para bloqueadas, incluye `blockedBy` con la lista de Prerequisites no cumplidos (`subjectId`, `subjectCode`, `type`).
- [ ] "Disponible" = todas las correlativas `para_cursar` están en estado `regular`, `aprobada` (o `equivalencia`) en el historial del alumno.
- [ ] `POST /api/me/simulator/evaluate` con `{ subjectIds[], termId }` retorna métricas:
  - Carga horaria total (suma de `weekly_hours` y `total_hours`).
  - Dificultad promedio ponderada por count de reviews por subject.
  - Histograma de combinaciones similares cursadas por otros alumnos con sus tasas (% aprobados, % abandonos).
- [ ] No persiste nada: solo computación de read models.
- [ ] Rechaza el evaluate si alguna materia del subset está `bloqueada`, con detalle de qué correlativa falta.
- [ ] Requiere `StudentProfile` activo.

### Frontend

- [ ] Página `/simulador` con selector de term + grid de materias disponibles/bloqueadas + panel lateral con métricas en vivo.

## Sub-tasks

- [ ] Domain service de correlativas (DAG check, evaluación de cumplimiento)
- [ ] Read model "available subjects" con Dapper que cruza Subjects del plan + EnrollmentRecords del alumno
- [ ] Read model de combinaciones similares (histograma cross-alumno)
- [ ] Endpoint `/available`
- [ ] Endpoint `/evaluate` con métricas agregadas
- [ ] UI simulador con selección live
- [ ] Integration tests: alumno con historial parcial ve disponibles correctas; combinación con bloqueada falla evaluate; alumno sin historial ve solo materias de 1° año

## Notas de implementación

- **Solo `para_cursar` para "disponible"**: ADR-0003 distingue dos tipos. Para "puedo inscribirme", basta con que el alumno haya regularizado o aprobado la previa. `para_rendir` solo aplica al rendir final, no a la inscripción.
- **No persiste**: ADR-0029. La simulación es efímera por default. Persistencia (US-023) es una feature aparte, premium reward (ADR-0028).
- **Cross-BC heavy**: el simulador cruza Academic (subjects, prerequisites, term) con Enrollments (historial del alumno) con Reviews (rating promedio, histograma). Read models con Dapper saltando schemas, sin FKs ni navigation cross-module (ADR-0017). El acoplamiento es a nivel SQL del read model, no del dominio.
- **Histograma de combinaciones**: complejo. Considerar materializar en un read model con event-driven updates si el endpoint se vuelve lento. MVP puede empezar con SQL directo y medir.
