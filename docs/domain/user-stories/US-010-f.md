## Como visitante, quiero un formulario de signup en la app para crear mi cuenta sin tocar curl

Como visitante, quiero el modo "Crear cuenta" del `AuthView` en `/auth` con form de email + password + confirm, validación cliente, llamada al backend de register existente, manejo de errores legible (409 "email ya registrado", 400 por campo) y al éxito redirección a la pantalla post-success que me indica revisar el correo.

Comparte el componente `AuthView` con el modo "Ingresar" (mismas tabs locales, mismo split layout) per el mockup. Aterriza en el mismo PR que [US-028-f](US-028-f.md) (login frontend) porque comparten componente.

## Acceptance Criteria

- [x] Página `(auth)/auth/page.tsx` renderiza `AuthView`. Soporta deep-link `?mode=signup` para abrir directo en la tab "Crear cuenta".
- [x] Form: email + password + confirm. Validación cliente con Zod (formato email, password ≥ 12, confirm matches).
- [x] Submit dispara server action `signUpAction` que llama `POST /api/identity/register`.
- [x] 201 → redirect a `/auth/check-inbox?email=<email>` con instrucciones de revisar mail.
- [x] 409 → muestra "Ya existe una cuenta con ese email" cerca del input email.
- [x] 400 → muestra el mensaje de validación del campo correspondiente (email / password).
- [x] Estados de UI: loading durante submit, button disabled.
- [x] Estilo usa primitivas del design system (`AuthSplit`, `Button`, `DisplayHeading`, `Lede`, `TextField`, `PasswordField`).
- [x] Tab "Ingresar" del switcher cambia el modo localmente (`setMode('signin')`), sin navegación. Per la convención del frontend (`frontend/CLAUDE.md`): una sola ruta `/auth` con switcher local; el backend mantiene endpoints separados.
- [x] Copy en español rioplatense.

## Notas de implementación

- **Single `/auth` route con switcher local**: el primer intento dividía en `/sign-up` + `/sign-in` con `<Link>` para el switching. Lo descartamos: switcher local con `setMode()` es más rápido (sin re-render server-side) y evita drift visual. Documentado en `frontend/CLAUDE.md`. El backend mantiene endpoints separados (sign-in / register / verify-email) sin cambios.
- **AuthView shared con sign-in**: aterrizó en el mismo PR que US-028-f porque las tabs internas comparten layout (`AuthSplit`), forms con misma estructura, y el switcher entre modos es local (no navega). Tener un solo componente evita drift visual entre las dos pantallas.
- **Server action en lugar de fetch desde cliente**: per ADR-0023, el cookie `planb_session` es httpOnly y solo se setea desde el server. La server action recibe el FormData, llama al backend, y reenvía los `Set-Cookie` headers al user-agent vía `forwardSetCookies` (lib helper).
- **Check-inbox como página separada**: post-201 redireccionamos a una ruta nueva (no toast en el form) para que el user tenga un destino claro y pueda volver al link del mail desde cualquier pestaña sin perder estado.
- **Validación cliente Y server**: Zod schema compartido entre `signUpAction` (server) y el form (client). El form valida UX-eager; el action revalida por seguridad y para los casos donde el cliente está roto o desactualizado.

## Diferencias intencionales con el mockup

El mockup de `docs/design/reference/components/screens.jsx` tiene cosas que decidimos **no replicar** acá:

- **Gate `email.endsWith('@unsta.edu.ar')`**: el mockup gateea el registro a emails institucionales UNSTA. La plataforma es multi-universidad (ADR-0001), gatear por dominio iría en contra de eso. Cualquier formato de email válido se acepta; la verificación de afiliación institucional se hace después, fuera del flow de registro.
- **Botón "Continuar con Google"**: OAuth está fuera del MVP.
- **Campo "name"**: el backend `RegisterUser` toma solo email + password. El display name pertenece a `StudentProfile`, que aterriza en F3.
- **Checkbox "acepto términos"**: no hay términos publicados todavía.
- **Email hint inline**: el mockup muestra un hint en vivo ("Email UNSTA verificado" / "Tiene que terminar en @unsta.edu.ar"). Atado al gate institucional descartado.
