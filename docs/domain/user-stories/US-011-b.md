# US-011-b: Verify email backend

**Status**: Done
**Sprint**: S1
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: High
**Effort**: S
**UC**: [UC-011](../use-cases/UC-011.md)
**ADR refs**: ADR-0033

## Como user recién registrado, quiero un endpoint que consuma mi token de verificación para activar mi cuenta

Como user recién registrado, quiero que el sistema acepte el token del link de verificación, lo valide (no expirado, no consumido, no invalidado), marque mi `EmailVerifiedAt = now()` y responda con confirmación, de forma idempotente: doble click sobre el link no rompe la UX y el frontend lo trata como cuasi-éxito.

## Acceptance Criteria

- [x] `POST /api/identity/verify-email` con body `{ token }` consume el `VerificationToken` con `purpose=UserEmailVerification`.
- [x] Happy path → 200 con `VerifyEmailResponse(UserId, VerifiedAt)`. Marca `User.EmailVerifiedAt = now()` y setea `ConsumedAt` en el token. Atómico dentro del aggregate User.
- [x] Token inexistente o sin match → 404 con title `identity.verification.invalid`.
- [x] Token expirado (TTL vencido sin consumir) → 409 con title `identity.verification.expired`.
- [x] Token ya consumido (segunda llamada con el mismo token, ej. doble click) → 409 con title `identity.verification.already_consumed`. El frontend lo trata como cuasi-éxito ("ya verificaste").
- [x] Token invalidado (alguien pidió resend, ver US-021-b) → 409 con title `identity.verification.invalidated`.
- [x] Token vacío o malformado → 400 con title `identity.verification.token_required` (ValidationException de FluentValidation).
- [x] Como parte del slice, `EmailVerificationToken` queda refactorizado a `VerificationToken` child entity de `User` con `Purpose` discriminator (`UserEmailVerification`, `TeacherInstitutionalVerification`) per ADR-0033.
- [x] Integration tests cubren: happy path con Martín, doble click (segunda llamada retorna `already_consumed`), token expirado, token inválido, token invalidated por resend.

## Sub-tasks

- [x] Refactor `EmailVerificationToken` a `VerificationToken` con `Purpose` enum
- [x] Migration `RefactorVerificationTokensAsChildEntity` (renombra y agrega columna `purpose`)
- [x] Domain method `User.ConsumeVerificationToken(rawToken, clock)` + `UserErrors` con los 5 códigos de la tabla AC
- [x] Comando `VerifyEmailCommand` + handler retornando `Result<VerifyEmailResponse>`
- [x] `VerifyEmailRequest` (token), `VerifyEmailResponse` (UserId + VerifiedAt), `VerifyEmailValidator` (FluentValidation)
- [x] Endpoint Carter `VerifyEmailEndpoint` mapeando `Result<T>` a HTTP status (200 / 400 / 404 / 409)
- [x] Integration tests cubriendo los 5 paths: happy + already_consumed + expired + invalid + invalidated

## Notas de implementación

- **`POST` con body, no `GET ?token=`**: el AC original mencionaba `GET`. La implementación final usa `POST /api/identity/verify-email` con body para evitar que el token quede en logs de servidor o referer headers.
- **200 con body, no 204**: la respuesta incluye `{ userId, verifiedAt }` para que el frontend tenga el timestamp exacto y pueda mostrar UI de confirmación. El AC original pedía 204 (no body) pero la implementación final agrega el body por UX. Doc actualizado para reflejar la realidad.
- **VerificationToken child de User**: ADR-0033. Vive dentro del aggregate, no como aggregate independiente. El método `User.ConsumeVerificationToken(rawToken, clock)` encapsula la lógica de validación y mutación atómica.
- **5 códigos distinguibles, no 3**: el AC original juntaba "invalid / expired / invalidated" como 3 errores 400. La implementación distingue 5 (suma `already_consumed` y `token_required`) y mapea cada uno a su HTTP status correcto (400 / 404 / 409). El frontend (US-011-f) usa el title como discriminator de copy.
