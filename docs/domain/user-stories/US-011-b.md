## Como user recién registrado, quiero un endpoint que consuma mi token de verificación para activar mi cuenta

Como user recién registrado, quiero que el sistema acepte el token del link de verificación, lo valide (no expirado, no consumido), marque mi `EmailVerifiedAt = now()` y responda 204 (o redirect a UI de confirmación), de forma idempotente: doble click no rompe.

## Acceptance Criteria

- [ ] `POST /api/identity/verify-email` con body `{ token }` consume el `VerificationToken` con `purpose='user_email_verification'`.
- [ ] Idempotente: doble click no rompe. Segundo intento retorna error code `already_consumed` que el frontend trata como cuasi-éxito (US-011-f).
- [ ] Token inválido / expirado / invalidated → 400 con error code distinguible (`invalid_token`, `expired_token`, `invalidated_token`).
- [ ] Token sin match → `invalid_token`.
- [ ] Marca `User.email_verified_at = now()` y setea `consumed_at` en el VerificationToken.
- [ ] Retorna 204.
- [ ] Como parte del slice, `EmailVerificationToken` queda refactorizado a `VerificationToken` child entity de `User` con `purpose` discriminator (ADR-0033).
- [ ] Integration tests cubren: happy path con Martín, doble click (segunda llamada retorna `already_consumed`), token expirado, token inválido, token invalidated por resend.

## Sub-tasks

- [ ] Refactor `EmailVerificationToken` a `VerificationToken` con `purpose` enum (`user_email_verification`, `teacher_institutional_verification`)
- [ ] Comando `VerifyEmailCommand` + handler
- [ ] Endpoint Carter `VerifyEmailEndpoint`
- [ ] Validación de TTL (`expires_at > now()`) y `consumed_at IS NULL`
- [ ] Integration tests para happy path + edge cases (doble click, expirado, inválido, invalidated)

## Notas de implementación

- **`POST` con body, no `GET ?token=`**: el AC original mencionaba `GET`. La implementación final del frontend (US-011-f) usa `POST /api/identity/verify-email` con body para evitar que el token quede en logs de servidor o referer headers. Mantener consistencia: este backend AC ya refleja el endpoint final.
- **VerificationToken child de User**: ADR-0033. Vive dentro del aggregate, no como aggregate independiente. El método `User.ConsumeVerificationToken(rawToken, clock)` encapsula la lógica de validación y mutación atómica.
- **Distinguir `invalid` de `expired` de `already_consumed` en el error code**: importante para que el frontend muestre copy distinto. Un user con link reusado entiende "ya verificaste"; un user con link viejo entiende "pedí uno nuevo".
