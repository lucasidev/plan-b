## Como visitante anónimo, quiero navegar el catálogo de universidades, carreras, planes y materias para tener contexto antes de registrarme

Como visitante anónimo, quiero navegar el catálogo (universidades → carreras → planes → materias) sin necesidad de loguearme, para evaluar la plataforma y entender qué cubre.

## Acceptance Criteria

### Backend

- [ ] `GET /api/universities` retorna listado paginado con (name, short_name, country, city, slug).
- [ ] `GET /api/universities/{slug}/careers` retorna las carreras de una universidad.
- [ ] `GET /api/careers/{id}/plans` retorna sus CareerPlans con el vigente destacado (`effective_to IS NULL`).
- [ ] `GET /api/career-plans/{id}/subjects` retorna materias agrupadas por `year_in_plan` + `term_kind`/`term_in_year`, incluyendo correlativas (con tipo `para_cursar` o `para_rendir`).
- [ ] Sin autenticación requerida.
- [ ] Reads vía Dapper (read models de presentación), no EF (ADR-0018).

### Frontend

- [ ] Páginas RSC bajo `(public)` route group: `/universidades`, `/universidades/[slug]/carreras`, `/carreras/[id]/planes`, `/planes/[id]/materias`.
- [ ] Prefetch en RSC + HydrationBoundary (ADR-0021), navegación con `<Link>` y prefetch de Next.
- [ ] Plan vigente claramente destacado vs planes históricos.

## Sub-tasks

- [ ] Endpoints Carter de lectura en `Planb.Academic.Endpoints`
- [ ] Read models con Dapper en `Planb.Academic.Infrastructure`
- [ ] Páginas frontend en App Router (RSC con TanStack Query prefetch)
- [ ] Integration tests cubriendo paginación + plan vigente vs histórico
- [ ] Validar que multi-universidad funcione con al menos 2 universities seedeadas

## Notas de implementación

- **Reads con Dapper, no EF**: ADR-0018 separa writes (EF) de reads complejos (Dapper). Listar materias agrupadas por año con correlativas requiere agregaciones que en EF salen feas; Dapper resuelve con un SQL directo en el read model.
- **Multi-universidad desde día 1**: ADR-0001. Las URLs incluyen `slug` de university (`/universidades/unsta/...`), no asume una universidad por defecto. La paginación y los filtros son por university desde el endpoint.
- **Versionado de planes**: ADR-0002. La página de plan muestra `effective_from` y `effective_to`. Un alumno puede estar bajo el "Plan 2019" aunque exista un "Plan 2024" más nuevo. La UI tiene que reflejar esto sin ocultar planes históricos.
