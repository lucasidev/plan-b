## Como member verificado, quiero reclamar identidad docente sobre un Teacher del catálogo para empezar el flow de verificación

Como member con email verificado, quiero un endpoint que cree un TeacherProfile con verified_at=null asociado a un Teacher del catálogo, validando que no haya claim previo del mismo user al mismo teacher.

## Acceptance Criteria

### Backend

- [ ] `POST /api/me/teacher-claims` con `{ teacherId }`.
- [ ] Requiere user authenticated + email verified + `role = 'member'`.
- [ ] Valida que el `teacherId` exista en el catálogo Academic (cross-BC vía PublicContract).
- [ ] Valida que no haya claim previo del mismo user al mismo Teacher (constraint `UNIQUE(user_id, teacher_id)`).
- [ ] Crea `TeacherProfile` con `verified_at = NULL`, `verification_method = NULL`.
- [ ] Retorna 201 con el `claimId`.

### Frontend (deuda diferida)

- [ ] Búsqueda de Teacher + botón "Soy yo" para iniciar claim.

## Sub-tasks

- [ ] Comando `StartTeacherClaimCommand` + handler
- [ ] Endpoint Carter
- [ ] UI search Teacher + claim button

## Notas de implementación

- **Claim sin verificación no desbloquea capabilities**: ADR-0008. El `TeacherProfile` con `verified_at = NULL` existe pero no permite responder reseñas. La verificación (UC-031 o UC-032) es el unlocker.
- **Múltiples claim posibles entre users distintos al mismo Teacher**: el `UNIQUE(user_id, teacher_id)` previene que el mismo user claim dos veces. Pero dos users distintos pueden claimear al mismo Teacher; solo uno será aprobado (UNIQUE parcial sobre verified).
- **No hay flow para claim moderator/admin staff**: ADR-0008 establece roles exclusivos. Un User con `role != 'member'` no puede tener TeacherProfile (invariante data-model). El handler chequea el role antes de crear.
