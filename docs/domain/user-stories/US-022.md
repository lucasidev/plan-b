# US-022: Expirar registros no verificados (scheduled)

**Status**: Sprint backlog
**Sprint**: S1
**Epic**: [EPIC-02 — Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: Medium
**Effort**: S
**UC**: [UC-022](../use-cases/UC-022.md)
**ADR refs**: ADR-0015, ADR-0033

## Como sistema (scheduled job), quiero expirar registros no verificados después de 7 días

Como sistema, quiero un job daily a las 3 AM UTC que setee `expired_at = now()` en users con `email_verified_at IS NULL` y `created_at < now() - 7 days`, para que el email vuelva a estar disponible (vía índice único parcial), los `VerificationToken` activos se invaliden en cascada, y la cola de pending no crezca indefinidamente, sin acción del usuario.

## Acceptance Criteria

- [ ] Job scheduled implementado con **Wolverine `ScheduledJob`** (no `IHostedService`). Razón: Wolverine ya está configurado con persistence de PostgreSQL, message retry y outbox; reusar es más barato que mantener un timer ad-hoc. Cron `0 3 * * *` (3 AM UTC, daily).
- [ ] El job dispara `ExpireUnverifiedRegistrationsCommand`. Su handler ejecuta query Dapper read-side: `SELECT id FROM identity.users WHERE email_verified_at IS NULL AND expired_at IS NULL AND created_at < now() - INTERVAL '7 days'`.
- [ ] Por cada match, el handler invoca `User.ExpireRegistration(IClock)` en el aggregate. Esto: (a) setea `User.ExpiredAt = now()`, (b) invalida en cascada los `VerificationToken` activos del user (los marca con `InvalidatedAt = now()`), (c) emite `UnverifiedRegistrationExpired` domain event con `UserId` y `ExpiredAt`.
- [ ] Persistencia: el handler hace `IUserRepository.Update(user)` por cada user expirado. Wolverine `[Transactional]` middleware commitea en batch (configurable; default 1 transaction per command, OK para volumen esperado).
- [ ] El email del user expirado vuelve a estar disponible para nuevo registro. Garantizado por el índice único parcial (sub-task explícita).
- [ ] Logging estructurado: cantidad procesada por run + IDs (en debug). Si la query devuelve 0 matches, log info "No expired registrations to process at {timestamp}".
- [ ] Job idempotente: si corre dos veces seguidas, el segundo run no encuentra users (porque el primer run los marcó con `expired_at NOT NULL` y la query los excluye).
- [ ] Job tolerante a fallos: si Postgres está saturado y un user falla por lock timeout, el handler logea el ID y sigue con el resto. El user fallido se reprocesa en el próximo run.
- [ ] Integration test: insertar 4 users (uno old non-verified con tokens activos, uno old verified, uno reciente non-verified, uno already-expired), correr el job una vez, verificar que solo el primero quedó marcado expired + sus tokens invalidated. Re-correr el job, verificar 0 matches.

## Sub-tasks

- [ ] **Migration `AddExpiredAtToUser`**: agregar columna `expired_at TIMESTAMPTZ NULL` a `identity.users`. Sin default. Sin backfill (todas las rows existentes quedan NULL = activas).
- [ ] **Migration `AddPartialUniqueIndexOnUserEmail`**: `CREATE UNIQUE INDEX user_email_active ON identity.users (email) WHERE expired_at IS NULL`. Reemplaza el unique constraint actual sobre `email`. Antes de la migration, drop del unique existente.
- [ ] Domain: `User.ExpiredAt` property (nullable `DateTimeOffset?`). Configurar en `UserConfiguration.cs` con `HasColumnName("expired_at")`.
- [ ] Domain: `User.ExpireRegistration(IClock)`. Precondición: `email_verified_at IS NULL && expired_at IS NULL`. Setea `expired_at = clock.UtcNow`, recorre `VerificationTokens` activos y los invalida (`token.Invalidate(clock)`), emite `UnverifiedRegistrationExpired` domain event.
- [ ] Domain: `UnverifiedRegistrationExpired` domain event en `Planb.Identity.Domain.Users.Events` con `UserId` y `ExpiredAt`.
- [ ] Domain: `UserErrors.NotEligibleForExpiration` para precondiciones violadas (defensa: el query ya filtra los inelegibles, pero el aggregate también valida).
- [ ] Application: `ExpireUnverifiedRegistrationsCommand` (sin payload, el comando solo dispara la corrida) + handler que combina la query Dapper read-side + iteración + `IUserRepository.Update` por cada match.
- [ ] Application: `IIdentityReadService.GetExpiredCandidatesAsync(IClock, CancellationToken)` que ejecuta el SQL con Dapper. Devuelve `IReadOnlyList<UserId>`.
- [ ] Wolverine: registrar el `ExpireUnverifiedRegistrationsCommand` con `[ScheduledJob("0 3 * * *")]` o equivalente.
- [ ] Update sign-in flow: cuando el user existe pero `expired_at IS NOT NULL`, devolver 401 `identity.signin.invalid_credentials` (anti-enumeración: indistinguible de "user no existe", consistente con login).
- [ ] Integration test del job (los 4 escenarios listados en AC).

## Notas de implementación

- **Wolverine `ScheduledJob` ganó sobre `IHostedService`**: la US original dejaba la decisión abierta. Cierre: Wolverine. Razones: ya está configurado con PostgreSQL persistence (líneas 77-83 de `Program.cs`), maneja retry y outbox, y mantiene los scheduled jobs en la misma infra que el resto del messaging. `IHostedService` con `Timer` requiere boilerplate adicional (cancellation, error handling, scope) que Wolverine ya resuelve.
- **`created_at` reusa, no agregamos `registered_at`**: el AC original mencionaba `registered_at`. La columna existente `created_at` representa el momento de creación del row (que es el registro). No agregamos columna nueva por consistencia con el resto del schema.
- **Cascada sobre `VerificationToken`**: cuando un user expira, sus tokens activos se invalidan dentro del mismo aggregate operation (no por outbox). Esto asegura que si el job corre antes que un mail con link viejo le llegue al user, el link no funciona (verify-email rechaza con `identity.verification.invalidated`).
- **Por qué `expired_at` y no hard delete**: trazabilidad. Si el mismo email vuelve a registrarse después y termina haciendo abuso, podemos correlacionar con el registro fallido viejo. Hard delete pierde esa señal y los stats de "abandonment rate" del flow de registro.
- **Índice único parcial**: `CREATE UNIQUE INDEX user_email_active ON identity.users (email) WHERE expired_at IS NULL`. Permite que el email vuelva a estar disponible cuando el user previo expira sin perder la row vieja. Esto requiere droppear el unique constraint existente sobre `email` en la misma migration.
- **3 AM UTC para minimizar contención**: el job hace UPDATE de N rows. Correrlo en horario bajo evita pelearse con el tráfico de auth real. Si el sistema crece a multi-región, reconsiderar.
- **`UnverifiedRegistrationExpired` domain event**: lo emitimos para audit y para módulos interesados (analytics futuro). Hoy nadie lo consume pero es zero-cost emitirlo. Persistido vía outbox por la integración EF + Wolverine.
- **Integration con sign-in**: un user expirado que intenta loguearse debería recibir 401 (no 403 disabled): el row existe en DB pero está "como si nunca hubiera sido". Para la UX y la anti-enumeración, indistinguible de "user no existe".
