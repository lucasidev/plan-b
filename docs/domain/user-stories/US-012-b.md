## Como member con email verificado, quiero crear un StudentProfile asociando carrera + plan + año de ingreso para desbloquear capabilities de alumno

Subdivisión backend de US-012 (parent original). El frontend (form de selección University → Career → CareerPlan + año) queda para una US separada futura cuando aterrice el flow UI completo y el JwtBearer middleware en backend.

## Acceptance Criteria

- [x] `POST /api/me/student-profiles` con `{ userId, careerPlanId, enrollmentYear }` (UserId en body como hack temporal hasta que el backend tenga JwtBearer middleware; documentado en el endpoint).
- [x] Requiere User con `email_verified_at NOT NULL` (validado por el aggregate).
- [x] Valida CareerPlan existe en Academic vía `IAcademicQueryService.GetCareerPlanByIdAsync` (cross-BC read, no FK Postgres, ADR-0017).
- [x] Valida que no haya `StudentProfile` activo del mismo user para la misma carrera (constraint `UNIQUE(user_id, career_id) WHERE status = 'Active'` partial unique en DB + check en aggregate).
- [x] `enrollment_year` ∈ [`User.MinEnrollmentYear` = 2010, año actual].
- [x] Crea `StudentProfile` con `status = 'Active'`.
- [x] Solo users con `Role = Member` pueden crear profiles (validado por el aggregate, retorna `OnlyMembersCanHaveProfiles`).
- [x] Retorna 201 con `{ id, careerPlanId, enrollmentYear, status }`.
- [x] Integration tests cubriendo: happy path, CareerPlan inexistente (404), email no verificado (403), duplicate (409), year fuera de rango (400).

## Notas de implementación

- **Profile como capability, no rol**: ADR-0008. El user sigue siendo `Role = Member`. Crear `StudentProfile` desbloquea capabilities (cargar historial, simular, reseñar) pero no cambia el rol. Un mismo member puede después claimear `TeacherProfile` y tener ambas capacidades.
- **Validación cross-BC sin FK**: ADR-0017. El handler consulta Academic vía `IAcademicQueryService.GetCareerPlanByIdAsync(careerPlanId)`. Si Academic está caído, el create falla con error claro, no con FK violation.
- **CareerId denormalizado**: el StudentProfile guarda `CareerId` además de `CareerPlanId`. El handler lo deriva del `CareerPlanSummary` que devuelve Academic. Esto permite que el constraint `UNIQUE(user_id, career_id)` sea evaluable en DB sin JOIN cross-schema.
- **Multiple profiles por user**: el constraint es `UNIQUE(user_id, career_id)` partial sobre `Active`, no `UNIQUE(user_id)`. Lucía puede tener StudentProfile activo en TUDCS y otro en otra carrera (doble titulación, cambio de carrera con perfil viejo en estado Inactive futuro, etc.).
- **Auth gap del endpoint**: el UserId llega en el body porque el backend no tiene JwtBearer middleware configurado. Documentado explícitamente en el endpoint. Cuando ese middleware aterrice, el endpoint refactorea para extraer el `sub` del JWT y aplicar `RequireAuthorization()`.
