# US-051: Resolver report

**Status**: Backlog
**Sprint**: —
**Epic**: [EPIC-07 — Moderación](../epics/EPIC-07.md)
**Priority**: High
**Effort**: M
**UC**: [UC-051](../use-cases/UC-051.md)
**ADR refs**: ADR-0011

## Como moderator, quiero resolver un report como upheld o dismissed para limpiar la cola

Como moderator, quiero dos endpoints (uphold / dismiss) con resolutionNote: uphold cascadea a otros reports open de la misma review y la pone en removed; dismiss puede devolver la review a published si era el único open.

## Acceptance Criteria

### Backend

- [ ] `POST /api/admin/reports/{id}/uphold` con `{ resolutionNote }`:
  - Setea report `status = 'upheld'`, `moderator_id = actor.id`, `resolution_note`, `resolved_at = now()`.
  - Cascade: cualquier otro report `open` sobre la misma Review pasa a `upheld` con la misma resolution_note (ADR-0011).
  - Review pasa a `status = 'removed'`.
  - Audit log entry sobre la review con `action = 'removed'`.
- [ ] `POST /api/admin/reports/{id}/dismiss` con `{ resolutionNote }`:
  - Setea report `status = 'dismissed'`, `moderator_id`, `resolution_note`, `resolved_at`.
  - Si era el único report `open` y la review estaba `under_review` solo por el threshold (no por filter), vuelve a `published`.
- [ ] Requiere `role IN ('moderator', 'admin')`.
- [ ] Eventos `ReviewRemoved` (en uphold) o `ReviewRestored` (en dismiss que vuelve a published) vía Wolverine outbox.

### Frontend (deuda diferida)

- [ ] Modal de resolución con textarea para `resolutionNote`.

## Sub-tasks

- [ ] Comando `UpholdReportCommand` + handler con cascade
- [ ] Comando `DismissReportCommand` + handler
- [ ] Endpoints Carter
- [ ] Audit log entries
- [ ] Cross-BC events vía Wolverine outbox (ADR-0030)
- [ ] UI moderator
- [ ] Integration tests: uphold con cascade (3 reports open, todos quedan upheld), dismiss único vuelve a published, dismiss con otros reports open mantiene under_review

## Notas de implementación

- **Cascade on uphold**: ADR-0011. Si la review queda removed, no tiene sentido seguir teniendo reports `open` sobre ella; el outcome ya se decidió. La cascade marca todos como upheld con la misma nota para mantener consistencia.
- **Dismiss no revierte cascade**: ADR-0011 (no reversion on restore). Si un report fue upheld por cascade y después la review se restaura (US-052), los reports cascadeados quedan upheld históricamente. Es decisión histórica, no se reescribe.
- **Vuelve a published solo si under_review fue por threshold**: si la review está `under_review` por filter automático (US-017), un dismiss único no la libera; el moderator decide manualmente con un comando explícito. Distinguir el motivo del paso a under_review (filter vs threshold) en audit log.
