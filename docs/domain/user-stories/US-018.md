# US-018: Editar reseña propia

**Status**: Backlog
**Sprint**: —
**Epic**: [EPIC-05 — Sistema de reseñas](../epics/EPIC-05.md)
**Priority**: Medium
**Effort**: S
**UC**: [UC-018](../use-cases/UC-018.md)
**ADR refs**: ADR-0012

## Como autor de una reseña, quiero editar mi reseña para corregir o ampliar

Como autor de una Review en status='published', quiero un endpoint PATCH que re-corra el filter de contenido sobre el nuevo texto, marque "editada después de respuesta" si tenía TeacherResponse, y deje audit log.

## Acceptance Criteria

### Backend

- [ ] `PATCH /api/me/reviews/{id}` con cambios parciales.
- [ ] Valida ownership: la review pertenece al user.
- [ ] Solo desde `status = 'published'` (ADR-0012). Si está `under_review` o `removed`, retorna 409.
- [ ] Re-corre el filter sobre el nuevo texto. Si trigger, pasa a `under_review`.
- [ ] Si la review tenía `TeacherResponse`, se marca el `updated_at` de la review post-response. La UI muestra badge "editada después de respuesta del docente".
- [ ] Emite `ReviewEdited` integration event.
- [ ] Audit log entry con `action = 'edited'` y `changes = { before, after }` (JSONB).
- [ ] Si el embedding está habilitado, re-encola job de embedding (los textos cambiaron).
- [ ] Cooldown: máximo 5 edits por review en 24h por author. Excedido devuelve 429 `reviews.edit.cooldown_exceeded`.
- [ ] Longitud máxima por campo: subject_text 2000 chars, teacher_text 2000 chars. Excedido devuelve 400.

### Frontend

- [ ] Edit form con preview del badge "editada después de respuesta" si aplica.

## Sub-tasks

- [ ] Comando `UpdateReviewCommand` + handler
- [ ] Endpoint Carter PATCH
- [ ] Re-run del filter de contenido
- [ ] Audit log entry con diff JSONB
- [ ] Re-encolar embedding si feature flag habilitado
- [ ] UI edit form
- [ ] Integration tests: edit clean, edit que dispara filter, edit con response existente (badge), edit sobre review under_review (rechazado)

## Notas de implementación

- **Edit solo desde published**: ADR-0012. Una review en `under_review` no se edita; el moderator es el único que puede actuar (uphold/dismiss). Una review `removed` quedó así por una decisión de moderación, el autor no debería poder restaurarla con un edit. Si el autor apela, va por flow de moderación.
- **Diff en audit log**: ADR-0031. El `changes` JSONB guarda `{before, after}` de los campos modificados. Permite reconstruir la historia sin perder data.
- **No emitimos `ReviewQuarantined` desde edit aunque el filter trigger**: el evento es para la creación inicial. Para el edit, basta con cambiar `status` a `under_review` y registrar en audit log con motivo "edit triggered filter".
