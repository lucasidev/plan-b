## Como member, quiero que la home arranque con una pregunta dominante + CTA accent para saber qué hacer apenas entro

Primer slice del rebuild de `/home`. Reescribe el layout de la página y deja la pregunta dominante con su CTA accent funcional. Los bloques secundarios (cursando ahora, pendientes, actividad reciente) llegan en `-b` y `-c`.

El home actual (`US-043-f` shipped en S1) es un grid 2x2 de DecisionCards equivalentes, sin jerarquía. Este slice rompe ese pattern y deja la nueva estructura visual base lista para que los slices siguientes inserten sus bloques.

## Acceptance Criteria

- [ ] `app/(member)/home/page.tsx` reescrita con layout v2: header con pregunta + CTA, espacio reservado abajo para los 3 bloques (placeholder visual, no contenido real todavía).
- [ ] **Pregunta dominante** como heading display-size, copy variable según estado del alumno:
  - Si está en período de inscripción: "¿qué te anotás este cuatri?"
  - Si ya está cursando: "¿cómo va tu cuatri?"
  - Si no tiene historial cargado: "Empecemos por cargar tu historial"
- [ ] **CTA accent** debajo de la pregunta, con destino según el mismo estado:
  - "Anotarme" → `/planificar?tab=en-curso` (cuando US-016 / US-R-Planificar aterricen). Hoy redirige a stub o `/home` con TODO.
  - "Ver mi cuatri" → tab "En curso" de Planificar (mismo TODO).
  - "Cargar historial" → `/onboarding/welcome` (que el guard de onboarding ya rebota si tiene profile, así que no hay loop) o `/mi-carrera/historial` cuando aterrice US-013.
- [ ] **Helper `getDominantQuestion(session, profile, today)`** en `features/dashboard-home/lib/dominant-question.ts` con la lógica de los 3 estados. MVP: heurística simple basada en `enrollmentYear` + fecha actual + flag de historial cargado.
- [ ] **Borrar componentes viejos del placeholder**: `features/dashboard-home/components/decision-card.tsx`, `mock-decisions.ts`, y referencias en `home/page.tsx`. Si `coursing-now.tsx` se reusa en `-b`, no se borra; si no, también va a la basura.
- [ ] Cream background + hero typography consistentes con la línea visual del rediseño v2 (mismo tratamiento que `(auth)` y `(onboarding)`).

## Out of scope

- Bloques secundarios (cursando, pendientes, actividad): aterrizan en `-b` y `-c`.
- Rebuild del Sidebar / Topbar: deuda separada (no entra en US-044, queda para una US del shell del AppShell).
- Lógica fina de "qué pregunta mostrar" basada en calendario académico real: el helper MVP usa heurística (mes del año + flag historial). Cuando aterrice un calendario académico real, la lógica se enriquece.
- Animaciones de transición o microcopy A/B: copy fijo hardcoded en español.
- Mobile diseñado: web-first per ADR-0041.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User sin StudentProfile entra a `/home` | El guard de `(member)` redirige a `/onboarding/welcome` (US-037-f). No es responsabilidad de este slice. |
| User con profile pero sin historial cargado | `getDominantQuestion` devuelve "Empecemos por cargar tu historial" + CTA "Cargar historial". |
| User con profile + historial pero fuera de período de inscripción | "¿cómo va tu cuatri?" + CTA "Ver mi cuatri". |
| User con profile + sin historial + en período de inscripción | Prioridad: cargar historial primero. Pregunta = "Empecemos por cargar tu historial" (porque sin historial no tiene sentido planificar). |
| Helper recibe data inconsistente (ej. `enrollmentYear=null`) | Default seguro: "Empecemos por cargar tu historial". Nunca rompe el render. |
| CTA navega a una ruta que aún no existe (`/planificar?tab=en-curso`) | TODO en el código + redirige a `/home` con un toast "Próximamente" o renderea un stub temporal. |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** Lucía con StudentProfile + sin historial, **when** entra a `/home`, **then** ve "Empecemos por cargar tu historial" como heading + CTA "Cargar historial".
2. **Given** Lucía con StudentProfile + historial cargado + fecha en período de inscripción (Feb / Jul, MVP), **when** entra a `/home`, **then** ve "¿qué te anotás este cuatri?" + CTA "Anotarme".
3. **Given** Lucía con StudentProfile + historial cargado + fecha fuera de inscripción, **when** entra a `/home`, **then** ve "¿cómo va tu cuatri?" + CTA "Ver mi cuatri".
4. **Given** un user sin sesión, **when** entra a `/home`, **then** el guard de `(member)` lo redirige a `/sign-in` (no es responsabilidad de este slice testearlo, ya cubierto en US-036).

### Cobertura por capa

- **Unit / vitest**: `dominant-question.test.ts` con tabla de inputs (estados del alumno + fecha) → output esperado (texto + CTA destino).
- **Component / vitest + RTL**: `home/page.tsx` renderea con sesión mockeada y verifica que `<h1>` y CTA aparecen con el copy correcto según estado.
- **E2E Playwright**: skip por ahora. Cuando aterricen `-b` y `-c`, un solo spec cubre la home v2 completa.

## Notas de implementación

- **Helper aislado de la página**: `getDominantQuestion` recibe `profile` y `today` como argumentos puros para que sea testeable sin RSC. La página le pasa `new Date()` y la sesión.
- **CTA con TODO**: las rutas `/planificar?tab=...` no existen todavía. Hardcode el href + comment `// TODO(US-016): planificar`. Cuando aterrice US-R-Planificar, se enchufa.
- **Estado "sin historial" detectado por fetch separado**: la session tiene `userId` y `studentProfileId`, pero saber si tiene historial cargado requiere `GET /api/me/historial?count=1` o similar. MVP: feature flag hardcoded en `false` (asumimos sin historial siempre). Cuando aterrice US-013, el helper consulta el endpoint real.
- **Copy en español rioplatense** hardcoded. Sin pipe i18n.
