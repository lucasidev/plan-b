# US-047: Mi perfil (view + edit datos académicos + foto)

**Status**: Backlog
**Sprint**: candidato a S3
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: High
**Effort**: M
**ADR refs**: [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md)

## Como member, quiero ver y editar mis datos de perfil (nombre, mail, universidad, carrera, plan, año, legajo, estado regular, foto) en una pantalla dedicada para mantener mi identidad académica al día

La sesión de claude-design del 2026-05-02 separó "Mi perfil" (identidad del usuario) de "Ajustes" (config de la app). Mi perfil vive accesible desde el **menú del avatar** en el footer del sidebar. Ajustes va en su propio item en sección "Otros" (US-072, S5).

Los datos del StudentProfile ya se guardan en backend (US-012-b, shipped en S1). Esta US es la pantalla para verlos y editarlos.

## Acceptance Criteria

- [ ] Ruta `/mi-perfil` (route group `(member)`).
- [ ] Acceso desde **menú del avatar** en footer del sidebar (no desde ítem de nav directo).
- [ ] **View mode** (default) muestra:
  - Foto + nombre completo + email.
  - Datos académicos: universidad, carrera, plan, año cursando, legajo, estado regular (sí/no).
  - "Miembro desde" (`createdAt` del User).
  - Botón "Editar".
- [ ] **Edit mode** habilita campos editables:
  - Display name (texto).
  - Año cursando (select).
  - Legajo (texto, opcional, validar formato si la universidad lo exige).
  - Estado regular (toggle).
  - Foto (upload, max 2MB, formato png/jpg/webp).
  - Universidad / Carrera / Plan: **NO editables aquí**. Cambiar de plan es flow distinto (post-MVP).
  - Email: NO editable aquí. Cambiar email requiere re-verification (post-MVP).
- [ ] Validación: display name requerido, año entre 1 y 8.
- [ ] Botón "Guardar" envía `PATCH /api/students/me/profile` y revalida la query.
- [ ] Botón "Cancelar" descarta cambios y vuelve a view mode.
- [ ] Sección "Zona peligrosa" al pie con CTA "Dar de baja mi cuenta" → flow de US-075 (member self-disable).

## Sub-tasks

### Backend

- [ ] Endpoint `GET /api/students/me/profile` (devuelve StudentProfile + User datos públicos).
- [ ] Endpoint `PATCH /api/students/me/profile` con `{ displayName, yearOfStudy, legajo, regularStudent }`.
- [ ] Endpoint `POST /api/students/me/profile/photo` (multipart, almacenar en object storage / postgres bytea según ADR de assets: TBD).
- [ ] Validators server-side (display name 1-80 chars, year 1-8, legajo opcional).
- [ ] Tests integration: get, patch parcial, upload foto.

### Frontend

- [ ] `app/(member)/mi-perfil/page.tsx` (server component que prefetch perfil).
- [ ] `features/my-profile/{api.ts,actions.ts,schema.ts,components/{view-mode,edit-form,danger-zone}.tsx,types.ts}`.
- [ ] Edit form con TanStack Form (4+ campos = complejo, ADR-0022) + Zod schema compartido client/server.
- [ ] Foto upload con preview antes de guardar.
- [ ] Avatar menu en sidebar v2: agregar item "Mi perfil" + "Cerrar sesión".

## Notas de implementación

- **Foto storage**: ADR pendiente (¿object storage local? ¿bytea en Postgres? ¿Cloudinary self-hosted?). Para esta US, asumimos endpoint que persiste y devuelve URL servible. Si el ADR no aterriza antes del sprint, queda placeholder sin foto.
- **Universidad / Carrera / Plan no editables aquí**: cambiar de carrera es un flow grande (consume historial, recalcula plan, etc.) que merece US separada post-MVP.
- **Email no editable**: cambiar email requiere re-verify del nuevo email + invalidación del viejo. Post-MVP.
- **Zona peligrosa**: CTA "Dar de baja mi cuenta" abre el flow de US-075 (modal de confirmación → endpoint self-disable → logout).
