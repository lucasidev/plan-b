## Como user, quiero eliminar mi cuenta para ejercer mi derecho de supresión y sacar mis datos de planb cuando quiera

Hoy planb no expone un verb para que el user borre su cuenta. Esto es un gap en tres frentes:

1. **Compliance**: Ley 25.326 (Protección de Datos Personales, Argentina) art. 6 reconoce el derecho de supresión del titular sobre sus datos. Pre-deploy no aplica obligación legal porque no hay users reales, pero antes del primer deploy a entorno con URL accesible esto debe estar.
2. **UX**: cualquier producto serio permite "borrar mi cuenta". Su ausencia es una mancha.
3. **Testing**: los E2E que crean users (onboarding spec, futuros write-review, import-historial, etc.) hoy dejan basura porque no tienen un verb del dominio para limpiar (ver convenciones de testing en [`docs/testing/conventions.md`](../../testing/conventions.md), sección "Dominio vs infra"). Con este endpoint, los specs llaman `DELETE /api/me/account` en `test.afterEach` y queda limpio.

## Acceptance Criteria

- [ ] **Endpoint `DELETE /api/me/account`** autenticado (`[Authorize]`, role `member`). Borra el `User` + cascade de `StudentProfile` + integration event `UserAccountDeleted` para que otros módulos (reviews futuros, etc.) limpien su data owned.
- [ ] **Idempotente para un user borrado**: segundo DELETE devuelve 404 (el user ya no existe), no 500.
- [ ] **Cascade respeta boundaries de módulos**: dentro del módulo Identity, FK cascade nativo. Cross-BC, integration event con handlers en cada módulo afectado (Reviews, Enrollments, Moderation cuando aterricen). Ver [ADR-0017](../../decisions/0017-persistence-ignorance.md).
- [ ] **Frontend**: pantalla en `/settings/account` (o donde aterrice "Mi perfil", US-047) con CTA "Eliminar mi cuenta" + modal de confirmación que pide retipear el email del user para confirmar (anti accidental).
- [ ] Después del DELETE, frontend hace sign-out (limpia cookies/refresh token) y redirige a `/sign-in` con flash "Tu cuenta fue eliminada. Volvé cuando quieras."
- [ ] **Refresh tokens del user revocados**: la entrada en Redis de la lista de revocación se setea para que cualquier sesión activa caiga al próximo refresh.
- [ ] **Audit log inmutable**: se registra el delete en una tabla `identity.user_deletion_log` (o similar) con `userId`, `email_hash` (para detectar reactivaciones del mismo email sin guardar el email plano), `deleted_at`. Esto NO viola supresión: registramos el evento de borrado, no los datos del user.

## Out of scope

Esto NO incluye:

- **Soft delete / "marcar como eliminado"**: el delete es duro. Sin period de gracia, sin "30 días para recuperar". Compliance lo exige así (el user pidió supresión, no archivado).
- **Export antes de borrar** (GDPR-style "download my data"): diferido a US separada. No es requisito de Ley 25.326 art. 6.
- **Cascade de Reviews**: cuando aterrice US de reviews, esa US debe agregar el handler del integration event. Hoy no hay reviews persistentes, no hay nada que cascade.
- **Eliminación del propio `email_hash` del audit log**: el log es inmutable. El email original no se guarda; solo un hash que sirve para correlacionar reactivaciones. No hay "olvidar que existió" — es el gap aceptable de tener auditoría.
- **UI de admin para borrar otro user**: solo el dueño puede borrarse a sí mismo. Admin tiene caminos distintos (suspensión, ban) que NO son delete y son out of scope acá.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User abre el modal y cierra sin confirmar | No pasa nada. Cero llamada al backend. |
| User retipea el email mal en el modal | Submit deshabilitado o muestra error inline; no llama al backend. |
| User confirma con el modal pero el backend devuelve 500 | UI muestra "No pudimos eliminar tu cuenta. Probá de nuevo o contactanos." Cuenta sigue activa. |
| User confirma y el backend OK pero el sign-out falla | UI igual redirige a `/sign-in` (la cuenta ya no existe; cualquier intento futuro de usar la sesión devuelve 401 y limpia). |
| Mismo email se re-registra después del delete | Sign-up funciona normal — backend no chequea contra `user_deletion_log` para bloquear. El audit log sirve para correlacionar (forensics), no para impedir reactivación. |
| Dos tabs abiertas: una borra, la otra hace acción cualquiera | Tab que borró: sign-out OK. Tab residual: próxima request 401 (refresh token revocado o user no existe), redirige a `/sign-in`. |
| User borra cuenta mientras tiene un upload de historial en progreso (cuando aterrice US-014) | El upload abortando es OK. La US de upload define su propia cancelación — ésta solo borra el user. |
| Cascade de cross-BC events falla parcialmente (ej. Reviews module down) | Wolverine outbox retiene el event hasta que Reviews vuelva. El user del módulo Identity ya está borrado; los reviews quedan huérfanos hasta que el handler procese. Aceptable — el user del lado del producto ve "borrado", el cleanup interno es eventual. |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** Lucía está logueada con su sesión de member, **when** llama `DELETE /api/me/account`, **then** el endpoint devuelve 204, el user de Lucía no existe en `identity.users`, su StudentProfile no existe, y se publica `UserAccountDeleted` integration event.
2. **Given** Lucía borró su cuenta, **when** alguien intenta sign-in con sus credenciales viejas, **then** el backend devuelve 401 con `auth.invalid_credentials` (no leak de "este user fue borrado").
3. **Given** el endpoint está expuesto, **when** un user no autenticado lo llama, **then** 401 (estándar).
4. **Given** Mateo está logueado, **when** intenta `DELETE /api/users/{lucia_id}` (endpoint distinto, hipotético), **then** no existe ese endpoint y devuelve 404 — el único delete es self-service.
5. **Given** un E2E creó un user descartable y completó su flow, **when** el spec corre `test.afterEach` que llama `DELETE /api/me/account` autenticado, **then** la DB queda limpia para el próximo run.
6. **Given** Lucía borró su cuenta y el integration event no llegó al módulo Reviews (Reviews down), **when** Reviews vuelve, **then** Wolverine reintrega el event y los handlers procesan el cleanup pendiente.

### Cobertura por capa

- **Unit / domain** (xUnit): regla de "cascade interno del aggregate Identity" (User borra StudentProfile asociado).
- **Handler unit** (xUnit + NSubstitute): `DeleteAccountCommandHandler` invoca el repo, publica el integration event, revoca refresh tokens en Redis.
- **Integration** (WebApplicationFactory + Postgres real): endpoint `DELETE /api/me/account` con auth real, verifica row gone + audit log inserted.
- **E2E** (Playwright): el spec del onboarding (US-037-f) y futuros agregan `test.afterEach` que llama el endpoint para cleanup.

## Notas de implementación

- **Por qué hard delete y no soft**: Ley 25.326 art. 6 habla de "supresión", no de archivado. Hard delete cumple; soft delete deja data del user "presente pero oculta" y eso es ambiguo legalmente.
- **El audit log con email hash es la concesión a auditoría**: si en el futuro hay disputa "yo nunca tuve cuenta acá", el hash permite verificar que sí existió y fue borrado por el propio user. NO es PII porque no se puede revertir el hash.
- **Cross-BC cleanup vía integration events**: respetamos [ADR-0017](../../decisions/0017-persistence-ignorance.md) (no FK cross-schema). Reviews, Enrollments, Moderation suscriben al `UserAccountDeleted` cuando aterricen y limpian su data owned. Identity no sabe ni le importa qué módulos consumen el evento.
- **Convención implícita que sale de esta US**: toda US futura que crea entidad owned por user debe planear su verb de delete. Ver `docs/testing/conventions.md` sección "Dominio vs infra" — es la guía operativa.
