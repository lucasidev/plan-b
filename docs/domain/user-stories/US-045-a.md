## Como member, quiero entrar a `/mi-carrera` y ver una sola pantalla con tabs claros para no saltar entre 4 entradas distintas del sidebar

Primer slice del rebuild de "Mi carrera". Crea la ruta `/mi-carrera`, el layout con nav de 5 tabs, el routing por query param + URL, y los stubs vacíos de cada tab. Los tabs reales (Plan, Correlativas, Materias + Docentes, Historial) aterrizan en `-b`, `-c`, `-d`, `-e`.

También consolida el sidebar: borra las 4 entradas viejas (`/plan`, `/subjects`, `/professors`, `/history`) y deja solo "Mi carrera".

## Acceptance Criteria

- [ ] Ruta `app/(member)/mi-carrera/page.tsx` (server component) con tabs nav + render del tab activo según `?tab=...`.
- [ ] **5 tabs** visibles en la nav: Plan / Correlativas / Materias / Docentes / Historial. Default activo: `plan` (si no hay query param o el valor es inválido).
- [ ] Cambiar de tab actualiza la URL (`router.replace` o `<Link>` para que sea sharable). Back / forward del browser navega entre tabs sin perder la sesión.
- [ ] Cada tab renderea por ahora un **stub `ComingSoon` específico** con copy "Próximamente: <nombre del tab>". El stub se reemplaza por el componente real cuando aterrice el slice correspondiente (`-b` Plan, `-c` Correlativas, `-d` Materias + Docentes, `-e` Historial).
- [ ] **Sidebar update**: borrar entries `/plan`, `/subjects`, `/professors`, `/history` del componente `Sidebar`. Agregar entry única "Mi carrera" → `/mi-carrera`.
- [ ] **Borrar stubs viejos** `app/(member)/{plan,subjects,professors,history}/page.tsx` con sus layouts si existen. Si tienen tests, borrarlos también.
- [ ] **Redirects de las rutas viejas** (opcional pero deseable): `/plan` → `/mi-carrera?tab=plan`, `/subjects` → `?tab=catalogo`, `/professors` → `?tab=docentes`, `/history` → `?tab=historial`. Deja las rutas legacy funcionando para compat de bookmarks.
- [ ] **Drawers** (`/mi-carrera/materia/[code]` y `/docente/[id]`): out of scope acá. Llegan con `-d`.

## Out of scope

- Contenido real de los tabs: 100% stubs `ComingSoon` en este slice.
- Drawers de detalle (materia / docente): aterrizan con `-d`.
- Loading / error states avanzados de los tabs reales: cada slice los define.
- Cambio del look del sidebar (rebuild visual): solo se cambian las entries, no el estilo.
- Empty states avanzados de cada tab: definidos por cada slice.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User entra a `/mi-carrera` sin `?tab` | Default tab `plan` activo. La URL no se reescribe a `?tab=plan` (o sí, según convención: MVP no reescribe, evita hops innecesarios). |
| User entra a `/mi-carrera?tab=invalido` | Fallback al default `plan`, mensaje en consola con warning. No 404. |
| User sin StudentProfile entra a `/mi-carrera` | Guard del `(member)/layout.tsx` redirige a `/onboarding/welcome` (cubierto en US-037-f, no es responsabilidad de este slice). |
| User entra a `/plan` (legacy) | Redirige a `/mi-carrera?tab=plan` con `redirect()` server-side. |
| Click en tab actual | No hace nada (no re-render, no nuevo entry en history). |
| Browser back desde tab `historial` a `plan` | URL cambia, contenido cambia, no recarga la app. |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** Lucía con StudentProfile, **when** entra a `/mi-carrera`, **then** ve los 5 tabs con `Plan` activo + stub "Próximamente: Plan".
2. **Given** Lucía en `/mi-carrera?tab=plan`, **when** clickea el tab "Correlativas", **then** la URL cambia a `?tab=correlativas` y el stub correspondiente aparece.
3. **Given** Lucía en `/mi-carrera?tab=correlativas`, **when** hace browser back, **then** vuelve a `?tab=plan` y el stub Plan aparece.
4. **Given** Lucía entra a `/plan` (legacy), **when** la página resuelve, **then** se la redirige a `/mi-carrera?tab=plan`.
5. **Given** Lucía sin profile, **when** entra a `/mi-carrera`, **then** el guard la lleva a `/onboarding/welcome` (verificación cross-cutting, no test propio).

### Cobertura por capa

- **Component / vitest + RTL**: `mi-carrera-tabs.test.tsx` que renderea con mock `useSearchParams` y verifica que el tab activo refleje el query param.
- **E2E Playwright**: spec liviano `mi-carrera-shell.spec.ts` que navega tabs + browser back. Confirma que las URLs cambian. No asserta contenido (los stubs no tienen contenido testeable).

## Notas de implementación

- **Tab activo via searchParams (server)**: el server component lee `searchParams.tab`, valida con `parseTab`, y pasa el activo a `TabsNav` como prop. `TabsNav` es client component porque genera links + maneja highlight.
- **Tabs nav vs route group nested**: optamos por single page + query param (no nested layouts) porque los tabs son del mismo dominio, sharable, y no cambia el shell. Convención registrable en convenciones de routing si es necesaria.
- **Stubs específicos vs un único `ComingSoon` parametrizado**: optamos por 5 archivos individuales para que cuando aterricen los slices reales, solo se cambia el import en `page.tsx` + se borra el stub. Limpieza mecánica.
- **Sidebar update**: ojo con tests del sidebar (snapshot / role-based). Si rompen, actualizar.
- **Redirects legacy**: con `redirect('/mi-carrera?tab=plan')` server-side. Una línea de código, vale la pena.
