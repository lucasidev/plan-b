## Como visitante anónimo, quiero ver una materia con sus reseñas anonimizadas y agregados para evaluarla antes de inscribirme

Como visitante anónimo, quiero la página de una materia con metadata, agregados (rating promedio, histograma) y lista de reseñas published anonimizadas, para poder decidir si me interesa cursarla.

## Acceptance Criteria

### Backend

- [ ] `GET /api/subjects/{id}` retorna metadata (`code`, `name`, `year_in_plan`, `weekly_hours`, `total_hours`, `description`) + agregados (rating promedio de `difficulty_rating`, histograma 1..5, conteo total).
- [ ] `GET /api/subjects/{id}/reviews` retorna lista paginada de reviews `status = 'published'`, sin identidad del autor.
- [ ] Cada review muestra `difficulty_rating`, `subject_text`, `teacher_text`, nombre del docente reseñado (Teacher en title case), fecha relativa.
- [ ] Reviews `under_review` y `removed` se excluyen del listado público (data-model invariante de moderación).
- [ ] Sin autenticación requerida.

### Frontend

- [ ] Página `/materias/[id]` (RSC) con metadata + agregados arriba, lista de reseñas debajo.
- [ ] Paginación con cursor o page (decidir según volumen esperado).

## Sub-tasks

- [ ] Read model de agregados con Dapper (avg + histogram + count)
- [ ] Endpoint Carter listing reviews con paginación
- [ ] Anonimización en proyección (no en storage). El read model nunca selecciona `enrollment.student_id` ni `user_id`
- [ ] Página frontend con paginación
- [ ] Integration tests: review removed no aparece, review under_review no aparece, autor no se filtra en JSON

## Notas de implementación

- **Anonimato como regla de presentación**: ADR-0009 + invariante transversal del data-model. La identidad del autor existe en DB (necesaria para moderación) pero los DTOs públicos nunca la incluyen. Test de integración explícito que chequea ausencia de `student_id`, `user_id`, `email` en la respuesta.
- **Histograma agregado desde Dapper**: una sola query con `GROUP BY difficulty_rating` evita N+1 y mantiene el endpoint barato. Si el volumen crece a millones, considerar materializar en un read model con event-driven updates desde `ReviewPublished`.
- **Filtrado de status en query, no en aplicación**: el WHERE filtra `status = 'published'` directamente en SQL. Si el filtro fuese en código se podrían leakear reviews `under_review` por error.
