## Como user pendiente de verificar, quiero un botón "reenviar email" para no quedar atorado si no me llegó el primero

Como user que se registró pero no recibió el email o el token le venció, quiero un botón "Reenviar email de verificación" visible en la pantalla post-registro `/auth/check-inbox`, y otro en el banner de error del sign-in cuando aparece 403 `email_not_verified`. Click dispara la action al backend ([US-021-b](US-021-b.md)) y me da feedback honesto: enviado, en cooldown, o error genérico.

## Acceptance Criteria

- [ ] **No** se crea ruta nueva `/verifica-tu-email`. La acción se aloja en dos lugares ya existentes:
  - **`/auth/check-inbox`** (página post-registro existente, US-010-f). Se agrega un botón secundario "Reenviar email de verificación" debajo del mensaje principal. El email viene del search param `?email=...`.
  - **Sign-in form** (US-028-f). El banner de error 403 `identity.account.email_not_verified` actualmente tiene un link workaround "Registrate de nuevo con el mismo email". Se reemplaza por un botón "Reenviar email" que dispara la action sin recargar el form. El email viene del campo del form (ya tipeado por el user).
- [ ] Click dispara server action `resendVerificationAction` que llama `POST /api/identity/resend-verification` con el email.
- [ ] Estados de UI: idle (botón habilitado), loading (botón disabled + spinner), sent (toast verde "Te lo mandamos de nuevo, revisá tu bandeja"), rate-limited (toast amarillo "Esperá unos minutos antes de pedir otro"), error genérico (toast rojo "No pudimos enviarlo, probá en un rato").
- [ ] Anti-enumeración: el botón se muestra siempre. La respuesta del backend es 204 igual exista o no el email; el toast siempre dice "Te lo mandamos de nuevo" (no "tu email no está registrado"). Solo el rate limit (429) muestra mensaje distinto.
- [ ] Cooldown visual: después de un click exitoso, el botón queda disabled por 60 segundos con copy "Reenviado". Esto evita doble click y educa al user sobre el rate limit antes de chocar con el 429.
- [ ] Copy en español rioplatense.

## Sub-tasks

- [ ] Feature `features/resend-verification/` con `actions.ts` (server action), `api.ts` (fetcher), `schema.ts` (Zod, email no vacío), `types.ts` (FormState con discriminated union sent/rate-limited/error).
- [ ] Componente `<ResendVerificationButton email="..." />` en `features/resend-verification/components/` que encapsula el botón + toast + cooldown 60s. Reusable entre `/auth/check-inbox` y el sign-in form.
- [ ] Integrar en `(auth)/auth/check-inbox/page.tsx` (existente, US-010-f) debajo del mensaje principal.
- [ ] Integrar en `features/sign-in/components/sign-in-form.tsx` reemplazando el link workaround "Registrate de nuevo con el mismo email" del banner 403.
- [ ] Actualizar US-028-f para reflejar que el workaround dejó de existir cuando esta US aterrizó.
- [ ] Test unit del componente con happy path + rate-limited + error.

## Notas de implementación

- **Consolidar en `/auth/check-inbox`, no ruta nueva**: la US original mencionaba `/verifica-tu-email`. Descartado para no proliferar rutas. La página existente ya cumple la función de "estás esperando que llegue el mail"; agregar el botón ahí es lower friction y no requiere navegación.
- **Botón también en el sign-in form (banner 403)**: el path real-user es: registra, cierra el browser, vuelve días después, intenta loguearse, recibe 403 "no estás verificado". Sin botón de resend ahí, el user queda atorado o usa el link workaround. Con esta US aterrizando, el banner mejora y US-028-f puede sacar la deuda documentada.
- **Cooldown visual de 60s en el botón**: el rate limit del backend es 5 por hora por IP. 60s es un cooldown soft post-click que evita que el user spamee, sin acercarse al cap real del backend. Si choca con el 429 del backend, el toast educa.
- **Anti-enumeración honesta**: aunque el toast diga "Te lo mandamos de nuevo" cuando el backend silenció (email no existe), no es mentira frontal: literalmente el backend devolvió 204 y nosotros no sabemos por qué. El user honesto recibe su mail; el atacante no descubre nada.
- **`features/resend-verification/` como feature separado**: aunque el botón vive embedido en otras dos pantallas (check-inbox + sign-in), la lógica (action + schema + componente) es atómica y reusable. Separar en feature propia respeta la convención del frontend (1 use case = 1 feature folder).
