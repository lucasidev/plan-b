## Como visitante, quiero crear una cuenta con email + password para entrar a la plataforma

Como visitante, quiero registrarme desde la app: ingresar email + password en `/auth` modo "Crear cuenta", recibir un email de verificación, y quedar con cuenta lista para verificar y loguearme. La parte backend persiste el User con role=member, hashea la password, emite un VerificationToken y dispara el envío SMTP. La parte frontend monta el form en el AuthView, valida con Zod, llama el server action y redirige a `/auth/check-inbox`.

Esta US es el **parent value-increment** del slice de registro. Para implementar se subdividió en dos work items que ya están done:

- [US-010-b](US-010-b.md): Register backend
- [US-010-f](US-010-f.md): Register frontend (sign-up tab del AuthView)

## Sub-tasks (subdivision del parent)

### Backend (cubierto por US-010-b, done)

- [x] Endpoint `POST /api/identity/register` con `{ email, password }`.
- [x] Domain: aggregate `User` con role=member + child `VerificationToken` purpose `UserEmailVerification`.
- [x] BCrypt cost 12 para password hashing.
- [x] SmtpEmailSender que dispara mail post-201.
- [x] Anti-enum coherente con login: 409 si email duplicado (trade-off documentado en US-010-b).
- [x] Integration tests: happy path + duplicate + invalid input.

### Frontend (cubierto por US-010-f, done)

- [x] Página `(auth)/auth/page.tsx` con AuthView en modo "Crear cuenta" (deep-link `?mode=signup`).
- [x] Form con email + password + confirm. Validación cliente Zod.
- [x] Server action `signUpAction` que llama el backend.
- [x] 201 redirige a `/auth/check-inbox?email=<email>`.
- [x] Mensajes de error legibles para 409 / 400.

## Notas de implementación

- **Subdivisión retroactiva**: este parent doc se creó después de que las subdivisiones aterrizaran, para alinear con la metodología "parent en backlog, subdivide en sprint". US-010-b vivió en S0 y US-010-f en S1.
- **Convención multi-universidad**: ADR-0001. El registro acepta cualquier email válido; la afiliación institucional se verifica después fuera del flow base.
- **Cookie httpOnly via server action**: ADR-0023. El frontend nunca toca cookies de sesión directamente.
- **Email único entre non-expired**: el constraint es `UNIQUE WHERE expired_at IS NULL`. Cuando un registro vence (US-022), el email vuelve a estar disponible.
