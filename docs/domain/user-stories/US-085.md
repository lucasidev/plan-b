# US-085: Strike system + pedir edición al autor + ocultar+banear (decisiones de moderación extendidas)

**Status**: Backlog
**Sprint**:
**Epic**: [EPIC-07: Moderación](../epics/EPIC-07.md)
**Priority**: Medium
**Effort**: L-XL (probablemente splittear en sub-slices al entrar a sprint)
**Extends**: [US-051](US-051.md) (uphold/dismiss base + shell del detalle del report)
**ADR refs**: [ADR-0011](../../decisions/0011-cascade-on-uphold-sin-reversion-on-restore.md), [ADR-0040](../../decisions/0040-notifications-como-bounded-context.md), [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md)

> Esta US es el **complemento de [US-051](US-051.md)**. US-051 entrega el shell visual del detalle del report + las 2 opciones base (`Aprobar` y `Ocultar reseña`). US-085 enchufa al mismo shell las 3 opciones restantes del canvas (`canvas-mocks/admin-screens-3.jsx::AdmReporteDetalle`): **strike system**, **pedir edición al autor con plazo 48h**, **ocultar + banear con quarantine cascade**.
>
> El split US-051 / US-085 fue decisión del 2026-05-09 porque cada una de las 3 sub-features arrastra dominio propio (state machine de strikes con decay y auto-ban, scheduled job de deadline 48h, cascade de quarantine al banear) que combinadas hacen a la US-051 imposible de shippear en un solo PR. **Sin US-051 implementada, esta US no puede aterrizar**: depende del shell del detalle como hard dep, no como nice-to-have.
>
> Sub-splitting posible dentro de US-085: `US-085-a` strike system, `US-085-b` pending-author-edit, `US-085-c` ocultar+banear con quarantine. Cada sub-slice shipea independientemente. La decisión concreta de splitting se toma en planning.

## Como moderator, quiero opciones de decisión más granulares que uphold/dismiss para responder al espectro de casos (desde crítica dura pero válida hasta abuso grave), educando al autor cuando corresponde y escalando consecuencias progresivamente

El canvas v2 (`AdmReporteDetalle`) muestra **5 opciones de decisión** en el detalle del report:

1. **Aprobar** ([US-051](US-051.md), opción `dismiss` actual).
2. **Pedir edición al autor** (sub-feature `-b` de esta US). Mensaje al autor con plazo 48h para suavizar.
3. **Ocultar reseña** ([US-051](US-051.md), opción `uphold` actual).
4. **Ocultar + advertir al autor (strike 1)** (sub-feature `-a` de esta US). Otra y va a ban temporal.
5. **Ocultar + banear al autor** (sub-feature `-c` de esta US). Acción terminal, solo casos serios.

Las opciones 1 y 3 ya las cubre US-051. Esta US cubre 2, 4 y 5.

## Acceptance Criteria

### Sub-feature `-a`: Strike system

- [ ] **Nuevo campo** `strikeCount` en `User` (default 0).
- [ ] **Tabla nueva** `user_strike` (audit trail por strike): `{ id, userId, issuedAt, issuedBy (moderatorId), reportId (FK al report que originó), reason, decayedAt (nullable) }`.
- [ ] **Endpoint** `POST /api/admin/reports/{id}/uphold-with-strike` con `{ resolutionNote }`:
  - Equivale a `uphold` (review pasa a `removed` + cascade en otros reports abiertos).
  - **Adicionalmente** incrementa `User.strikeCount += 1` y crea `UserStrike` row.
  - Si `strikeCount` cruza threshold (config-driven, default 3 strikes activos): **dispara auto-ban temporal** (US-068 con razón `auto:strikes_threshold` + duración por config, default 14 días).
- [ ] **Decay temporal**: scheduled job semanal marca como `decayed` los strikes con `issuedAt > 6 meses`. Strikes decayed no cuentan para el threshold (pero quedan en audit).
- [ ] **Threshold por env var**: `MODERATION__STRIKES_AUTO_BAN_THRESHOLD` (default 3) + `MODERATION__STRIKES_AUTO_BAN_DAYS` (default 14) + `MODERATION__STRIKES_DECAY_MONTHS` (default 6).
- [ ] **Audit log entry** con `action='strike_issued'` + cascade entry de auto-ban si aplica.
- [ ] **UI del detalle del usuario** (US-068) muestra `strikeCount` activo + tabla de strikes con razón + fecha + moderator que lo emitió.
- [ ] **Endpoint admin** para anular un strike manualmente (rollback de error): `DELETE /api/admin/user-strikes/{id}` con `{ reason }`. Si la anulación pone el count por debajo del threshold y hay un auto-ban activo, **NO se revierte el ban** (ADR-0011 spirit: no reversion on restore).

### Sub-feature `-b`: Pedir edición al autor

- [ ] **Nuevo estado de Review**: `pending_author_edit` (extiende el state machine actual: `published → pending_author_edit` por decisión moderator).
- [ ] **Endpoint** `POST /api/admin/reports/{id}/request-edit` con `{ moderatorMessage, deadlineHours? }` (default 48h):
  - Review pasa a `pending_author_edit` con `editRequestedAt`, `editDeadlineAt = now() + 48h`, `editRequestMessage`.
  - Este report queda `awaiting_author_edit` (estado nuevo del report, intermedio entre `open` y `upheld/dismissed`).
  - Otros reports abiertos sobre la misma review quedan en `awaiting_author_edit` también (cascade).
  - **Emite `ReviewEditRequested` event via Wolverine outbox** → US-077-b (notifications) dispara notification al autor.
- [ ] **Flow del autor** (cruza US-018 editar reseña):
  - Autor recibe notification con CTA "Revisar pedido del moderator".
  - Al abrir, ve la review actual + el `moderatorMessage` + countdown del deadline.
  - 2 opciones:
    - **Editar y reenviar**: usa el editor de US-018 con flag `?editingForModerator=reportId`. Al publicar, review vuelve a `published`, report pasa a `resolved_by_author_edit` (estado nuevo).
    - **Renunciar (no edito)**: review pasa a `removed` (equivale a un uphold tardío), report pasa a `upheld`.
- [ ] **Scheduled job** `EnforceEditDeadlinesJob` (Wolverine durable, corre cada hora):
  - Busca reviews con `editDeadlineAt < now()` y estado `pending_author_edit`.
  - Decisión automática default: **hide** (review → `removed`, report → `upheld_after_deadline`). Razón: si el autor no respondió en 48h, asumimos que no le importa defenderla.
  - Emite `ReviewEditDeadlineExpired` event.
- [ ] **Cancelar pedido de edición** (rollback del moderator): `POST /api/admin/reports/{id}/cancel-edit-request` que vuelve la review a `published` y el report a `open`.
- [ ] **Audit log**: `action='edit_requested'`, `action='edit_completed'`, `action='edit_deadline_expired'`, `action='edit_cancelled'`.

### Sub-feature `-c`: Ocultar + banear al autor

- [ ] **Endpoint** `POST /api/admin/reports/{id}/uphold-with-ban` con `{ resolutionNote, banDurationDays? }` (nullable = ban permanente).
- [ ] Cascade en orden:
  - Review pasa a `removed` + otros reports abiertos `upheld` (igual que uphold simple).
  - User pasa por **flow US-068 disable** con `reason='moderation:report_uphold_with_ban'` + `disabledUntil = now() + banDurationDays` (si nullable, permanente).
  - **Adicionalmente** TODAS las reviews del user pasan a `quarantine` (estado nuevo: oculto al público pero recuperable si se desbloquea al user). Decisión de Lucas: el ban del autor implica ocultar todo su corpus, no solo la review reportada.
- [ ] **UI del detalle del report** (canvas): la opción "Ocultar + banear" es **danger color** (rojo) y requiere typed-confirm `BAN` antes de aplicar.
- [ ] **Audit log entry** con `action='upheld_with_ban'` + cascade entries por cada review quarantined.

### Común a las 3 sub-features

- [ ] **Detalle del report** (port de `AdmReporteDetalle`):
  - Sección "Decisión" con 5 radio options visualmente diferenciadas:
    - Aprobar (default verde live).
    - Pedir edición al autor (draft yellow).
    - Ocultar reseña (draft yellow).
    - Ocultar + advertir (danger red).
    - Ocultar + banear (danger red).
  - Textarea `Nota interna (opcional)` para `resolutionNote`.
  - Botón "Aplicar decisión" (primary, full-width).
- [ ] **Sección "Contexto del autor"** (port del canvas):
  - Cuenta desde (cuatri de creación).
  - Reseñas escritas (count).
  - Reportes recibidos (count + breakdown: N desestimados / M pendientes).
  - **Strikes activos** (count, color amber si > 0).
  - Estado: `activo` / `strike` / `baneado` (pill).
- [ ] **Authorization**: `role IN ('moderator', 'admin')`.

## Out of scope

- **El flow base uphold / dismiss**: cubierto por [US-051](US-051.md). Esta US lo extiende sin duplicar.
- **Apelación del autor a un strike o ban**: out de MVP. Si el user discute, va por canal de soporte.
- **Strike "negativo" / kudos positivos para moderators**: out.
- **Notification real al autor**: depende de US-077-b. Si US-077-b no está, esta US registra el evento pero el notify es no-op (deuda).
- **Edición desde la app por moderator**: el moderator no edita la reseña del autor, solo pide al autor que la edite. Si el moderator quiere editar directamente, eso es otra historia (probablemente no debería existir).
- **i18n**: español rioplatense hardcoded.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| Moderator emite strike pero el user ya está baneado | Strike se registra igual (audit trail). El threshold no aplica (ya está baneado). |
| Author edita su review en `pending_author_edit` después del deadline | El job ya corrió y la review está `removed`. La edición queda como draft local no persistido. Toast "El plazo de edición venció". |
| Author edita pero no cambia nada significativo (mismo texto) | Validación del backend: si el contenido es idéntico al previo, rechaza con "No detectamos cambios. Si querés mantenerla así, renunciá a editar." |
| Moderator pide edición y el author ya hizo la edición voluntariamente antes (race) | El backend valida `editRequestedAt > review.updatedAt` antes de aceptar. Si race, devuelve 409. |
| Auto-ban dispara por strike 3 pero el user ya tenía ban manual previo activo | Ban manual prevalece. El strike auto-ban se registra como "skipped: user already banned". |
| Strikes decayed (>6 meses) y el user comete otra infracción | Cuentan solo los activos. El user que tuvo 2 strikes hace 1 año y comete otro hoy, queda con 1 activo, no con 3. |
| Cancel-edit-request después de que el author ya editó | 409: la edición ya fue aplicada, no se puede cancelar. |
| Moderator selecciona "Ocultar + banear" sin typed-confirm | Botón "Aplicar decisión" disabled. Tooltip "Escribí BAN para confirmar". |
| `banDurationDays = null` (permanente) | User queda baneado sin auto-restore. Solo un admin con endpoint manual lo desbanea (US futura). |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** moderator decide "Ocultar + advertir" sobre un user con 0 strikes activos, **when** aplica, **then** review → `removed`, user.strikeCount = 1, audit log entry, sin auto-ban.
2. **Given** user con 2 strikes activos, **when** moderator emite strike 3, **then** auto-ban temporal por 14 días + audit cascade.
3. **Given** moderator decide "Pedir edición" con deadline 48h, **when** aplica, **then** review → `pending_author_edit`, evento `ReviewEditRequested` emitido, notification stub al autor (vía US-077-b cuando aterrice).
4. **Given** author en `pending_author_edit` edita y reenvía, **when** publica, **then** review → `published`, report → `resolved_by_author_edit`.
5. **Given** author no responde en 48h, **when** scheduled job corre, **then** review → `removed`, report → `upheld_after_deadline`.
6. **Given** moderator decide "Ocultar + banear" con `banDurationDays=30`, **when** typed-confirm `BAN` + aplica, **then** review → `removed`, user → `disabled` con `disabledUntil = now()+30d`, todas las reviews del user → `quarantine`.
7. **Given** strike emitido por error, **when** admin clickea "Anular strike", **then** strike → `cancelled` + count se decrementa + audit log entry. Si había auto-ban derivado, el ban NO se revierte.

## Notas de implementación

- **Threshold de strikes por env var**: igual patrón que ADR-0010 (auto-hide threshold).
- **Decay 6 meses**: balance entre "perdonar errores viejos" y "registro de mal comportamiento crónico". Configurable.
- **Quarantine vs removed para reviews del user banneado**: `quarantine` es reversible (si el user se desbanea, reviews vuelven a published). `removed` es decisión específica del moderator. Distinguir importa para el state machine.
- **Pedir edición no es solo notification**: implica state machine de la review + deadline tracking. Sin notifications BC (US-077-b) operativo, el flow funciona pero el autor no recibe el aviso. Documentar como deuda.
- **`pending_author_edit` cascadea a otros reports abiertos** de la misma review: si hay 3 reports y se decide "pedir edición", los 3 quedan `awaiting_author_edit`. Cuando el autor edita o vence el deadline, los 3 cierran en cascada.
- **Splitting al sprint**: cuando se planifique, considerar:
  - `US-085-a` strike system standalone (no depende de notifications).
  - `US-085-b` pending-author-edit (depende de US-077-b + US-018).
  - `US-085-c` ocultar+banear con quarantine (depende de US-068 funcionando).
  - Cada sub-slice puede shippearse independientemente.
