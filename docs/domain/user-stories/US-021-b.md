# US-021-b: Reenviar verification email (backend)

**Status**: Done
**Sprint**: S1
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: High
**Effort**: S
**UC**: [UC-021](../use-cases/UC-021.md)
**ADR refs**: ADR-0033, ADR-0034

## Como user pendiente de verificar, quiero un endpoint que reenvíe el email de verificación con un token nuevo

Como user que se registró pero no recibió el email (o el token le venció), quiero `POST /api/identity/resend-verification` que invalide tokens activos previos del purpose `UserEmailVerification`, emita uno nuevo, dispare el envío SMTP y aplique rate limit + anti-enumeration, para recuperar el flujo sin exponer si un email está registrado.

Endpoint anónimo: el user no está logueado todavía (la verificación es prerequisito del login). Lo invoca el frontend desde la pantalla post-registro `/auth/check-inbox`.

## Acceptance Criteria

- [x] `POST /api/identity/resend-verification` con body `{ email }`. Sin auth (endpoint público).
- [x] Siempre 204 (anti-enumeración). Misma respuesta si: el user no existe, ya está verificado, está disabled, o el flow tuvo éxito.
- [x] Si el user existe, no está verificado, y no está disabled: el aggregate invoca `User.RequestVerificationResend(IClock)` que invalida el token activo del purpose `UserEmailVerification` (si lo hay) y emite uno nuevo con TTL 24h. Después manda mail vía `ISmtpEmailSender` con el link `{frontend}/auth/verify-email?token={raw}`.
- [x] Si el user no aplica (no existe / verificado / disabled): silencio, no manda mail.
- [x] Rate limit por IP: máximo 5 requests por hora a `/resend-verification`. Excedido devuelve 429 con title `identity.rate_limit.exceeded`. Reusa la abstracción `IRateLimiter` introducida por [US-033-i](US-033-i.md) con key prefix `identity:ratelimit:resend-verification:{ip}` (Patrón #2 ADR-0034).
- [x] Domain events emitidos por el aggregate: `VerificationTokenInvalidated` (si había token previo) + `VerificationTokenIssued` (siempre que se emite uno). Ambos ya existen como `Domain Events`, no integration events. Wolverine los persiste por outbox para audit.
- [x] Validación FluentValidation: email no vacío + formato válido. Email malformado retorna 400 con title `identity.email.invalid`.
- [x] Integration tests: happy path (Lucía registrada sin verificar pidiendo resend, mail llega vía Mailpit), user verificado responde 204 silencioso (sin mail), user inexistente responde 204 silencioso (sin mail), user disabled responde 204 silencioso (sin mail), exceso de rate limit retorna 429, email malformado retorna 400.

## Sub-tasks

- [x] Domain: `User.RequestVerificationResend(IClock)` que verifica precondiciones (`!emailVerified && !disabled`), invalida token activo de `UserEmailVerification` si existe, emite uno nuevo. Atómico dentro del aggregate.
- [x] `UserErrors`: agregar `EmailInvalid` para el caso 400 de validation.
- [x] Application: `ResendVerificationCommand` + handler. Si el user no aplica (lookup falla, ya verificado, disabled), retorna `Result.Success` silencioso (siempre 204). Si aplica, llama al método del aggregate y dispara el SMTP.
- [x] Endpoint Carter `ResendVerificationEndpoint`. Mapea Result a 204 / 400 / 429.
- [x] Wire del rate limiter al endpoint usando `IRateLimiter` de US-033-i con la key prefix mencionada.
- [x] Reusar `ISmtpEmailSender` y la template de verify-email existente (la misma que usa register).
- [x] Integration tests con Postgres real + Mailpit cubriendo los 6 escenarios de AC.

## Notas de implementación

- **Anti-enumeración con respuesta única (204)**: idéntica respuesta evita que un atacante mapee emails registrados haciendo `resend-verification` para cada uno. Alineado con US-033-i (forgot-password) y US-028-b/f (login).
- **Rate limit por IP, no por user_id**: la US original decía "≤ 3 por hora por user", pero el endpoint es anónimo y acepta `{ email }` sin auth. Identificar al user requiere lookup por email, lo cual hace el rate limit dependiente del estado del user (existe / no existe). Más simple y consistente con US-033-i: rate limit por IP, 5 req/hora.
- **5/hora alineado con forgot-password**: misma cuota para todos los flows anónimos de envío de mail. Si después necesitamos diferenciar por flow, los key prefixes ya están separados.
- **Reusa abstracción de US-033-i**: la primera US que introduce `IRateLimiter` es US-033-i (porque la quiere para forgot-password). Esta US aterriza después y solo wirea el endpoint. Si por orden de implementación esta US sale antes que US-033-i, este ítem se mueve acá y US-033-i lo reusa.
- **Domain events, no integration events**: `VerificationTokenIssued` y `VerificationTokenInvalidated` viven en `Planb.Identity.Domain.Users.Events` como domain events. Wolverine los persiste por outbox para audit, pero no son cross-BC (no escuchan otros módulos). Actualizado para precisión de vocabulario.
- **Endpoint anónimo**: el user no está logueado cuando llega acá (la verificación es pre-requisito del login). Por eso aceptamos `{ email }` en el body en lugar de inferir del session. El rate limit por IP compensa la falta de identificación de user.
- **Email malformado retorna 400**: violar la validación de formato no es enumeración (cualquier atacante con el AC puede ver que `not-an-email` falla por formato). 400 es honesto y no expone state.
- **Sin AC para user logueado**: este endpoint no se invoca desde sesión. Si el user verificado lo invoca, recibe 204 silencioso por el path anti-enum. Si quisiéramos un flow para "cambiar email verificado" eso sería otra US.
