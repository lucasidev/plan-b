## Como user pendiente de verificar, quiero un endpoint que reenvíe el email de verificación con un token nuevo

Como user pendiente de verificar, quiero `POST /api/identity/resend-verification` que invalide tokens activos previos del purpose UserEmailVerification, emita uno nuevo, dispare el envío SMTP y aplique rate limit + anti-enumeration, para recuperar el flujo si el email no llegó o el token expiró sin exponer si un email está registrado.

## Acceptance Criteria

- [ ] `POST /api/identity/resend-verification` con `{ email }`.
- [ ] Si user no existe o ya está verificado: retorna 200 sin revelar diferencia (anti-enumeration).
- [ ] Si user existe y no está verificado: invalida tokens activos purpose=UserEmailVerification y emite uno nuevo via `User.RequestVerificationResend()`.
- [ ] Rate limit: ≤ 3 reenvíos por hora por user.
- [ ] Emite `VerificationTokenInvalidated` + `VerificationTokenIssued` (Wolverine outbox).
- [ ] Integration tests: happy path, user verificado retorna 200 silencioso, user inexistente retorna 200 silencioso, exceso de rate limit retorna 429.

## Sub-tasks

- [ ] Domain method `User.RequestVerificationResend()` que invalida tokens activos + crea uno nuevo
- [ ] Comando `ResendVerificationCommand` + handler Wolverine
- [ ] Endpoint Carter `ResendVerificationEndpoint`
- [ ] Rate limiter (Postgres counter por user_id, ventana de 1 hora)
- [ ] Integration tests con Postgres real + Mailpit (chequear que el email llegó)

## Notas de implementación

- **Anti-enumeration silencioso**: el endpoint responde 200 igual si el email no existe, ya está verificado, o está disabled. Idéntica respuesta evita que un atacante mapee emails registrados haciendo `resend-verification` para cada uno.
- **Invalidar tokens activos antes de emitir uno nuevo**: el constraint `UNIQUE(owner_id, purpose) WHERE consumed_at IS NULL AND invalidated_at IS NULL` solo deja un token activo por purpose por owner. La domain method `User.RequestVerificationResend()` setea `invalidated_at` en el token vigente y emite uno nuevo en la misma operación.
- **Rate limit en Postgres**: el counter por user_id se mantiene en Postgres con una sliding window de 1h. No vale la pena montar una tabla Redis para esto: el volumen es bajo y el costo de un counter en Postgres con índice es trivial. Para `forgot-password` (US-033) sí usamos Redis porque el rate-limit es por IP, no por user.
