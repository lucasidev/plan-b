# US-044: Inicio v2 con pregunta dominante

**Status**: Backlog
**Sprint**: candidato a S2
**Epic**: [EPIC-04 — Planificación de cuatrimestre](../epics/EPIC-04.md)
**Priority**: High
**Effort**: M
**ADR refs**: [ADR-0041](../../decisions/0041-rediseño-ux-post-claude-design.md)

## Como member logueado, quiero que la home arranque con una pregunta dominante que dirija mi acción para no perder tiempo eligiendo qué hacer

El home actual (US-043-f shipped en S1) es un placeholder visual con DecisionCards en grid + sección "Cursando ahora", sin jerarquía clara. La sesión de claude-design del 2026-05-02 zanjó que el Inicio debe arrancar con **una pregunta dominante** ("¿qué te anotás este cuatri?") y todo lo demás subordinado en bloques secundarios.

## Acceptance Criteria

- [ ] Página `app/(member)/home/page.tsx` reemplazada con el layout v2 del mockup.
- [ ] **Pregunta dominante** arriba: heading display-size con la pregunta del momento. El texto puede variar según el estado del alumno:
  - Si está en período de inscripción: "¿qué te anotás este cuatri?".
  - Si ya está cursando: "¿cómo va tu cuatri?".
  - Si no tiene historial cargado: "Empecemos por cargar tu historial" (estado vacío que invita a completar).
- [ ] **CTA accent** debajo de la pregunta, navegando al destino apropiado:
  - "Anotarme" → `/planificar?tab=en-curso` (cuando US-016 + US-R-Planificar aterricen).
  - "Ver mi cuatri" → tab "En curso" de Planificar.
  - "Cargar historial" → onboarding paso 03 o `/mi-carrera/historial` (cuando aterrice US-013).
- [ ] **Bloques secundarios** debajo (subordinados):
  - "Cursando ahora": cards de las materias del período activo con su progreso (semana 8/16, próximos parciales, asistencia).
  - "Pendientes": reseñas que el alumno tiene que escribir + acciones tipo "completá tu historial".
  - "Actividad reciente": las últimas notifs / reseñas del entorno cercano (cuando aterrice US-Notifications).
- [ ] **Sin DecisionCards en grid 2x2** del placeholder actual: ese pattern (4 cards equivalentes) se reemplaza por jerarquía pregunta+CTA dominante + secundarios.
- [ ] Las stub data del placeholder (DecisionCard, CoursingNow, mock-decisions, mock-coursing) se reemplazan por mock data nueva alineada al v2 del mockup. Cuando aterricen las features reales (Planificar, Reseñas, Notifications), cada bloque consume datos reales.

## Sub-tasks

- [ ] Reescribir `app/(member)/home/page.tsx` con el layout v2.
- [ ] Crear `features/dashboard-home/components/{question-block,cursando-now-v2,pending-list,recent-activity}.tsx`.
- [ ] Mock data en `features/dashboard-home/data/`. Cada array con TODO claro de qué feature reemplaza el mock cuando aterrice.
- [ ] Helper para decidir qué pregunta dominante mostrar según estado del alumno (período activo / historial cargado / inscripción abierta). MVP: lógica simple (3 estados). Cuando aterrice un calendario académico real, la lógica se enriquece.
- [ ] Borrar componentes viejos: `features/dashboard-home/components/decision-card.tsx`, `coursing-now.tsx` (si no se reusan), `mock-decisions.ts`, `mock-coursing.ts`.
- [ ] Test snapshot del componente principal (opcional).

## Notas de implementación

- **Estado vacío post-onboarding**: si el alumno termina onboarding sin cargar historial, la home arranca con "Empecemos por cargar tu historial" como pregunta dominante. Eso conecta con la decisión cross-cutting de "estados vacíos invite-to-complete" del ADR-0041.
- **Mock data hasta que aterricen las features reales**: `Cursando ahora` viene de Enrollments + Academic (US-013, US-016). `Pendientes` viene de Reviews (US-017). `Actividad reciente` viene de Notifications (US-Notif backend). Hasta que aterricen, cada bloque tiene mock con TODO.
- **Sidebar v2 + Topbar v2** son trabajo separado pero podría ser práctico aterrizarlos juntos en S2 si entran los 3 en el mismo sprint (Auth rebuild + Onboarding + Inicio v2). El AppShell actual sigue funcionando con la nueva home; el rebuild del shell es US separada cuando aterrice.
