## Como sistema, quiero un BC de Notifications con el aggregate + persistencia + endpoints de read y mutations para que el frontend (US-077-f) consuma datos reales y los prĂłximos slices b-2 / b-3 enchufen subscribers y email delivery encima

## Acceptance Criteria

### Aggregate y persistencia

- [ ] Aggregate `Notification` en `Planb.Notifications.Domain` con campos:
  - `id` (UUID).
  - `userId` (UUID, FK lĂłgica a Identity user, sin constraint cross-schema).
  - `kind` (enum: `review_response | review_useful | review_new_in_subject | plan_period_open | plan_change | system | review_edit_requested`).
  - `title` (string, â‰¤200 chars, human-readable).
  - `body` (string?, â‰¤500 chars).
  - `actorAvatarInit` (string?, 2 chars para mostrar avatar).
  - `targetUrl` (string, deep link al destino).
  - `createdAt`, `readAt` (nullable), `archivedAt` (nullable).
  - `emailStatus` (enum: `not_required | pending | sent | failed`, default `not_required`).
  - `emailRequestedAt`, `emailSentAt`, `emailFailureReason` (nullable).
- [ ] Tabla `notifications.notification` en migration EF Core. Indexes:
  - `(user_id, created_at DESC)` para read API.
  - `(user_id) WHERE read_at IS NULL` partial index para count de no leĂ­das.
  - `(email_status, email_requested_at) WHERE email_status = 'pending'` para que el job de b-3 levante candidatos.
- [ ] Schema separado `notifications` (no FKs cross-schema).
- [ ] Persistence ignorance: el dominio no sabe de EF Core; configuraciĂłn via Fluent API en `NotificationsDbContext`.

### Read API

- [ ] `GET /api/me/notifications` con cursor pagination (default 20, max 100):
  - Query params: `?unread=bool`, `?kinds=<comma-separated>`, `?since=<ISO>`, `?cursor=<opaque>`.
  - Response: `{ items: Notification[], nextCursor: string | null, counts: { total, unread } }`.
  - `items` ordenados por `created_at DESC`.
- [ ] `GET /api/me/notifications/unread-count` retorna `{ count }`. Usado por el bell badge (polling 30s).
- [ ] Authorization: solo el owner. `user_id == actor.id`. 403 si pide notifs ajenas.
- [ ] Soft caching en frontend (TanStack Query con `staleTime: 30s`). No agregar cache server-side en MVP.

### Mutations

- [ ] `PATCH /api/me/notifications/{id}/read`:
  - Setea `read_at = now()` si no estaba leĂ­da. Si ya estaba, idempotente (200 con el body actual).
  - Si la notif no es del actor: 403.
- [ ] `PATCH /api/me/notifications/read-all`:
  - Marca todas las notifs del actor con `read_at IS NULL` como `read_at = now()`.
  - Devuelve `{ markedCount }`.
- [ ] Mutations NO emiten events outbox (son solo read-side updates).

### Admin seed endpoint (solo Development)

- [ ] `POST /api/admin/notifications/seed` con `{ userId, kind, title, body?, targetUrl }` para crear notifs manualmente en dev/test. Ăštil hasta que aterrice b-2.
- [ ] Authorization: solo en `IHostEnvironment.IsDevelopment()`. En prod devuelve 404.

### Tests

- [ ] Integration tests:
  - Crear notif (via seed), `GET` la lista, verificar shape.
  - PaginaciĂłn cursor: 30 notifs, fetch primeros 20, fetch resto, verificar sin duplicados.
  - Filter `?unread=true` solo devuelve no leĂ­das.
  - Mark-as-read individual: idempotente.
  - Mark-all-as-read: marca solo del actor, no de otros users.
  - Authorization: actor A no puede `GET` notifs de actor B (403).

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| User con 0 notifs | Read API devuelve `{ items: [], nextCursor: null, counts: { total: 0, unread: 0 } }`. |
| User con > 1000 notifs | Cursor pagination funciona. nextCursor opaco base64. |
| Notif con `archivedAt` seteado | Aparece en read API por default. `?archived=true|false` reservado pero out de MVP. |
| Mark-as-read de notif ya leĂ­da | Idempotente, devuelve 200 sin cambiar `read_at`. |
| Race: dos requests simultĂˇneos de mark-all-as-read | Ambos completan; el `markedCount` puede diferir entre los dos responses (UNIQUE no aplica). |
| Notif kind desconocido | Validador rechaza (enum strict). |
| `target_url` mal formado | Validador rechaza (regex bĂˇsico: empieza con `/` o `https://`). |

## Notas de implementaciĂłn

- **Patron de mĂłdulo standard**: copiar el scaffolding de `Planb.Identity` o `Planb.Moderation`. Schema separado `notifications`, configuraciĂłn EF Core via Fluent API, registro en `Program.cs`.
- **Cursor opaco**: base64 de `{ createdAt, id }`. Decode en el handler. Si invalid, 400.
- **`unread-count` separado del list**: lo consume el bell badge con polling 30s. No queremos pedir lista entera solo para contar.
- **Sin events de mutation**: mark-as-read es local (no necesita avisar a otros BCs). Si en futuro queremos auditarlos en US-086 audit log, se agrega event sin tocar el frontend.
- **`actorAvatarInit` precomputado**: el handler que crea la notif (en b-2) resuelve el avatar de quien disparĂł (ej. "B" para Brandt si es ReviewResponded). Frontend renderea el cĂ­rculo con la inicial sin tener que ir a Identity. Si actor es sistema, queda null y el frontend muestra icono genĂ©rico.
