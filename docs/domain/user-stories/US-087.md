# US-087: Feed global de actividad reciente (dashboard ops admin)

**Status**: Backlog
**Sprint**:
**Epic**: [EPIC-09: Backoffice de cuentas staff](../epics/EPIC-09.md)
**Priority**: Medium
**Effort**: M
**ADR refs**: [ADR-0042](../../decisions/0042-audit-log-per-bc-no-central.md), [ADR-0031](../../decisions/0031-review-audit-log-como-projection.md), [ADR-0017](../../decisions/0017-persistence-ignorance.md)
**Pattern siblings**: [US-053](US-053.md) (per-review), [US-086](US-086.md) (per-user). Esta US es la versión global cross-BC, cross-user.

## Como admin del equipo plan-b, quiero ver un feed cronológico de actividad reciente del sistema completo (último admin que decidió un report, última uni afiliada, último merge, último user baneado, etc.) en el dashboard ops, para detectar lo que está pasando sin tener que navegar por BCs separados

[US-081](US-081.md) (dashboard ops del admin) describe un panel "Feed de actividad reciente" en su layout. Esta US implementa ese read model cross-BC, sin filtro por user (a diferencia de [US-086](US-086.md) que filtra por user) y sin filtro por aggregate (a diferencia de [US-053](US-053.md) que filtra por review).

[ADR-0042](../../decisions/0042-audit-log-per-bc-no-central.md) decidió mantener projections per-BC. Esta US implementa el read model Dapper cross-schema que UNION-ea todas las projections con ORDER BY `at DESC` + LIMIT N + paginación.

## Acceptance Criteria

### Backend

- [ ] **Endpoint** `GET /api/admin/ops/activity-feed` con paginación cursor (default 30 items, máximo 100). Query params:
  - `?bc=<comma-separated>` filtra por BC origen (ej. `?bc=moderation,academic`).
  - `?actions=<comma-separated>` filtra por tipo de acción (ej. `?actions=plan.imported,subject.merged`).
  - `?since=<ISO>` filtra desde fecha.
  - `?actorRole=<comma-separated>` filtra por role del actor (`admin`, `moderator`, `member`, `system`).
- [ ] **Read model Dapper cross-schema** (mismo patrón que [US-086](US-086.md), sin filter por user):
  ```sql
  SELECT 'reviews' as bc, ... FROM moderation.review_audit_log
  UNION ALL
  SELECT 'identity' as bc, ... FROM identity.user_audit_log
  UNION ALL
  SELECT 'moderation' as bc, ... FROM moderation.action_log
  UNION ALL
  SELECT 'academic' as bc, ... FROM academic.catalog_audit_log
  ORDER BY at DESC
  LIMIT N
  ```
- [ ] **Shape uniforme** (mismo que US-086):
  ```
  {
    at: ISO timestamp,
    bc: 'reviews' | 'identity' | 'moderation' | 'academic',
    action: string,
    actor: { id, displayName, role },
    target: { type, id, label },
    context: object
  }
  ```
- [ ] **Authorization**: solo `role = 'admin'`. NO los moderators (ellos tienen su propia vista enfocada en moderation; el feed global es del equipo plan-b).
- [ ] **Cache Redis** con TTL 60s (mismo patrón que el resto del dashboard ops, [US-081](US-081.md)). Key: `admin:activity-feed:{filtersHash}:{cursor}`.
- [ ] **Performance**: índice coordinado por `at DESC` en cada projection source + materialization opcional si la query se vuelve lenta (out de MVP).

### Frontend

- [ ] **Panel "Actividad reciente"** en `/admin` (dashboard ops, US-081). Posición: aside derecho o sección debajo de los KPIs (decisión visual final en US-081).
- [ ] **Timeline view** (estilo igual a US-086 para consistency):
  - Vertical, ordenado por `at DESC`.
  - Cada item con:
    - Timestamp relativo (`hace 12 min`).
    - Pill de BC origen.
    - Icono per-action.
    - Copy human-readable (reusa el template registry de US-086).
    - Actor con link al detalle.
    - Target con link al detalle.
- [ ] **Limited preview**: por default carga 10 items. CTA "Ver feed completo" navega a `/admin/ops/actividad` con timeline expandido + filtros.
- [ ] **Vista expandida** `/admin/ops/actividad`:
  - Header con filtros (chips por BC, por role del actor).
  - Timeline con cursor pagination.
  - Vacío: "Sin actividad registrada en el período seleccionado."
- [ ] **Auto-refresh**: en el panel del dashboard, refresh manual con botón (sin polling automático).

## Out of scope

- **Audit log per-user**: cubierto por [US-086](US-086.md).
- **Audit log per-review**: cubierto por [US-053](US-053.md).
- **Real-time push**: out de MVP. Refresh manual.
- **Búsqueda full-text en el feed**: out. Si llega, filter por action + by + actor cubre la mayoría de los casos.
- **Export del feed a CSV**: out.
- **Alertas / notifications cuando aparece un tipo de acción**: out (eso vive en monitoring / observability, no en audit log).
- **Para university_staff** (rol futuro con scope a una uni): out. El feed global es admin-only. Cuando aterrice scoping per-uni para staff, va US separada.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| Sistema sin actividad reciente (BCs nuevos, 0 events) | Empty state celebratorio: "Sin actividad por ahora. Andá a tomar mate." |
| Volumen alto (>100 events en último minuto) | Default 30 items + "Ver feed completo" + paginación. No saturar el dashboard. |
| Bug en un projector (un BC sin entries por X tiempo) | El UNION ALL devuelve sin entries de ese BC. NO falla la query. Mostrar warning subtle "BC `academic` sin events recientes; ¿projection ok?" si pasa > 24h sin events de un BC. |
| Filter `?bc=moderation,academic` | Solo se queryean esas tablas. UNION ALL parcial. |
| Cursor inválido (manipulación URL) | Backend devuelve 400 con "Cursor inválido". Front muestra error + reset al inicio. |
| Action no tiene template human-readable | Fallback "Acción `{action}` en `{bc}` por `{actor}`". Log warning en CI. |
| Admin sin permisos a una uni específica (futuro scoping) | Por ahora no aplica (admin = full). Cuando aterrice, filtrar entries cuyo target sea de unis fuera del scope. |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** admin loguea a `/admin`, **when** la página renderea, **then** ve panel "Actividad reciente" con 10 items ordenados por `at DESC`.
2. **Given** click en "Ver feed completo", **when** navega, **then** abre `/admin/ops/actividad` con timeline expandido + filtros.
3. **Given** filter chip `Moderation` activo, **when** se inspecciona el feed, **then** solo se muestran entries con `bc: 'moderation'`.
4. **Given** moderator (NO admin) intenta acceder a `/admin/ops/actividad`, **when** el guard chequea, **then** redirige a `/admin/moderacion` (su vista).
5. **Given** sistema sin actividad, **when** abre el feed, **then** ve empty state celebratorio.
6. **Given** click en actor de un row, **when** se inspecciona, **then** navega al detalle del actor (member → US-068, admin → futura vista de staff users).
7. **Given** dashboard ya cargado con 10 items, **when** admin clickea refresh, **then** vuelve a fetch + actualiza la lista.

## Notas de implementación

- **Reuso de componentes con US-086**: ambas vistas (per-user y global) tienen la misma estructura de timeline. Compartir `<AuditTimeline>`, `<AuditRow>`, `<BcPill>` para evitar duplicación. La diferencia es solo el endpoint que consumen.
- **No hay tabla nueva**: esta US solo agrega un endpoint que joinea cross-schema las projections existentes de US-086 + US-053 source tables. Si US-086 todavía no creó las projections, esta US no puede aterrizar (hard dep).
- **Cache invalidation**: en MVP TTL 60s simple. Pattern más sofisticado (invalidar al recibir event) es overengineering para el caso de uso (el admin no necesita real-time del feed; lo refresca manual).
- **Performance**: UNION ALL cross-schema sobre 4 tablas con índices buenos por `at DESC` debería ser <100ms hasta el orden de 10M rows. Si crece más, evaluar materialized view o cache pre-computado.
- **Scoping per-uni futuro**: hoy admin ve todo. Cuando aterrice rol `university_staff` con scope a una uni (post-MVP), agregar `?universityId=` filter. El read model joinea con metadata de los aggregates (review.career.university, user.profile.university, etc.) que ya existe.
- **Actor con role**: el read model resuelve el role del actor (admin / moderator / member / system) en el JOIN con identity.users. Sistema = events automáticos (scheduled jobs).
