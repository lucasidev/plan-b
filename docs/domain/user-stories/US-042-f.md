## Como member logueado, quiero un layout consistente con sidebar de navegación, topbar y menú de cuenta para moverme por la aplicación

Como Lucía recién logueada, quiero ver inmediatamente la estructura del producto: a la izquierda un sidebar con las secciones (Mi cuatrimestre, Comunidad, Cuenta), arriba breadcrumbs + búsqueda, abajo a la izquierda mi avatar con un menú que tiene "Cerrar sesión". Cualquier ruta autenticada me da el mismo chrome.

Frontend-only: ningún endpoint nuevo. Reemplaza el placeholder actual de `/dashboard` que solo tiene un botón de sign-out.

## Acceptance Criteria

- [x] Layout `app/(member)/layout.tsx` envuelve todas las rutas autenticadas con sidebar + topbar + área de contenido.
- [x] Sidebar (port 1:1 del mockup `docs/design/reference/components/shell.jsx::Sidebar`):
  - Logo + label de universidad/carrera del user.
  - 3 secciones de navegación con sus items: Mi cuatrimestre (Inicio, Plan de estudios, Simulador), Comunidad (Materias, Docentes, Mis reseñas), Cuenta (Historial académico, Configuración).
  - Item activo se resalta basado en la ruta actual (`usePathname`).
  - Items que apuntan a rutas todavía no implementadas llevan a stubs `Próximamente` (ver US-043-f para la primera ruta implementada).
- [x] Avatar dropdown abajo a la izquierda:
  - Iniciales generadas del email (no hay name todavía en session, deuda diferida a US-012 cuando aterrice StudentProfile).
  - Click abre dropdown con: Mi perfil (stub), Configuración (stub), Ver onboarding otra vez (stub), Ayuda y contacto (stub), Cerrar sesión (funcional, llama a `signOutAction`).
  - Click fuera del dropdown lo cierra.
- [x] Topbar:
  - Breadcrumbs derivados de la ruta (configurable por página vía metadata o prop).
  - Search bar con placeholder "Buscar materia, docente, código..." y kbd hint `⌘K`. No funcional todavía (deuda diferida a una US de search).
- [x] Mobile: sidebar colapsable a un drawer (off-canvas) con un menú trigger en la topbar. Si lo dejamos para después, dejar TODO claro y degradar a sidebar fijo en mobile temporalmente.
- [x] Stubs `Próximamente` para las rutas no implementadas: cada una es una página chica con título de la sección + mensaje "Esta sección va a aterrizar en {feature/sprint}". No bloquea navegación; es honesto sobre el estado.

## Notas de implementación

- **Avatar con iniciales del email, no del nombre**: el campo `name` no está en la session todavía porque US-012 (StudentProfile) no aterrizó. Helper saca las dos primeras letras del email antes del `@`. Cuando US-012 aterrice, se reemplaza con `firstName + lastName`.
- **Items hardcoded del sidebar**: las 3 secciones del mockup están porteadas literal. No inventamos labels. Cuando aterricen las features (Plan, Simulador, etc.), las rutas activas dejan de ser stubs.
- **Mobile como TODO explícito**: el mockup es desktop-first. Mobile-friendly es un esfuerzo separado; en este sprint dejamos el sidebar fijo y un TODO. La regla de UX honesta dice que no escondemos features detrás de un menú roto.
- **Search ⌘K no funcional**: el input está visible, no responde. Deuda explícita a una US de búsqueda global futura.
