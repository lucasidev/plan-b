## Como university staff, quiero ver agregados anónimos de mi universidad para tener insight sobre tasas de aprobación, dificultad de materias, combinaciones que más fallan

Como university staff, quiero un endpoint que retorne agregados scoped a mi universidad (rating + cantidad por subject/teacher; tasas de abandono y recursadas; combinaciones de cursada con peores tasas), sin acceso a reseñas individuales ni identidades.

## Acceptance Criteria

### Backend

- [ ] `GET /api/staff/dashboard` retorna agregados scopeados a `User.universityId` del staff (asignado en US-067).
- [ ] Métricas:
  - Rating promedio + count de reviews por subject (de la universidad).
  - Rating promedio + count por teacher.
  - Tasas de abandono y recursadas por subject (computadas desde EnrollmentRecord status).
- [ ] Sin acceso a reseñas individuales ni identidades de alumnos/docentes.
- [ ] Sin acceso a data de otra university.
- [ ] Requiere `role = 'university_staff'`.
- [ ] Time range del dashboard: configurable con query params `?from=YYYY-MM-DD&to=YYYY-MM-DD`. Default: últimos 12 meses.
- [ ] Refresh frequency: cache de 1 hora en Redis con key `analytics:institutional:{universityId}:{from}:{to}`. Patrón #3 hot reads ADR-0034.

### Frontend

- [ ] Dashboard con charts (Recharts o equivalente).

## Sub-tasks

- [ ] Read models Dapper agregados (cross-schema, OK por ser presentation, ADR-0017)
- [ ] Endpoint Carter scoped
- [ ] Authorization policy `university_staff` con scope check
- [ ] UI dashboard con charts
- [ ] Integration tests: staff de university A no ve data de university B, agregados sin identidades

## Notas de implementación

- **Cross-schema query es legal acá**: ADR-0017 prohíbe FKs cross-schema y navigation cross-module en EF, pero para reads de presentación con Dapper se permite. La aggregation cruza Identity + Academic + Enrollments + Reviews; el read model es legítimo, no acopla dominio.
- **Scope hard-enforced en query**: el `WHERE university_id = @userUniversityId` se inyecta en cada read. No depender de filtros del frontend ni de un check en código aparte. Riesgo de leak si se olvida en algún endpoint.
- **Sin PII**: ADR-0009. El staff ve métricas, no rows. Si una "tasa de abandono de MAT201" baja a 1 alumno cursando, el endpoint debería redondear o suprimir para evitar de-anonimización por k-anonymity.
- **Post-MVP la métrica de "combinaciones con peores tasas"**: heurística por definir. Puede ser pares de subjects con `COUNT(EnrollmentRecord students_que_cursaron_ambas) AND tasa_abandono > X`. Validar con UNSTA si es útil antes de implementar. Splittear a US-081 (post-MVP) cuando se aborde la heurística de scoring.
