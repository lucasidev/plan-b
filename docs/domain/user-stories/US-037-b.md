# US-037-b: Onboarding backend: endpoints públicos del catálogo Academic + GET StudentProfile

**Status**: Sprint actual (S2)
**Sprint**: S2
**Epic**: [EPIC-02: Identidad y autenticación](../epics/EPIC-02.md)
**Priority**: High
**Effort**: S
**Parent US**: [US-037](US-037.md) (onboarding integral)
**ADR refs**: [ADR-0017](../../decisions/0017-persistence-ignorance.md), [ADR-0018](../../decisions/0018-ef-core-writes-dapper-reads.md)

## Como frontend del onboarding (US-037-f), quiero endpoints públicos para listar universidades / carreras / planes y un endpoint para chequear si el user tiene StudentProfile, para poder construir las cascadas del paso 02 y el guard "ya completó onboarding"

Los endpoints HTTP del módulo Academic no existen todavía: `IAcademicQueryService` solo expone reads cross-BC para Identity (validar que un CareerPlan existe). El form del paso 02 del onboarding necesita 3 lookups públicos (sin auth) y el guard del layout `(member)` necesita un read de StudentProfile por user.

## Acceptance Criteria

- [ ] `GET /api/academic/universities` devuelve 200 con la lista completa de universidades del catálogo, ordenada alfabéticamente. Sin auth.
- [ ] `GET /api/academic/careers?universityId={id}` devuelve 200 con las carreras de esa universidad. Sin auth. Lista vacía si la uni no existe (no 404: input adversarial).
- [ ] `GET /api/academic/career-plans?careerId={id}` devuelve 200 con los planes de esa carrera, todos los Status. Sin auth. Lista vacía si la career no existe.
- [ ] `GET /api/me/student-profiles?userId={id}` devuelve 200 con el StudentProfile del user, o 404 si no tiene. Mismo gap de auth que US-012-b (userId en query).
- [ ] Los 3 GET de Academic devuelven 400 si el query param requerido falta (`universityId`, `careerId`).
- [ ] `IAcademicQueryService` extendido con `ListUniversitiesAsync`, `ListCareersByUniversityAsync(universityId)`, `ListCareerPlansByCareerAsync(careerId)`. Implementación Dapper.
- [ ] `IIdentityReadService` extendido con `GetStudentProfileByUserIdAsync(userId)`. Implementación Dapper.
- [ ] Integration tests para los 4 endpoints (happy path + edge cases listados abajo).

## Out of scope

Esto NO incluye:

- **Auth real (JwtBearer middleware)**: el endpoint Identity recibe `userId` por query param porque sigue el mismo gap que US-012-b. Cuando aterrice el middleware se cambia a leer del claim `sub` del JWT y se elimina el query param.
- **Paginación de los listados Academic**: el catálogo MVP es chico (4 unis, ~18 carreras, 1 plan por carrera). Se agrega cuando algun listado crezca a >50 items.
- **Filtrado server-side por `Status` de plan**: el endpoint devuelve TODOS los planes para que admin UIs futuras puedan reusar. El cliente filtra `Status == "current"` del lado UI.
- **Búsqueda / fuzzy match**: los listados son "completos" sin filtros más allá del id padre. Cuando aterrice US-071 (búsqueda global) ese caso lo cubre Meilisearch, no estos endpoints.
- **Cache HTTP / CDN edge**: respuestas son deterministic y cacheables, pero no agregamos `Cache-Control` ahora. Optimization diferida.
- **Endpoint de estados del flow de onboarding (`/api/me/onboarding-state`)**: el frontend lo infiere del `GET /api/me/student-profiles` (404 = no completó paso 2). No hace falta state machine extra.

## Edge cases

| Caso | Comportamiento esperado |
|---|---|
| `GET /api/academic/careers` sin `universityId` | 400 con `title: academic.careers.missing_university_id`. |
| `universityId` con Guid.Empty (`00000000-...`) | 400 (mismo título). El cliente nunca debería mandar Empty pero lo cubrimos como input adversarial. |
| `universityId` parsea como Guid pero no existe en DB | 200 con lista vacía. Razón: el caller ya validó la uni en el dropdown previo; un Id "no existe" es defense in depth, no un error de usuario. |
| `careerId` mismo patrón que `universityId` | Mismo comportamiento (400 sin / vacío para no existente). |
| `userId` Guid.Empty en `GET /api/me/student-profiles` | 400 con `title: identity.student_profile.missing_user_id`. |
| `userId` válido pero user no existe | 404 con `title: identity.student_profile.not_found`. Ambiguo intencional con "user existe sin profile" para no leakear "este user existe / no existe" desde un endpoint sin auth. |
| Catálogo Academic vacío en CI / local fresh | El seeder corre al startup (DevSeedHostedService). Si por algún motivo NO seedeó, los listados devuelven `[]` con 200: el caller verá "no hay unis disponibles". |

## Test scenarios

### Críticos (Given-When-Then)

1. **Given** el catálogo seed de Academic con 4 universidades, **when** se hace `GET /api/academic/universities`, **then** devuelve 200 con las 4 universidades ordenadas alfabéticamente por `name`.
2. **Given** UNSTA tiene 4 carreras en el seed, **when** se hace `GET /api/academic/careers?universityId={UnstaId}`, **then** devuelve 200 con esas 4 carreras y cada `universityId` matchea UnstaId.
3. **Given** una `universityId` random que no existe, **when** se hace `GET /api/academic/careers?universityId={random}`, **then** devuelve 200 con lista vacía (no 404).
4. **Given** Lucía está registrada+verificada y completó `POST /api/me/student-profiles`, **when** se hace `GET /api/me/student-profiles?userId={LuciaId}`, **then** devuelve 200 con el profile (UserId, CareerId, CareerPlanId, EnrollmentYear).
5. **Given** Mateo está registrado+verificado pero NO creó profile, **when** se hace `GET /api/me/student-profiles?userId={MateoId}`, **then** devuelve 404 con `title: identity.student_profile.not_found`.
6. **Given** un `userId` random que no existe, **when** se hace `GET /api/me/student-profiles?userId={random}`, **then** devuelve 404 (mismo shape que "existe sin profile": ambiguidad anti-enum).

### Cobertura por capa

- **Integration backend** (xUnit + WebApplicationFactory + Postgres real): los 6 escenarios de arriba como tests separados.
- **Domain unit / handler unit**: no aplica: son reads simples sin business logic. Dapper directo no necesita unit tests.

## Dependencies

- **Depende de**: ninguna abierta. El `POST /api/me/student-profiles` (US-012-b) ya existe y es referenciado por los tests pero no es bloqueante.
- **Bloquea a**: [US-037-f](US-037-f.md) (frontend del onboarding): el form de cascadas del paso 02 consume los endpoints Academic; el guard del layout `(member)` consume el endpoint Identity.

## Notas de implementación

- **`AllowAnonymous` explícito en los 3 endpoints Academic**: el catálogo es público (visitor en `/sign-up` puede ver qué unis están soportadas antes de registrarse). No expone PII.
- **Sin `[Transactional]` middleware**: son reads sin side effects.
- **Errores ambiguos para el endpoint Identity**: tanto "user no existe" como "user sin profile" devuelven 404. Eso es intencional para no leakear info del catálogo de users desde un endpoint sin auth.
- **El gap de auth (userId en query)** está documentado en cada endpoint con instrucciones de migración cuando aterrice JwtBearer middleware.
