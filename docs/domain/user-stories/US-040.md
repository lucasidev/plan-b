## Como docente verificado, quiero responder una reseña sobre mí para dar mi perspectiva

Como TeacherProfile verificado donde teacher_id = review.docente_reseñado_id, quiero un endpoint que cree una sola TeacherResponse asociada a la review.

## Acceptance Criteria

### Backend

- [ ] `POST /api/reviews/{id}/teacher-response` con `{ text }`.
- [ ] Valida que el user logueado tenga un `TeacherProfile` verificado donde `teacher_id = Review.docente_reseñado_id`.
- [ ] UNIQUE: una sola `TeacherResponse` por review.
- [ ] La review debe estar en `status = 'published'` (no se puede responder a reviews `under_review` ni `removed`).
- [ ] `text` no vacío.
- [ ] Crea `TeacherResponse` con `status = 'published'`.
- [ ] Emite `TeacherResponsePublished` integration event.
- [ ] Audit log entry sobre la review (acción: response published).
- [ ] Longitud del `text`: mínimo 50 chars, máximo 2000. Validación FluentValidation.
- [ ] Idempotencia: si el client reintenta POST con la misma review_id, segundo request devuelve la response existente (200 con el body de la primera). Garantizado por la regla 'una sola response por review'.

### Frontend

- [ ] Form de respuesta visible solo para el docente verificado correspondiente.

## Sub-tasks

- [ ] Aggregate `TeacherResponse` como child entity de Review (data-model)
- [ ] Comando `RespondToReviewCommand`
- [ ] Endpoint Carter
- [ ] Authorization check: TeacherProfile verified + teacher_id match
- [ ] UI form respuesta
- [ ] Integration tests: docente verified responde, docente no-verified rechazado, docente otro teacher rechazado, segunda response rechazada (UNIQUE)

## Notas de implementación

- **Authorization cross-BC**: Reviews necesita validar contra Identity (TeacherProfile verified) y Academic (teacher_id válido). Vía PublicContracts. ADR-0017 prohíbe FK cross-schema; los handlers preguntan a los otros BCs.
- **Response visible públicamente**: a diferencia del autor de la review (anónimo), el docente que responde aparece con su nombre. Es la mecánica que motiva al docente a responder (visibility) y al lector a ponderar la respuesta (autoridad).
- **`docente_reseñado_id` único permite la authorization**: el data-model garantiza que una review apunta a un solo Teacher como objeto. La authorization es match exacto. Si la review apuntara a varios docentes, habría que generalizar.
