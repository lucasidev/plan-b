# US-031: Verificar docente por email institucional

**Status**: Backlog
**Sprint**: 
**Epic**: [EPIC-06: Claim e identidad docente](../epics/EPIC-06.md)
**Priority**: Medium
**Effort**: M
**UC**: [UC-031](../use-cases/UC-031.md)
**ADR refs**: ADR-0008, ADR-0033

## Como member con claim pendiente, quiero verificar mi identidad docente vía email institucional para desbloquear capability de responder reseñas

Como member con TeacherProfile pending, quiero un endpoint que reciba mi email institucional, valide que el dominio esté en la lista de la University, genere un VerificationToken (purpose=TeacherInstitutionalVerification) y envíe el link al email institucional. Otro endpoint consume el token y marca el profile como verified.

## Acceptance Criteria

### Backend

- [ ] `POST /api/me/teacher-claims/{id}/institutional-email` con `{ email }`.
- [ ] Valida ownership del claim.
- [ ] Valida que el dominio del email esté en `Teacher.university.institutional_email_domains`. Si no, rechaza con sugerencia de UC-032 (verificación manual).
- [ ] Guarda `institutional_email` en el TeacherProfile.
- [ ] Genera `VerificationToken` con `purpose = 'teacher_institutional_verification'`, TTL 24h.
- [ ] Envía email al `institutional_email` con link `{frontend}/verify-teacher?token=...`.
- [ ] `POST /api/me/teacher-claims/verify` con body `{ token }` consume el token y setea `verified_at = now()`, `verification_method = 'institutional_email'`.
- [ ] Si ya hay otro `TeacherProfile` verified para el mismo Teacher (UNIQUE parcial), rechaza el verify con error claro: "este docente ya fue reclamado".
- [ ] Token expirado / consumido → 400 con error code distinguible.
- [ ] Rate limit por user: 3 requests/hora del flow de verify por institutional email. Excedido devuelve 429.

### Frontend

- [ ] Página `/verify-teacher` que consume el token y muestra estado.

## Sub-tasks

- [ ] Endpoint request institutional verification
- [ ] Endpoint consume token
- [ ] Reuso de `VerificationToken` (ADR-0033) con `purpose = 'teacher_institutional_verification'`
- [ ] Email template `teacher-verification.html`: subject 'Verificá tu cuenta como docente en plan-b', body con link `{frontend}/verify-teacher?token={raw}` + texto explicativo de la verificación institucional.
- [ ] UI flow

## Notas de implementación

- **Reuso de VerificationToken**: ADR-0033. Mismo modelo que `user_email_verification`, distinto `purpose`. Evita una tabla nueva. El `owner_id` apunta a `teacher_profile.id` para este purpose (no a `user.id`).
- **Validar dominio ANTES de generar token**: si el dominio no está en la lista, no consumimos un slot de token ni mandamos mail. UX más rápida.
- **UNIQUE parcial en verify, no en claim**: dos users pueden tener claims pendientes al mismo Teacher. Solo uno termina verified. El segundo que llegue al verify recibe el rechazo.
- **No aceptamos verificación cuando User es staff**: invariante data-model. `User.role != 'member'` no puede tener TeacherProfile.
