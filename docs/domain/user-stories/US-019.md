# US-019: Reportar reseña

**Status**: Backlog
**Sprint**: —
**Epic**: [EPIC-05 — Sistema de reseñas](../epics/EPIC-05.md)
**Priority**: Medium
**Effort**: M
**UC**: [UC-019](../use-cases/UC-019.md)
**ADR refs**: ADR-0010

## Como user logueado (no autor), quiero reportar una reseña que considero inapropiada para que la modere alguien

Como user logueado (no autor), quiero un endpoint que cree un ReviewReport (con UNIQUE por reporter) y, si los reports open superan el threshold configurable, ponga la review en under_review automáticamente.

## Acceptance Criteria

### Backend

- [ ] `POST /api/reviews/{id}/reports` con `{ reason, details? }`.
- [ ] `reason` ∈ enum `review_report_reason` (`spam`, `datos_personales`, `lenguaje_inapropiado`, `difamacion`, `off_topic`, `otro`).
- [ ] Valida reporter ≠ author de la review (cross-BC: el reporter no debe ser el `EnrollmentRecord.student.user_id` de la review).
- [ ] UNIQUE `(review_id, reporter_id)`: si ya reportó, retorna 409.
- [ ] Crea `ReviewReport` con `status = 'open'`.
- [ ] Si `COUNT(reports open sobre la review) >= threshold` (env var, ADR-0010): emite event que pasa la review a `under_review`.
- [ ] Emite `ReviewReported` integration event vía Wolverine outbox.
- [ ] Audit log entry con `action = 'reported'`.
- [ ] Requiere user logueado.
- [ ] Rate limit por reporter: máximo 10 reports/hora. IRateLimiter con key `moderation:ratelimit:report:{userId}`.
- [ ] Longitud máxima de `details`: 2000 chars. Excedido devuelve 400 `moderation.report.details_too_long`.

### Frontend

- [ ] Botón "Reportar" en cada review card que abre modal con `reason` (radio) + `details` (textarea opcional).

## Sub-tasks

- [ ] Aggregate `ReviewReport` en módulo Moderation
- [ ] Comando `ReportReviewCommand` + handler
- [ ] Endpoint Carter
- [ ] Threshold config via env var `MODERATION__AUTO_HIDE_THRESHOLD` (default 3, ADR-0010)
- [ ] Auto-quarantine via integration event Moderation → Reviews
- [ ] UI botón reportar + modal
- [ ] Integration tests: report válido, reporter = author rechazado, segundo report del mismo user rechazado, threshold reached dispara auto-quarantine

## Notas de implementación

- **Threshold por env var, no hardcoded**: ADR-0010. Permite ajustar sensibilidad sin redeploy de código. Default conservador (3) para evitar abuso de auto-hide colectivo.
- **Cross-BC: Moderation owns Report, Reviews owns Review**: el report vive en Moderation. El paso a `under_review` lo hace Reviews al consumir el event. Mantiene el lenguaje propio de cada BC (Moderation: "report life cycle"; Reviews: "review life cycle").
- **Reporter ≠ author**: el chequeo cross-BC requiere que Moderation pregunte a Reviews quién es el author. Vía `IReviewQueryService.GetAuthorUserIdAsync`. Si la review está removed, el report igual se acepta para tener trazabilidad de pedidos.
- **`under_review` no es "removed"**: pasar a `under_review` saca la review de la presentación pública pero no la borra. El moderator decide después en US-051.
