# US-061: Gestionar Career + CareerPlan

**Status**: Done
**Sprint**: S9
**Epic**: [EPIC-08: Backoffice de catálogo](../epics/EPIC-08.md)
**Priority**: High
**Effort**: M
**UC**: [UC-061](../use-cases/UC-061.md)
**ADR refs**: ADR-0049 (supersede ADR-0002)

## Como admin, quiero gestionar Careers y sus versiones de plan para reflejar cambios curriculares

Como admin, quiero crear Career bajo University y gestionar sus CareerPlans (un plan por año, con estado vigente/deprecado) para reflejar la oferta curricular.

> **Nota de modelo (2026-07-18)**: el spec original planteaba CareerPlan con `version_label` + `effective_from/effective_to` (rangos de vigencia) + domain service de overlap. El modelo ya implementado (en uso por el crowdsourcing US-088, los contracts y `StudentProfile.career_id`) es más simple: **un plan por `(career, year)` con `status` Active/Deprecated**, que cubre el caso de ADR-0002 (planes paralelos: viejo Deprecated, nuevo Active) sin la complejidad de los rangos, que no tenía un caller real. US-061 se adapta a ese modelo en vez de migrarlo.

## Acceptance Criteria

### Backend

- [ ] CRUD Career bajo `/api/academic/universities/{universityId}/careers`:
  - Create con `{ name, slug, shortName?, code?, degreeType?, durationYears?, modality?, description? }`:
    - `degreeType`: Grado / Posgrado / Tecnicatura (enum `CareerDegreeType`).
    - `durationYears`: entero 1-15 (hay tecnicaturas de 3, medicina de 6).
    - `modality`: cadencia mayoritaria, **reusa el enum `TermKind`** (el mismo de AcademicTerm/Subject); el form ofrece Anual / Cuatrimestral / Semestral.
    - `description`: texto corto visible al alumno.
    - Todos los campos académicos son nullable en el aggregate (el crowdsourcing US-088 no los carga); el form admin los completa.
  - List (activas + inactivas), get (detail), update, soft delete (deactivate) + reactivate.
- [ ] Slug único por `(university_id, slug)` (constraint `UNIQUE`).
- [ ] `code` opcional pero único por `(university_id, code)` cuando se provee (índice parcial `WHERE code IS NOT NULL`).
- [ ] Soft delete de Career vía `is_active` (no hard delete: CareerPlans/Subjects referencian por id sin FK cross-schema, hard-deletear dejaría filas colgadas).
- [ ] CRUD CareerPlan bajo `/api/academic/careers/{careerId}/plans`:
  - Create con `{ year, label? }` (`label` = identificador editorial, ej. "plan-2023"; el alumno ve "Plan 2023". El `year` sigue siendo la clave de unicidad).
  - List (Active + Deprecated), deprecate (Active a Deprecated), reactivate (Deprecated a Active).
  - `year` único por career (constraint `UNIQUE(career_id, year)`).
- [ ] Requiere `role = 'admin'` en todos los endpoints.
- [ ] Un plan pasa a `Deprecated` cuando aparece uno nuevo; los `StudentProfile` asociados NO se migran automáticamente, quedan en el plan viejo (migración manual, post-MVP).

### Frontend

- [x] UI admin (US-061-f): listado de carreras por universidad, alta/edición (name, slug, nombre corto, código, tipo de título, duración, modalidad, descripción), detalle con panel de planes (alta + deprecar/reactivar). El timeline con estado "transición" y la acción "Migrar" quedan para [US-084](US-084.md); el selector de visibilidad Borrador/Beta/Pública para [US-091](US-091.md); las columnas Materias/Alumnos para US-062/US-093.

## Notas de implementación

- **Versionado de planes**: ADR-0049 (supersede ADR-0002). Múltiples `CareerPlan` por `Career`, uno por año. Un alumno cursa una `Career` bajo un `CareerPlan` específico (su `StudentProfile.career_id` apunta al plan, no a la career).
- **Cuándo deprecar un plan**: cuando aparece un plan nuevo. El plan viejo pasa a `Deprecated` (deja de ser vigente para nuevos ingresos) pero los alumnos en ese plan siguen ahí. Materias del plan viejo siguen existiendo.
- **`code` opcional pero único cuando se provee**: índice parcial `UNIQUE(university_id, code) WHERE code IS NOT NULL`. Permite distinguir carreras con códigos institucionales sin obligar.
- **Soft delete de Career vs deprecate de CareerPlan**: la Career se archiva con `is_active=false` (deja de ofrecerse); el CareerPlan se archiva con `status=Deprecated` (semántica distinta: el plan sigue vigente para sus alumnos). Son operaciones separadas.
