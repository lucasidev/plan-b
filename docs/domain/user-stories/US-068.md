## Como moderator o admin, quiero deshabilitar una cuenta member para suspender el acceso ante abuso o pedido del usuario

Como moderator o admin, quiero un endpoint que setee `disabled_at`, `disabled_reason` y `disabled_by` sobre una User, impida el login mientras esté disabled y soft-flag sus reseñas en presentación, sin borrar nada.

## Acceptance Criteria

### Backend

- [ ] `POST /api/admin/users/{id}/disable` con `{ reason }`.
- [ ] Requiere `role IN ('moderator', 'admin')`.
- [ ] Setea `User.disabled_at = now()`, `disabled_reason = reason`, `disabled_by = actor.id`.
- [ ] Invariante: `disabled_by` apunta a un User con `role IN ('moderator', 'admin')` (data-model invariant).
- [ ] User no puede loguearse mientras esté disabled. El endpoint de sign-in retorna 403 con `title: identity.account.disabled` (US-028-b).
- [ ] Sus reviews quedan en DB; la proyección pública oculta el contenido cuando `user.disabled_at IS NOT NULL`.
- [ ] Emite `UserDisabled` integration event vía Wolverine outbox.
- [ ] Idempotente: disable sobre user ya disabled retorna 200 sin tocar.
- [ ] Endpoint `enable`: NO existe en esta US. Re-habilitar una cuenta deshabilitada va a US-069 (post-MVP). El admin que se equivocó debe corregir manualmente en DB hasta que aterrice.
- [ ] User disabled puede leer su propio perfil con un GET autenticado: NO. Cookie de sesión también se invalida en el disable (revoca todos los refresh via IRefreshTokenStore.RevokeAllForUserAsync, infra de US-033-i). El user disabled queda 100% offline hasta enable.

### Frontend

- [ ] Botón disable en backoffice con confirmación + textarea para reason.

## Sub-tasks

- [ ] Comando `DisableUserCommand` + handler
- [ ] Endpoint Carter
- [ ] Authorization policy moderator|admin
- [ ] Login pipeline rechaza users con `disabled_at NOT NULL` (ya implementado en US-028-b)
- [ ] Integration event Wolverine
- [ ] Reviews proyección + projection update via UserDisabled integration event (cuando exista el módulo Reviews; deuda hasta entonces).
- [ ] Integration tests: disable + login rechazado, reviews del user ocultas en presentation, audit trail visible

## Notas de implementación

- **Soft suspend, no hard delete**: el dato del user se preserva. Razones: trazabilidad (qué hizo, cuándo), apelación posible, requerimientos legales. Hard delete pierde la historia.
- **Reviews quedan en DB pero ocultas en presentation**: el evento `UserDisabled` lo escucha Reviews, que actualiza una proyección "user_review_visibility". Endpoints públicos de reviews filtran las del disabled user.
- **`disabled_by` con invariante de role**: data-model lo enforca a nivel app-validation. Si un member o un disabled-admin se las arregla para invocar el endpoint, el handler rechaza.
- **No hay endpoint `enable` en esta US**: separar como otra US si hace falta. Este flujo es el "kick the abuser".
