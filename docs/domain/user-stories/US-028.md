## Como user verificado y activo, quiero iniciar sesión con email + password para acceder al área autenticada

Como user con cuenta verificada y activa, quiero el modo "Ingresar" del AuthView en `/auth`, ingresar email + password, y al éxito recibir cookies `planb_session` (access JWT, 15 min) + `planb_refresh` (refresh, 30 días) httpOnly + SameSite=Lax. Si las credenciales fallan, banner "Email o contraseña incorrectos" sin distinguir cuál. Si la cuenta no está verificada, banner específico con botón "Reenviar email" (cuando US-021 aterrice). Si está disabled, banner explicativo. Backend persiste el refresh hasheado en Redis (Patrón #1 ADR-0034) para soportar revocación en sign-out.

Esta US es el **parent value-increment** del slice de login. Para implementar se subdividió en dos work items que ya están done:

- [US-028-b](US-028-b.md): Login backend
- [US-028-f](US-028-f.md): Login frontend

## Notas de implementación

- **Anti-enumeración 401**: mismo error y copy para email no existe vs password mal. Atacante no puede mapear emails registrados midiendo respuestas.
- **403 vs 401**: 401 reservado para credenciales malas; 403 para credenciales OK pero estado de cuenta que bloquea. El frontend usa el title del ProblemDetails para discriminar.
- **Refresh token con hash en Redis (Patrón #1 ADR-0034)**: si Redis se compromete, el atacante solo ve hashes. Cookie httpOnly mantiene el raw en el navegador.
- **Refresh endpoint en mismo PR**: sin él, sesión moriría a los 15 min. Splitear en otro sprint dejaría el feature inutilizable para demos.
