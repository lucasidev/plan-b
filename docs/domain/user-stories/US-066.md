## Como admin, quiero revisar evidencia subida por members con claim pendiente y aprobar o rechazar

Como admin, quiero ver la cola de claims pendientes de verificación manual y aprobar/rechazar cada uno; al aprobar, el TeacherProfile pasa a verified con verification_method='manual'. Si ya hay otro verified, requiere intervención.

## Acceptance Criteria

### Backend

- [ ] `GET /api/admin/teacher-claims/pending` lista cola de claims con evidencia subida y `verified_at IS NULL`.
- [ ] Cada item: claim id, member email, Teacher (full name), University, links a archivos de evidencia, fecha de upload.
- [ ] `POST /api/admin/teacher-claims/{id}/approve` setea `verified_at = now()`, `verification_method = 'manual'`, `verified_by = admin.user_id`.
- [ ] `POST /api/admin/teacher-claims/{id}/reject` con `{ rejection_reason }`.
- [ ] Si ya hay otro TeacherProfile verified para el mismo Teacher (UNIQUE parcial), aprobar falla con error claro.
- [ ] Audit log entry de la decisión.
- [ ] Requiere `role = 'admin'`.

### Frontend

- [ ] Cola con preview de evidencia + botones approve / reject.

## Sub-tasks

- [ ] Endpoint listing pending
- [ ] Endpoints approve / reject
- [ ] Validación UNIQUE verified per teacher
- [ ] UI admin queue

## Notas de implementación

- **`verified_by` requiere admin role**: invariante data-model. El check se hace en authorization policy. Tests deben incluir `verified_by` apuntando a User no-admin → fail.
- **Conflict con teacher ya verified**: el data-model tiene constraint `UNIQUE(teacher_id) WHERE verified_at IS NOT NULL`. El handler lo chequea antes para dar mensaje útil ("ya hay un claim verified, contactá al solicitante o desverifica el previo") en lugar de un constraint violation.
- **Sin auto-approve por cantidad de evidencia**: la decisión es humana. No hay heurística que sustituya al admin viendo el DNI del solicitante.
